| **US-OPT-19b** | เก็บ `user.major` ใน JWT Session เพื่อตัด DB Query ซ้ำ | ในฐานะ **นักศึกษา**<br>ฉันต้องการ **ให้ระบบ Load กิจกรรมที่ตรงกับสาขาฉันได้โดยไม่ต้องดึงข้อมูลเพิ่ม**<br>เพื่อให้ **ลด load บนฐานข้อมูลโดยไม่กระทบ UX** | - เพิ่มฟิลด์ `major` เข้า JWT Session token ผ่าน NextAuth `session` callback<br>- ลบ `db.query.users.findFirst({ columns: { major } })` ออกจาก `/api/events` และ routes อื่นที่ดึง major ซ้ำ<br>- Session อัปเดต `major` ทันทีเมื่อนักศึกษาแก้ไข Profile (force token refresh)<br>- Query count ต่อ request ใน `/api/events` ลดจาก 4 เหลือ 3 queries | 📋 วางแผน |
| **US-OPT-19c** | ย้าย `checkAndAwardClosedForms()` ออกจาก Hot-path `/api/houses` | ในฐานะ **ผู้ดูแลระบบ (Admin)**<br>ฉันต้องการ **ให้การคำนวณคะแนนกิจกรรมที่ปิดแล้วไม่ทำให้ Leaderboard โหลดช้า**<br>เพื่อให้ **นักศึกษาเห็น House Points ได้ทันทีโดยไม่ต้องรอ background task** | - ลบ `checkAndAwardClosedForms()` ออกจาก `GET /api/houses`<br>- ย้ายการเรียกไปไว้ที่ Cron Job `/api/cron/award-points` (มีอยู่แล้ว) ตั้ง interval ทุก 5 นาที<br>- Response time ของ `/api/houses` ลดลงเฉลี่ย ≥ 30% เมื่อไม่มี pending forms<br>- Cron Job บันทึก log เมื่อ Award สำเร็จพร้อม timestamp | 📋 วางแผน |
| **US-OPT-19d** | Cache Announcement Singleton อย่างเหมาะสม | ในฐานะ **นักศึกษา**<br>ฉันต้องการ **เห็นแถบประกาศบน Dashboard โหลดทันทีพร้อม Page**<br>เพื่อให้ **ไม่มี Layout Shift หรือ Flash ขณะรอข้อมูลประกาศ** | - Announcement ใช้ `unstable_cache` tag `"announcement"` อายุ 300 วินาที<br>- Revalidate tag `"announcement"` ทันทีเมื่อ Admin บันทึกประกาศใหม่<br>- หน้า Dashboard pre-fetch announcement ฝั่ง Server Component แทนการ fetch client-side<br>- ยกเลิก polling announcement แยก (รวมไปกับ page load แทน) | 📋 วางแผน |
| **US-OPT-19e** | กำหนด Cache-Control Headers ที่ถูกต้องในทุก API Route | ในฐานะ **Developer**<br>ฉันต้องการ **ให้ทุก API Route มี Cache-Control Header ที่เหมาะสมกับลักษณะข้อมูล**<br>เพื่อให้ **Browser และ CDN (nginx) Cache ข้อมูลได้ถูกต้อง ลด round-trip ซ้ำซ้อน** | - Routes ที่เป็น Public ไม่เปลี่ยนบ่อย (house list): `public, s-maxage=30, stale-while-revalidate=60` (มีอยู่แล้ว — คงไว้)<br>- Routes ที่เป็น Private per-user (individual standing, profile): `private, no-store` (มีอยู่แล้ว — คงไว้)<br>- Routes ที่ไม่มี header เลย (events, announcements): เพิ่มให้ครบตามประเภทข้อมูล<br>- สร้าง audit checklist ตรวจสอบทุก route ว่ามี header ถูกต้อง | 📋 วางแผน |
| **US-OPT-19f** | วัด Baseline และตรวจสอบ Core Web Vitals | ในฐานะ **Developer**<br>ฉันต้องการ **มีตัวเลข Baseline ก่อนและหลัง Optimization เพื่อยืนยันว่าทำงานได้จริง**<br>เพื่อให้ **ตัดสินใจได้ว่า Optimization ไหนคุ้มค่าและ Production พร้อม** | - วัด Lighthouse Score (Performance, Best Practices) บนหน้า Dashboard และ Login ก่อน Sprint เริ่ม<br>- ตั้ง Target: LCP ≤ 2.5s, FID/INP ≤ 200ms, CLS ≤ 0.1 บน Mobile 4G Simulated<br>- Vercel Speed Insights (มีอยู่แล้ว) ใช้เป็น Continuous Monitoring หลัง Deploy<br>- บันทึก Before/After ตัวเลขไว้ใน Sprint Review | 📋 วางแผน |
