| :--- | :--- | :--- | :--- | :--- |
| **US-AUTH-01** | ระบบยืนยันตัวตน CMU | ในฐานะ **นักศึกษา**<br>ฉันต้องการ **ล็อกอินผ่าน Google Account CMU**<br>เพื่อให้ **เข้าใช้งานระบบได้อย่างปลอดภัยโดยไม่ต้องสมัครบัญชีใหม่** | - ต้องบล็อกไม่ให้อีเมลที่ไม่ใช่ลงท้าย `@cmu.ac.th` เข้าระบบ<br>- ข้อมูลบทบาท (`role`) ต้องเก็บใน JWT Session | [x] |
| **US-ONBD-02** | ระบบ Onboarding และเก็บข้อมูลสุขภาพ | ในฐานะ **นักศึกษาใหม่**<br>ฉันต้องการ **กรอกข้อมูลส่วนตัวและข้อจำกัดทางการแพทย์ในการใช้งานครั้งแรก**<br>เพื่อให้ **ระบบสามารถดูแลความปลอดภัยของฉันได้ตลอดกิจกรรม** | - จำกัดความยาวรหัสนักศึกษา 9 หลัก (เฉพาะตัวเลข)<br>- จำกัดความยาวเบอร์โทรศัพท์ 10 หลัก (เฉพาะตัวเลข)<br>- ต้องบันทึกสถานะ `profileCompleted` เป็น true หลังกรอกเสร็จ<br>- ข้อมูลสุขภาพต้องได้รับการเข้ารหัสก่อนบันทึก | [x] |
| **US-ONBD-03** | ระบบ Sync Session ทันที | ในฐานะ **นักศึกษา**<br>ฉันต้องการ **เข้าสู่ Dashboard ได้ทันทีหลังกรอกข้อมูล Onboarding เสร็จ**<br>เพื่อให้ **ใช้งานระบบต่อได้ทันทีโดยไม่ต้อง Log out เพื่อ Refresh สิทธิ์** | - เมื่อบันทึกโปรไฟล์เสร็จ ต้องอัปเดต Session Cookie ใหม่บนเบราว์เซอร์ทันที (JWT callback session sync)<br>- ระบบไม่แสดงหน้า Onboarding ซ้ำ | [x] |
| **US-QR-04** | โทเค็นคิวอาร์เช็คอินความปลอดภัยสูง | ในฐานะ **นักศึกษา**<br>ฉันต้องการ **ใช้ QR Code ในการสแกนเข้างานที่รีเฟรชตัวเองตลอดเวลา**<br>เพื่อให้ **ป้องกันไม่ให้คนอื่นแคปเจอร์หน้าจอไปสแกนแทนได้** | - โทเค็นต้องเซ็นด้วย HMAC-SHA256 อ้างอิง `AUTH_SECRET`<br>- รหัส QR ต้องมีอายุ 5 นาที และมีช่วงเวลา Grace period 30 วินาที<br>- ระบบใช้ Timing-Safe ในการเปรียบเทียบโทเค็นเพื่อกัน Side-channel | [x] |
| **US-SCAN-05** | การสแกนเช็คอินแบบสองขั้นตอน | ในฐานะ **ผู้รับลงทะเบียน (Registration/SMO)**<br>ฉันต้องการ **สแกนคิวอาร์นักศึกษาและดูสัญญาณข้อมูลทางการแพทย์เพื่อยืนยันอีกครั้ง**<br>เพื่อให้ **ตรวจสอบและคัดกรองความปลอดภัยของนักศึกษาก่อนยืนยันเข้าร่วม** | - สแกนครั้งที่ 1 (Scan): แสดงสัญญาณสุขภาพ (โรคประจำตัว/ยาฉุกเฉิน)<br>- สแกนครั้งที่ 2 (Confirm): บันทึกข้อมูลและเวลาสแกนจริงสำเร็จ<br>- ระบบจำกัดการยืนยันตัวตนเฉพาะคนที่มีสถานะ `registered` เท่านั้น | [x] |
| **US-QUOT-06** | การควบคุมโควต้าลงทะเบียนและ Walk-in | ในฐานะ **ผู้จัดการกิจกรรม (Admin/Organizer)**<br>ฉันต้องการ **จำกัดสิทธิ์ความจุการเข้าร่วมกิจกรรมและโควต้า Walk-in**<br>เพื่อให้ **จำนวนผู้เข้าร่วมไม่เกินเพดานที่สถานที่รองรับ** | - ระบบลงทะเบียนและเช็คอิน Walk-in ต้องรันด้วย SQL `FOR UPDATE` (Row Lock)<br>- ตรวจสอบโควต้ากิจกรรมรวม, โควต้าผู้เข้าร่วมไทย/อินเตอร์ และจำกัดโควต้า Walk-in แยกต่างหากอย่างแม่นยำ | [x] |
| **US-AUDT-07** | ล็อกความปลอดภัยที่แก้ไขไม่ได้ | ในฐานะ **ผู้ดูแลระบบความปลอดภัย (Super Admin)**<br>ฉันต้องการ **เก็บบันทึกประวัติการเปลี่ยนแปลงข้อมูลและการเข้าถึงข้อมูลอ่อนไหว**<br>เพื่อให้ **ตรวจสอบย้อนหลังได้โปร่งใสและไม่มีใครสามารถแก้ไขล็อกนั้นได้** | - ข้อมูลล็อกในตาราง `auditLogs` ต้องเชื่อมโยงกันด้วย SHA-256 Hash Chain<br>- ระบบต้องไม่อนุญาตให้รันคำสั่ง UPDATE หรือ DELETE บนตารางล็อก<br>- หากมีล็อกแถวใดโดนแก้ไข ระบบตรวจเช็คพบผ่านฟังก์ชัน `verifyChainIntegrity` | [x] |
| **US-PDPA-08** | การจำกัดสิทธิ์ข้อมูลแพทย์ตาม PDPA | ในฐานะ **ผู้รับลงทะเบียน (Registration)**<br>ฉันต้องการ **เห็นเฉพาะสัญญาณเตือนเรื่องสุขภาพโดยไม่เห็นรายละเอียดโรค**<br>เพื่อให้ **ปฏิบัติตามกฎหมายความเป็นส่วนตัว (PDPA) อย่างเคร่งครัด** | - บทบาท Registration/Organizer เห็นเพียง "สัญญาณแจ้งเตือนและกลุ่มภาวะทางแพทย์" (เช่น โรคประจำตัว หรือ ยาฉุกเฉิน)<br>- บทบาท Admin เท่านั้นที่สามารถดูรายละเอียดประวัติการรักษาพยาบาล (Medical History) และยารักษาโรคได้ | [x] |
| **US-SMO-09** | สิทธิ์จำกัดเฉพาะหน้าสแกน (Scanner-only) | ในฐานะ **สโมสรนักศึกษา (SMO)**<br>ฉันต้องการ **เข้าใช้งานเฉพาะกล้องสแกนเช็คอินกิจกรรมได้ทันที**<br>เพื่อให้ **ช่วยสแกนหน้างานได้สะดวกรวดเร็วโดยไม่เห็นข้อมูลหลังบ้านของแอดมิน** | - บทบาท SMO เมื่อเข้าลิงก์ `/admin` จะเด้งอัตโนมัติไปที่หน้าสแกน QR เท่านั้น<br>- ระบบใช้ `src/proxy.ts` (Middleware) ในการด่านกรองสิทธิ์ไม่ให้ SMO เข้าหน้าแอดมินส่วนอื่น | [x] |

### 1.2 ความต้องการที่ควรจะมี (Should Have)

| ID | Epic / Feature | User Story (ความต้องการของผู้ใช้) | เกณฑ์การรับมอบ (Acceptance Criteria) | สถานะ |
| :--- | :--- | :--- | :--- | :--- |
| **US-FORM-10** | ระบบฟอร์ม KAS และแตกแขนงคำถาม | ในฐานะ **ผู้ประเมินผลกิจกรรม (Admin)**<br>ฉันต้องการ **สร้างฟอร์มประเมินที่ข้ามคำถามได้ตามเงื่อนไขคำตอบก่อนหน้า**<br>เพื่อให้ **นักศึกษาประเมินความรู้/ทักษะได้อย่างเหมาะสมกับตัวเอง** | - รองรับฟอร์ม 4 แบบ: `K_pre`, `K_post`, `A`, `S`<br>- รองรับ JSON Question v2 แตกเซกชันและข้ามด้วยเงื่อนไข (`visibleIf`, targetSection)<br>- คำนวณคะแนนตามเส้นทางเดินข้อสอบจริงที่มีคะแนน (`graded`) | [x] |
| **US-FORM-11** | บล็อกการทำฟอร์มผู้ไม่ร่วมงาน | ในฐานะ **ผู้จัดกิจกรรม (Organizer)**<br>ฉันต้องการ **บล็อกไม่ให้นักศึกษาที่ไม่ได้เข้าสแกนกิจกรรมเปิดทำแบบประเมิน**<br>เพื่อให้ **คะแนนสะสมมาจากผู้เข้าร่วมงานที่ได้เรียนรู้จริงเท่านั้น** | - ระบบตรวจเช็คว่านักศึกษาต้องมีประวัติเช็คอิน (`attended`) บนกิจกรรมก่อน<br>- หากไม่มีประวัติเช็คอิน จะแสดงหน้าต่างแจ้งเตือนสีแดงในภาษาของนักศึกษา | [x] |
| **US-LEAD-12** | ลีดเดอร์บอร์ดบ้านและคะแนนตัวเอง | ในฐานะ **นักศึกษา**<br>ฉันต้องการ **เห็นอันดับคะแนนของบ้านและอันดับรายบุคคลของฉันสะสมแบบ Real-time**<br>เพื่อให้ **เกิดความสนุกสนานและรับรู้สถานการณ์จัดอันดับของฉันได้ทันที** | - ดึงข้อมูลอันดับตัวเอง (`/api/houses/individual/me`) ทันทีที่ล็อกอินสำเร็จ<br>- นักศึกษาที่คะแนน <= 0 จะแสดงผลเป็น "ยังไม่ถูกจัดอันดับ" (Unranked) | [x] |
| **US-SHOP-13** | ระบบร้านค้า Merch และอัปโหลดสลิป | ในฐานะ **นักศึกษา**<br>ฉันต้องการ **สั่งซื้อเสื้อ/ของที่ระลึกของคณะและอัปโหลดหลักฐานการโอนเงิน**<br>เพื่อให้ **จองของที่ระลึกได้อย่างรวดเร็วและตรวจสอบประวัติการซื้อได้** | - แสดงสินค้าแบบมีตัวเลือก (Variants) เช่น ไซซ์ พร้อมระบุสต็อกคงเหลือแยกชิ้น<br>- เก็บไฟล์ภาพหลักฐานการโอนเงิน (สลิป) ไว้ใน Private Folder ห้ามแชร์เป็นลิงก์สาธารณะ<br>- มีขั้นตอนการอนุมัติ/ปฎิเสธออเดอร์โดยแอดมิน | [x] |
| **US-ANNC-14** | แถบประกาศหน้า Dashboard | ในฐานะ **ผู้จัดกิจกรรม (Admin)**<br>ฉันต้องการ **อัปเดตข้อความประกาศสำคัญหน้า Dashboard ของนักศึกษาได้สด**<br>เพื่อให้ **แจ้งข่าวสารเร่งด่วนได้ทันทีโดยไม่ต้อง Deploy ระบบใหม่** | - แอดมินสามารถเปิด-ปิด แถบข้อความและแก้ไข rich text ได้ผ่านหน้าจัดการแอดมิน<br>- บันทึกในรูปแบบ Singleton ในฐานข้อมูลและส่งสัญญาณอัปเดตผ่าน API ทันที | [x] |
