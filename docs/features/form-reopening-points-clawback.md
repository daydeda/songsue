# Feature Spec — Points Clawback on Form Re-opening (ระบบคืนคะแนนเมื่อเปิดแบบประเมินใหม่)

> **STATUS: IMPLEMENTED (shipped on PR #88).**

---

## 1. Context / ปัญหาและที่มา
เมื่อจบการประเมินกิจกรรมและปิดระบบ แบบประเมิน (KAS Forms) จะทำการคำนวณและแจกจ่ายคะแนนโบนัสหรือคะแนนส่วนร่วมไปยังผู้เรียนและกลุ่มบ้านในตาราง `houses` พร้อมบันทึกหลักฐานในตาราง ledger `score_history` 

อย่างไรก็ตาม หากแอดมินมีความจำเป็นต้อง **"เปิดแบบประเมินอีกครั้ง"** เพื่อรองรับการเก็บตกหรือแก้ไขข้อมูล จะเกิดปัญหาคะแนนซ้ำซ้อนหรือคะแนนเฟ้อขึ้น หากระบบไม่มีกลไกดึงคะแนนรอบเก่ากลับคืนก่อนเริ่มการประเมินรอบใหม่

---

## 2. การออกแบบกลไกการดึงคะแนนคืน (Technical Design)

### 2.1 ธุรกรรมฐานข้อมูลร่วม (Atomic Database Transaction)
ระบบจะรันกระบวนการย้อนกลับคะแนนทั้งหมดใน Transaction เดียวเพื่อรักษาความสมบูรณ์ถูกต้องของข้อมูล (Data Consistency) โดยทำตามลำดับขั้นตอนดังนี้:

1. **ดึงประวัติ ledger:** ค้นหาแถวคะแนนที่เคยแจกจ่ายสำหรับฟอร์มนี้โดยเฉพาะในตาราง `score_history` ผ่าน `formId`
2. **หักลบคะแนนบ้าน (Clawback):** นำคะแนน delta ที่บันทึกไว้ในแถวเหล่านั้นไปคำนวณหักลบกลับออกจากตาราง `houses` ทันทีเพื่อคืนสู่คะแนนก่อนหน้าแบบแม่นยำ
3. **ลบประวัติเดิม:** ทำการลบ (DELETE) แถวข้อมูลประวัติคะแนนที่ชี้ไปยัง `formId` นั้นในตาราง `score_history`
4. **รีเซ็ตสถานะฟอร์ม:** อัปเดตสถานะในตาราง `forms`:
   - ตั้งค่า `isAwarded = false` (เพื่อระบุว่าคะแนนถูกดึงคืนและยังไม่ได้ทำการแจกคะแนนครั้งใหม่)
   - ตั้งค่า `isActive = true` (เพื่อเปิดใช้งานฟอร์มให้ส่งคำตอบได้อีกครั้ง)
5. **Commit:** ยืนยันธุรกรรมฐานข้อมูลสำเร็จเป็นอันเสร็จสิ้น

## 3. การจัดการประสิทธิภาพและแคช (Performance Integration)
* ระบบจะทำการยิงคำสั่ง Invalidate Cache (`revalidateTag`) สำหรับ API บอร์ดและคะแนนบ้านทันทีหลังจาก Commit Transaction สำเร็จ
* ช่วยให้คะแนนสะสมและลีดเดอร์บอร์ดบนหน้าจอแสดงผลรวม (Dashboard/Leaderboard) อัปเดตลดลงตามจริงทันทีโดยไม่มีอาการหน่วงหรือแสดงผลขัดแย้งกับฐานข้อมูลจริง
