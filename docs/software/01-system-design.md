## 1. ภาพรวมสถาปัตยกรรมระบบ (System Architecture Overview)
ActiveCAMT พัฒนาด้วยสถาปัตยกรรมแบบ **Modular Monolith** บน **Next.js 16 (App Router)** ร่วมกับ **Drizzle ORM** และฐานข้อมูล **PostgreSQL** โดยระบบประกอบด้วยระบบย่อย (Subsystems) ที่แยกบทบาทหน้าที่กันอย่างชัดเจน ดังนี้:

```
+-------------------------------------------------------------------------+
|                               Next.js App                               |
|  +------------------+  +------------------+  +-----------------------+  |
|  |  Onboarding/UI   |  |   Admin Portal   |  |     Merch Shop UI     |  |
|  +------------------+  +------------------+  +-----------------------+  |
+-------------------------------------------------------------------------+
|                              Service Layer                              |
|  +------------------+  +------------------+  +-----------------------+  |
|  |  Users Service   |  |  Events Service  |  |    Houses Service     |  |
|  +------------------+  +------------------+  +-----------------------+  |
|  +------------------+  +------------------+                             |
|  |  Audit Service   |  |  KAS Form Service|                             |
|  +------------------+  +------------------+                             |
+-------------------------------------------------------------------------+
|                            Data & Middleware                            |
|  +----------------------------------------+  +-----------------------+  |
|  |      NextAuth v5 (Google CMU Auth)     |  |   Realtime SSE + IPC  |  |
|  +----------------------------------------+  +-----------------------+  |
|  +----------------------------------------+  +-----------------------+  |
|  |      Drizzle ORM + PostgreSQL          |  |  Immutable Audit Log  |  |
|  +----------------------------------------+  +-----------------------+  |
+-------------------------------------------------------------------------+
```

## 2. รายละเอียดระบบย่อย (Core Subsystems)

### 2.1 ระบบ Real-time Event Streaming (SSE)
* **วัตถุประสงค์:** ส่งข้อมูลอัปเดต เช่น คะแนนบ้าน, ลีดเดอร์บอร์ด และประวัติเช็คอินไปยังบราวเซอร์ของนักศึกษาและแอดมินโดยตรง โดยมี Latency ต่ำสุด (sub-50ms)
* **กลไกข้าม Process (Cross-Process IPC):** Next.js แยกการทำงานของ Worker Thread ทำให้อีเวนต์แบบ In-Memory ทั่วไปไม่สามารถส่งข้าม Process ได้ ระบบแก้ไขโดยเขียนอีเวนต์สั้นๆ ลงไฟล์ใน `/scratch/realtime-events/` และใช้ `fs.watch` ในการดักจับไฟล์ใหม่เพื่อส่งสัญญาณไปยังฝั่ง Client ผ่าน Server-Sent Events (SSE) และลบไฟล์ทิ้งทันที
* **การกรองข้อมูลตามบทบาทส่วนบุคคล (PDPA Filter):**
  * **Admin / Staff:** ได้รับข้อมูลเช็คอินดิบแบบละเอียด รวมถึงชื่อและสถิติการเช็คอินแบบ Real-time
  * **Student:** ระบบจะกรองข้อมูลเช็คอินออกทั้งหมดเพื่อรักษาความเป็นส่วนตัวตาม PDPA โดยส่งเฉพาะคะแนนบ้านสะสมและข้อความแจ้งเตือนเมื่อมีกิจกรรมใหม่เท่านั้น
  * **ความปลอดภัย:** ส่ง heartbeat ping ทุก 15 วินาทีเพื่อป้องกัน proxy/reverse proxy ตัดการเชื่อมต่อ

### 2.2 ระบบเช็คอินด้วย QR Code ความปลอดภัยสูง (Secure QR Code Check-in)
* **การลงนามโทเค็น QR (Signed Token):** ป้องกันการ Replay Attack และการปลอมแปลง QR Code โดยใช้โทเค็นรูปแบบ `{userId}.{exp}.{signature}`
  * **HMAC-SHA256 Signature:** เข้ารหัสลายเซ็นโดยใช้ `AUTH_SECRET`
  * **Dynamic Window Shift (สลับเวลา):** โทเค็นมีอายุ 5 นาที โดยมีการคำนวณช่วงเวลาเหลื่อมกันรายคน (Dynamic Offset) เพื่อป้องกันไม่ให้นักศึกษาทุกคนส่งคำขอรีเฟรชมาพร้อมกันจนระบบล่ม และยอมรับเวลาคลาดเคลื่อน (Grace Period) ได้ 30 วินาที
  * **Timing-Safe Verification:** ใช้ฟังก์ชันป้องกัน Timing-Side Channel Attack เพื่อไม่ให้บอทคำนวณช่องว่างของความเร็วในการเปรียบเทียบโทเค็นได้
* **การบันทึกสิทธิ์แบบเช็คอินสองระดับ (Scan-Confirm Flow):**
  * **Scan Phase:** เมื่อแอดมินสแกนคิวอาร์ ระบบจะดึงและถอดรหัสข้อมูลสุขภาพ/ความปลอดภัย (Medical Condition Alert) เพื่อเตือนผู้ดูแลงานก่อน
  * **Confirm Phase:** แอดมินกดยืนยันเพื่อบันทึกเวลาจริง (Check-in time) ซึ่งจะประมวลผลแบบเขียนข้อมูลครั้งเดียว (Atomic Update) เฉพาะคนที่มีสถานะ `registered` เท่านั้น
* **ระบบโควต้าป้องกันการแย่งสิทธิ์ (Quota Lock Guard):**
  * ใช้คำสั่ง SQL `FOR UPDATE` (Row Lock) ล็อกแถวข้อมูลที่เกี่ยวกับโควต้าขณะทำการลงทะเบียนและเช็คอิน เพื่อป้องกันสิทธิ์ลงทะเบียนเกิน (Race Condition / TOCTOU)
  * ประมวลผลและตรวจสอบโควต้ารวม โควต้าแยกนักศึกษาไทย/อินเตอร์ และโควต้า Walk-in อย่างแม่นยำ

### 2.3 ระบบฟอร์มแบบประเมินและทดสอบความรู้ (KAS Form Engine)
* **ประเภทฟอร์ม (4 Types):** `K_pre` (ประเมินก่อนกิจกรรม), `K_post` (ประเมินหลังกิจกรรม), `A` (ทัศนคติ), `S` (ทักษะเฉพาะด้าน - ประเมินโดยอาจารย์/ผู้จัดงาน)
* **โครงสร้างการแตกแขนงคำถามแบบมีเงื่อนไข (Section Branching & Conditionals):**
  * ใช้ JSON Schema โครงสร้างแบบ v2 (`{version: 2, sections: []}`) รองรับการกระโดดข้าม Section ตามคำตอบของนักศึกษา (`__next__`, `__submit__` หรือ target section ID) และซ่อนคำถามแบบมีเงื่อนไข (`visibleIf`) พร้อมตรรกะตรวจเช็คและตัดการวนลูปไม่สิ้นสุด (Circular Reference Protection)
  * คิดคะแนนสอบ (Quiz Score) อ้างอิงเฉพาะข้อที่ตั้งค่าเป็น `graded` และเฉพาะข้อใน Section ที่นักศึกษาเดินทางผ่านตามเงื่อนไข Branching จริง
* **กลไกการเปิด-ปิดและส่งคะแนนอัตโนมัติ:**
  * ตรวจสอบสถานะฟอร์มอัตโนมัติตามช่วงเวลา `opensAt`/`closesAt` ของแต่ละฟอร์ม ร่วมกับปุ่ม `isActive` ที่สามารถเปิด-ปิดด้วยมือได้
  * เมื่อจบเวลาประเมินและมีการกดแจกคะแนน (`isAwarded = true`) ระบบจะล็อกไม่ให้ส่งฟอร์มซ้ำเพื่อสะสมคะแนนเพิ่ม โดยใช้ Unique Constraint `(form_id, student_id)` ควบคุม
* **ระบบตรวจสอบการเช็คอิน (Attendance Warning Modal):**
  * ป้องกันนักศึกษาที่ไม่เข้าร่วมกิจกรรมจริงเข้าไปทำฟอร์ม โดยระบบจะเช็คสถานะการเข้าเช็คอิน (`attended`) ก่อนเปิดหน้าฟอร์ม หากไม่ผ่านเกณฑ์จะล็อกปุ่มและแสดงหน้าต่างคำเตือน

### 2.4 ระบบร้านค้าของที่ระลึก (Merch Shop)
* **สินค้าและข้อมูลตัวแปรย่อย (Products & Variants):**
  * สินค้าประกอบด้วยตัวแปรย่อย (เช่น ขนาดเสื้อ, สี) โดยระบุราคาและข้อมูลสต็อกแยกกันอย่างอิสระ
  * เก็บประวัติออเดอร์แบบ Snapshot (เช่น ชื่อและราคา) ไว้ในตารางย่อย เพื่อให้ประวัติการสั่งซื้อไม่เพี้ยนหรือพังแม้จะลบตัวสินค้าออกจากระบบภายหลัง
  * จำกัดสิทธิ์การสั่งซื้อสูงสุดต่อคน (`maxPerOrder`) เพื่อความเท่าเทียม
* **ระบบการชำระเงินและการอนุมัติออเดอร์ (Transaction Slip Flow):**
  * นักศึกษาเลือกสินค้า -> ตรวจสอบสต็อก -> โอนเงินตามข้อมูล PromptPay -> อัปโหลดสลิปธนาคาร
  * **Private Storage Bucket:** เพื่อรักษาความเป็นส่วนตัวตามกฎหมาย PDPA ไฟล์สลิปการโอนเงิน (มีชื่อและเลขบัญชีของผู้โอน) จะไม่ถูกเก็บเป็นลิงก์สาธารณะ แต่จะเก็บในโฟลเดอร์ส่วนตัวและเรียกดูได้เฉพาะแอดมินหรือเจ้าของออเดอร์ผ่าน API Route ที่มีการตรวจสอบสิทธิ์ (Auth Guarded Endpoint) เท่านั้น

### 2.5 ระบบจัดเก็บล็อกความปลอดภัยแบบแก้ไขไม่ได้ (Tamper-Evident Audit Trail)
* **กลไก Hash Chain ย้อนกลับ:**
  * ล็อกความปลอดภัยแต่ละแถวในตาราง `auditLogs` จะถูกเชื่อมโยงกันคล้ายบล็อกเชน โดยทุกครั้งที่บันทึกแถวใหม่ ระบบจะดึงค่า Hash ของแถวก่อนหน้า (`prevHash`) มารวมคำนวณกับฟิลด์แถวใหม่ เพื่อสร้าง Hash ของตัวเอง (`rowHash`) ด้วย SHA-256
  * ใช้ advisory lock ของฐานข้อมูล (`pg_advisory_xact_lock`) เพื่อควบคุมการเขียนล็อกของแต่ละ API ไม่ให้ชนกันและรันต่อเนื่องตามลำดับอย่างแม่นยำ
  * ตรวจสอบความถูกต้องของล็อกได้ผ่านฟังก์ชัน `verifyChainIntegrity()` ซึ่งจะไล่คำนวณ Hash ใหม่ทุกสายเพื่อตรวจหาจุดและเวลาที่โดนแก้ไข
* **ขอบเขตการเก็บข้อมูล:** ครอบคลุมการเข้าดูประวัติสุขภาพนักศึกษาแบบละเอียด, การเปลี่ยนบทบาทสิทธิ์ผู้ใช้, การลบผู้ใช้, การแก้ไขแบบประเมินระบบ KAS, และการเข้าสู่ระบบของแอดมิน
