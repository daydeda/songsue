### Houses / Staff Provisioning
- **ปัญหาเดิม** — `provisionStaffBypass` ใช้ `pickBalancedHouseId()` เดียวกับนักศึกษา ซึ่งนับสมาชิก *ทุกคน*; เพราะจำนวนนักศึกษามากกว่าสตาฟมาก สตาฟจึงถูกแจกบ้านแบบเบี้ยว (มกร 6 / มอม 7 / โต 3 / ลวง 5)
- **`HousesService.pickBalancedHouseIdForStaff()`** (method ใหม่) — query นับเฉพาะ `users` ที่มี `role = "staff"` ต่อบ้าน; บ้านที่มีสตาฟน้อยที่สุดถูกเลือก; ties ตาม query order เหมือน `pickBalancedHouseId`; คืน `null` ถ้ายังไม่มีบ้านในระบบ (caller ปล่อย `houseId` ไว้ว่างแทนที่จะ crash)
- **`UsersService.provisionStaffBypass()`** — เปลี่ยนมาเรียก `pickBalancedHouseIdForStaff()` แทน `pickBalancedHouseId()`; นักศึกษาที่ผ่าน `POST /api/profile` ยังคงใช้ `pickBalancedHouseId()` (total members) เหมือนเดิม ไม่มีอะไรเปลี่ยน
- **`src/lib/staff-bypass.ts`** — เพิ่มสตาฟใหม่ 2 คน: พี่สุขวัญ (`sukwan24.56@gmail.com`) และ พี่วันใหม่ (`chonthicha.ro@gmail.com`); ด้วยจำนวนสตาฟปัจจุบันทั้งสองจะ land ที่บ้านโตเมื่อ login ครั้งแรก ทำให้สตาฟเท่ากัน (→ มกร 6 / มอม 7 / โต 5 / ลวง 5)

## อัปเดตรอบที่ 2 — perf(polling): ลด Active CPU Vercel

ช่วง: 22 มิ.ย. 69 · สรุปไว้สำหรับลง Discord และให้ทีมที่เกี่ยวข้องอ่าน

---

## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **ไม่มีการเปลี่ยนแปลงที่นักศึกษาสังเกตได้** — ข้อมูล Digital ID notification ยังคง refresh ทุก 4 วินาที, scanner และจำนวนเช็คอินยังเป็น real-time เหมือนเดิม ตัวเลขบน admin dashboard อาจช้าลงประมาณ 5–7 วินาที แต่ไม่กระทบการใช้งานในงานจริง

---

## ฝั่งทีม (technical changelog)

### Performance / Serverless CPU

- **ปัญหาเดิม** — Vercel Fluid Active CPU meter ขึ้นไปถึง 75% ของโควตาต่อเดือน (4 CPU-hour) เพราะ polling intervals สั้นทำให้ serverless function ถูก invoke ตลอดเวลา
- **Admin Dashboard (`src/app/admin/dashboard/page.tsx`)**
  - stats poll cadence: **5s → 10s**
  - award-check fire-and-forget nudge: **20s → 60s**
- **Admin Events (`src/app/admin/events/page.tsx`)**
  - events list poll cadence: **8s → 15s**
- ส่วนที่ *ไม่* ถูกแตะ: Digital ID notifications (4s) และ live scanner/attendance counts — ทีมตัดสินใจคงไว้ real-time เพราะใช้ระหว่างงาน

### CDN Caching

- **`GET /api/announcement`** — เดิมมี `export const dynamic = "force-dynamic"` ทำให้ทุก request ตี origin; เปลี่ยนเป็น `Cache-Control: s-maxage=30, stale-while-revalidate=60` (mirror pattern ของ `/api/houses`) ให้ CDN edge serve response แทน ลด cold-start serverless invocation
- **`/api/houses`** — ไม่ถูกแตะ เพราะใช้ caching pattern เดิมอยู่แล้ว

### DB / Migration

- **ไม่มีการเปลี่ยน schema และไม่มี migration**

### PDPA / สิทธิ์เข้าถึง

- **ไม่มีการเปลี่ยน access control** — การเปลี่ยนแปลงทั้งหมดอยู่ที่ timing/caching ไม่มีข้อมูลใหม่ถูกเปิดเผยหรือซ่อน
