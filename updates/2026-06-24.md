## ฝั่งทีม (technical changelog)

### PDPA / สิทธิ์เข้าถึง

- **ไม่มีการเปลี่ยน access control สำหรับไฟล์ PDPA** — `.uploads-private` (สลิปชำระเงิน, ไฟล์ฟอร์ม) ยังอยู่หลัง auth-guarded API เหมือนเดิม; การแก้ครั้งนี้แตะเฉพาะ `uploads` (public assets) ไม่ใช่ `uploads-private`

### Proxy / Middleware

- **ปัญหาเดิม** — บน self-hosted deploy ไฟล์โปสเตอร์/avatar/QR ถูก serve same-origin จาก `/uploads/*`; `matcher` ใน `src/proxy.ts` ครอบคลุม path นี้ด้วย ทำให้ทุก `<img>` request วิ่งผ่าน auth gate และ context ที่ยังไม่ login ถูก redirect 307 → `/login` (HTML page) แทนที่จะได้รูป; ยืนยันบน live container: ไฟล์อยู่บน disk และอ่านได้ปกติ แต่ `GET /uploads/<file>` ได้ 307 /login
- **`src/proxy.ts`** — เพิ่ม `uploads` เข้า exclusion list ของ `matcher` regex (เดิม: `(?!api|_next/static|_next/image|favicon.ico|smocamt-logo.png|smocamt-logo-icon.png|icon.png)` → เพิ่ม `|uploads` ต่อท้าย) พร้อม comment อธิบาย rationale; เปลี่ยนเพียง 1 บรรทัด (matcher string)
- **No-op บน Vercel** — Supabase-backed deploy serve โปสเตอร์จาก cross-origin URL ที่ proxy ไม่เคยเห็น; การเปลี่ยนนี้จึงไม่มีผลใด ๆ บน Vercel

### DB / Migration

## ฝั่งทีม (technical changelog)

### Performance / Infrastructure

- **DB connection pool ปรับได้ด้วย env var `DB_POOL_MAX`** (`src/db/index.ts`) — ตอนนี้ detect อัตโนมัติว่าเชื่อมผ่าน Supabase transaction pooler (port 6543, default pool = 5) หรือ Postgres ที่เราเป็นเจ้าของเอง (direct, default pool = 15); ปรับต่อ host ได้โดยไม่ต้อง rebuild image
- **App-layer caching 15s (shared across all viewers)** — เพิ่ม `unstable_cache` บน 4 endpoint: `/api/houses`, `/api/houses/activity`, `/api/houses/individual`, `/api/announcement`; เดิม `s-maxage` header ไม่มีผลใด ๆ เพราะ nginx ไม่ใช่ CDN ทำให้ทุก poll ตี DB โดยตรง; ตอนนี้ cost ไม่ scale ตามจำนวน viewer แล้ว
- **nginx long-cache สำหรับ `/uploads/`** (`nginx.conf`) — เพิ่ม `Cache-Control: public, max-age=2592000, immutable` บน uploaded media (โปสเตอร์, avatar, QR); ไฟล์แต่ละไฟล์ใช้ UUID ที่ไม่ซ้ำและไม่เคย overwrite จึง cache ได้ถาวรอย่างปลอดภัย
- **Node.js 20 → 22** (`Dockerfile`) — อัปเดต base image

### Real-Time / Live Updates

- **`src/app/dashboard/houses/page.tsx`** — เปลี่ยนจาก one-shot `useEffect` fetch เป็น `usePolling` (30s interval, overlap-safe, pause เมื่อ tab ซ่อน); refresh ทั้ง houses / activity / individual ranking และ `/api/houses/individual/me` (อันดับส่วนตัว) ในรอบเดียวกัน
- **`src/app/api/admin/scan/count/route.ts`** (endpoint ใหม่) — `GET /api/admin/scan/count?eventId=&sessionId=` คืนเฉพาะ aggregate count `{ checkedIn: number }` ไม่มี roster, ไม่มี PII, ไม่ต้องบันทึก audit log — ออกแบบมาเพื่อ poll ถี่ ๆ ระหว่างงาน; gate สิทธิ์เหมือน scan POST (รวม `club_president`, `major_president` ที่มี scope จำกัดตาม `managedByRoles`)
- **`src/app/admin/scanner/page.tsx`** — เพิ่มกล่อง "เช็คอินแล้ว / Checked In" พร้อม live indicator (จุดเขียวกระพริบ); poll `/api/admin/scan/count` ทุก 8 วินาที และ refresh ทันทีหลัง scan สำเร็จแต่ละครั้ง; reset count เป็น `—` เมื่อเจ้าหน้าที่เปลี่ยน event หรือ session
- **`src/app/api/admin/houses/points/route.ts`** — เพิ่ม `revalidateTag("house-standings")` หลัง manual point adjustment สำเร็จ ทำให้ leaderboard แสดงคะแนนใหม่ทันทีโดยไม่รอให้ cache หมดอายุ 15 วินาที (best-effort; ถ้า purge ล้มเหลว TTL จะ catch up เอง)

### PDPA / สิทธิ์เข้าถึง

- **`/api/admin/scan/count` ไม่แตะข้อมูลส่วนบุคคลเลย** — คืนเฉพาะตัวเลขนับ aggregate; ออกแบบให้แยกออกจาก `/api/admin/events/[id]/attendance` ที่อ่าน full roster รวม medical detail และบันทึก PDPA audit log ทุกครั้ง; endpoint นี้จึงไม่ต้องมี audit log และ poll ได้ถี่ ๆ โดยไม่เพิ่มโหลด audit trail
- **`src/lib/logger.ts`** — structured JSON logging พร้อม deep-redact key ที่ sensitive (medical fields, emergency contacts, token, cookie, secret ฯลฯ) ก่อน emit ทุกครั้ง; `captureException` เป็น single chokepoint สำหรับ error monitoring — รองรับ optional `ERROR_WEBHOOK_URL` (Discord webhook หรือ JSON endpoint) แบบ fire-and-forget; ปลอดภัยต่อ PDPA เพราะ redact ข้อมูลก่อนส่ง

### Tests + CI

- **Vitest bootstrap** (`vitest.config.ts`, `package.json`) — unit test suite ชุดแรกของโปรเจกต์; 96 tests pass ครอบคลุม 3 module หลัก:

  - `src/lib/qr-token.test.ts` — QR token sign / verify / tamper detection
  - `src/modules/audit/audit-chain.test.ts` — audit hash-chain integrity (append-only, tamper-evident)
- **`.github/workflows/ci.yml`** (ไฟล์ใหม่) — CI gate บน PR ทุกตัวที่เข้า `main`: hard gate คือ unit tests + production build; lint เป็น informational (มี pre-existing errors ที่ไม่เกี่ยวกับงานปัจจุบัน จะ clear แล้วเปลี่ยนเป็น hard gate ในรอบถัดไป)

### DB / Migration

- **ไม่มีการเปลี่ยน schema และไม่มี migration** — เป็นการ optimize และเพิ่ม layer บนของ DB เท่านั้น

## ฝั่งทีม (technical changelog)

### Scanner / Real-Time

- **ปัญหาเดิม** — กล่อง "Checked In" live counter ที่หน้า `/admin/scanner` (เพิ่งเปิดตัวใน PR #77) นับ attendance row ทุกแถวของ event โดยไม่กรอง status; ตาราง `attendance` เป็น dual-purpose: pre-registration จะ insert row ด้วย `status = 'registered'` (method `'pre-registered'`) ก่อน event เปิด แล้วเมื่อนักศึกษาสแกน QR จริงจึงจะเปลี่ยนเป็น `status = 'attended'`; ผลคือสำหรับ event ที่เปิด pre-registration ตัวเลขจะแสดงยอด "ลงทะเบียนแล้ว" เต็มทันทีที่ registration เปิด และไม่ขยับขึ้นเลยเมื่อนักศึกษาทยอยสแกนเข้างาน
- **`src/app/api/admin/scan/count/route.ts`** — เพิ่ม `eq(attendance.status, "attended")` เข้าใน `where(...)` ทั้ง 2 branch (per-session และ event-wide); ตอนนี้ endpoint คืนเฉพาะคนที่ผ่านการสแกน QR จริง และตัวเลขขยับขึ้น live ทุกครั้งที่มีนักศึกษาสแกนสำเร็จ
- **Walk-in events ไม่ได้รับผลกระทบ** — event ที่ไม่มี pre-registration จะ insert row ด้วย `status = 'attended'` ตั้งแต่แรก; ก่อนและหลัง fix ตัวเลขจึงถูกต้องเหมือนกัน

### PDPA / สิทธิ์เข้าถึง

- **Read-path เท่านั้น — ไม่มีการเปลี่ยน access control** — endpoint ยังคง gate สิทธิ์เหมือนเดิม (scan POST กลุ่มเดิมรวม `club_president`, `major_president`) และยังคืนเฉพาะตัวเลข aggregate `{ checkedIn: number }` โดยไม่มี PII; ไม่ต้องบันทึก audit log

### DB / Migration

- **ไม่มีการเปลี่ยน schema และไม่มี migration** — เปลี่ยนเฉพาะ query filter ใน route handler; `status` column มีอยู่แล้วใน `attendance` table
