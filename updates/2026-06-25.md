## ฝั่งทีม (technical changelog)

### Admin / Events — Attendance View

- **ปัญหาเดิม** — banner "Filtered: Showing X of Y records" ในหน้า Admin → Events → Attendance อ่านค่าจาก array ดิบ `filteredAttendance.length` / `attendance.length` ซึ่งเก็บข้อมูล **หนึ่งแถวต่อหนึ่งเซสชัน/วัน**; กิจกรรมหลายวัน (เช่น CAMT LINK ALL AROUND US, 2 วัน) จึงนับแถวซ้ำสำหรับนักศึกษาที่เข้าร่วมทั้ง 2 วัน — banner แสดง "521 of 601" แทนที่จะเป็นจำนวนคนจริง
- **root cause** — การ์ดรายชื่อ (`attendanceUnits`) และตัวเลขหัวคอลัมน์ (`tallyUnits`) ทำ deduplicate โดย studentId ไว้แล้ว; มีแค่ banner นี้เท่านั้นที่ยังชี้ไปที่ raw array เดิม
- **`src/app/admin/events/page.tsx`** — เปลี่ยน 1 บรรทัด:
  - `filteredAttendance.length` → `attendanceUnits.length` (คนที่ผ่าน filter ที่เลือก, distinct per student)
  - `attendance.length` → `tallyUnits.length` (คนทั้งหมด distinct, ตรงกับ `registeredCount`)
  - ทั้งสองค่า honor filter "วันที่เลือก" เช่นเดิม — banner จึงตรงกับการ์ดรายชื่อและตัวเลขหัวคอลัมน์ทุกกรณี

### PDPA / สิทธิ์เข้าถึง

- **ไม่มีการเปลี่ยน access control, PDPA gating หรือ audit log** — เป็นการแก้ไข display logic เท่านั้น; ข้อมูลที่อ่านและ endpoint ที่ใช้ยังเป็นชุดเดิม; audit trail ยังทำงานตามปกติทุกครั้งที่เปิดหน้า Attendance

### DB / Migration

- **ไม่มีการเปลี่ยน schema และไม่มี migration** — แก้ไขเฉพาะ client-side derived state ใน page component

## ฝั่งทีม (technical changelog)

### Admin / Forms — Re-open & Point Revert

- **ฟีเจอร์หลัก** — PATCH `/api/admin/events/[id]/form` ตรวจจับกรณี re-open (ฟอร์มอยู่ใน state `isAwarded = true` และ `closesAt` ถูกตั้งให้เป็นอนาคต); เมื่อตรวจพบจะ: (1) เรียก `revertFormAward()` ถอนคะแนนเดิมออก (2) flip `isAwarded = false`, `isActive = true` ใน transaction เดียว (3) บันทึก audit log การถอนคะแนน (4) bust leaderboard cache; ฟีเจอร์นี้จำกัดเฉพาะ `admin` และ `super_admin` เท่านั้น — role อื่นถูก gate ไว้ก่อนเข้าถึง logic นี้
- **`src/lib/award-points.ts`** — เพิ่ม helper `revertFormAward(tx, formId)`: ดึง ledger rows ทั้งหมดที่มี `form_id = formId`, คำนวณผลรวม net ต่อ House, ลบคะแนนออกจาก `score_history`, และลบ rows เดิมทิ้ง; idempotent — เรียกซ้ำบน formId เดิมไม่มีผลข้างเคียง; แก้ไข `awardFormPoints()` ให้ stamp `form_id` ลงบน ledger row ทุกแถวที่สร้างขึ้น (เดิม `form_id` เป็น null เสมอ)
- **`src/app/api/admin/events/[id]/form/route.ts`** — เพิ่ม re-open detection + revert flow ด้านบน; ไม่มีการเปลี่ยน logic สำหรับ path ปกติ (สร้าง/แก้ฟอร์มที่ยังไม่ awarded)

### PDPA / สิทธิ์เข้าถึง

- **Re-open gate — `admin` / `super_admin` เท่านั้น** — PATCH handler ตรวจ role ก่อน execute revert; `smo`, `organizer`, `registration` และ role อื่น ๆ ยังถูก reject ด้วย 403 เหมือนเดิม; access control บน route นี้ไม่เปลี่ยนจากเดิมสำหรับทุก role ยกเว้น path ใหม่ที่ตรวจ re-open
- **Audit log** — ทุก revert บันทึก audit entry ผ่าน `AuditService` ใน transaction เดียวกัน; ทีมสามารถตรวจสอบย้อนหลังได้ว่าใครสั่ง re-open เมื่อไหร่และมีผลอะไรกับคะแนน

### UI — Events Page

- **`src/app/admin/events/page.tsx`** — ปลดล็อค field "Closes at" และ "Opens at" ให้แก้ได้เมื่อฟอร์มอยู่ใน state `isAwarded = true` (เดิม disabled); เพิ่ม amber warning banner แจ้งว่า "ฟอร์มนี้ awarded แล้ว การขยายเวลาจะถอนคะแนนและเปิดใหม่"; เพิ่ม confirm dialog 4 ภาษา (EN/TH/MM/CN) ก่อนบันทึก re-open
- **ค่า default "House Points Reward" เปลี่ยนจาก 50 → 0** — ป้องกัน Admin ที่ลืมตั้งค่าจาก silently award คะแนน 50 โดยไม่ได้ตั้งใจ; ต้องตั้งค่าตัวเลขใหม่ทุกครั้งอย่างตั้งใจ

### Houses / Leaderboard

- **cache bust เมื่อ revert** — เรียก `revalidateTag("house-standings")` หลัง revert transaction สำเร็จ; leaderboard แสดงคะแนนที่ถูกถอนคืนทันทีโดยไม่รอ TTL 15 วินาที (best-effort)

### DB / Migration

- **`src/db/schema.ts`** — เพิ่มคอลัมน์ `form_id` (nullable uuid FK → `forms`, `ON DELETE SET NULL`) และ index บน `score_history`; `ON DELETE SET NULL` ออกแบบมาเพื่อให้ลบ form แม่ได้โดย ledger rows ประวัติยังอยู่ครบ แค่ตัดความสัมพันธ์
- **`src/db/migrate.ts`** — step 44 (idempotent, additive): `ADD COLUMN IF NOT EXISTS form_id`, guarded FK add (ตรวจ `pg_constraint` ก่อน add), `CREATE INDEX IF NOT EXISTS`; ไม่มี DROP หรือ DELETE — non-destructive ทุกขั้นตอน; ต้องรัน `npm run db:migrate:container` จาก Portainer console ก่อน deploy image ใหม่
- **`drizzle/0007_real_unicorn.sql`** — snapshot migration สำหรับ Drizzle Kit; ไม่ได้ใช้งาน migration บน self-hosted (ใช้ `migrate.ts` โดยตรง) แต่อัปเดตเพื่อให้ Drizzle schema state ตรงกัน

## ฝั่งทีม (technical changelog)

### Admin / Forms — Multi-day Attendance Gate Fix

- **root cause** — `POST /api/events/[id]/form` ใช้ `db.query.attendance.findFirst({ where: and(eventId, studentId) })` โดยไม่มี `orderBy`; Postgres คืนแถวตามลำดับที่ไม่แน่นอน — กิจกรรม 2 วันมี 2 attendance rows ต่อคน; ถ้า `findFirst` ดึงแถววันที่ 2 ซึ่งยัง `status = "registered"` ขึ้นมา guard `attRecord?.status !== "attended"` จะ reject นักศึกษาที่เช็คอินวันที่ 1 ไปแล้วอย่างผิดพลาด
- **แก้ไข** — เพิ่ม `eq(attendance.status, "attended")` เข้าใน `where` clause และเปลี่ยน condition เป็น `if (!attRecord)` — query จึงค้นหา "มีแถวใดก็ได้ใน event นี้ที่ studentId นี้ attended หรือเปล่า"; ถ้ามีอย่างน้อยหนึ่งวัน = ผ่าน gate; ถ้าไม่มีเลย = 403
- **scope** — ใช้กับ form type ที่ต้องเช็คอินทุกประเภท: K_post, A, F; `K_pre` และ `S` ข้าม attendance check อยู่แล้ว — ไม่กระทบ
- **`src/app/api/events/[id]/form/route.ts`** — แก้ไข 12 บรรทัด (เพิ่ม where clause + เปลี่ยน condition check); เพิ่ม comment อธิบาย "one row per session" ไว้ใน code; ไม่มีการเปลี่ยน logic อื่น
- **pending-forms banner** — `GET /api/forms/pending` ใช้ `attended.has(f.eventId)` ซึ่งตรวจสอบ Set ที่สร้างจาก distinct eventId ของแถวที่ `status = "attended"` ถูกต้องอยู่แล้วก่อน PR นี้ — submit gate และ banner banner จึง consistent กันแล้ว

### PDPA / สิทธิ์เข้าถึง

- **ไม่มีการเปลี่ยน access control, PDPA gating หรือ audit log** — แก้ไขเฉพาะ attendance query logic ใน submit gate; gate ยัง enforce อยู่สำหรับทุก type ที่ต้องเช็คอิน; ไม่มี endpoint ใหม่ ไม่มีข้อมูลใหม่ที่เปิดเผย
