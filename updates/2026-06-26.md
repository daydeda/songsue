## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **เช็คอินกิจกรรมแล้วได้คะแนนส่วนตัวทันที** — กิจกรรมบางรายการตอนนี้สามารถกำหนด "คะแนนรายคน" แยกออกมาจากคะแนน House ได้ หากผู้จัดตั้งค่าไว้ ทุกคนที่สแกน QR เข้าร่วมสำเร็จจะได้รับคะแนนส่วนตัว (แสดงใน Profile ของตัวเอง) ทันทีในครั้งนั้น — ไม่ต้องรอให้กิจกรรมจบและไม่ต้องแข่งกับ House อื่น
- คะแนนนี้นับแยกกันทุกวันสำหรับกิจกรรมที่มีหลายวัน — เข้าร่วมครบทุกวันก็ได้คะแนนสะสมมากขึ้น
- **ระบบ milestone ยังคงทำงาน** — ถ้าคะแนนสะสมส่วนตัวของนักศึกษาข้ามเส้น 100 คะแนน บ้านของนักศึกษาจะได้ +2 คะแนน House โบนัสตามปกติ
- คะแนน House แบบเดิม (รางวัลให้ House ที่ชนะเมื่อปิดกิจกรรม) ยังคงอยู่ครบถ้วน ระบบทั้งสองทำงานคู่กัน

## ฝั่งทีม (technical changelog)

### Events / Scanner — Individual Points per Attendee

- **ฟีเจอร์หลัก** — เพิ่ม field ใหม่ `individual_points_awarded` บน event; เมื่อ check-in กลายเป็น `attended` (transition จริง) สคริปต์ `ScannerService.awardAttendanceIndividualPoints()` ที่รันใน transaction เดียวกันจะ increment `users.points` ทันที; ครอบคลุมทุก path ของ check-in:
  - Pre-registered confirm (once-mode)
  - Pre-registered confirm (multi-session)
  - Walk-in
- **Idempotent** — ฟังก์ชันรันเฉพาะเมื่อ query update attendance คืน `rows.length > 0` (attendance row transition สำเร็จจริง); re-scan ที่ไม่ได้ transition จะไม่ award ซ้ำ; กิจกรรมที่ตั้ง 0 points จะ no-op ทันทีด้วย early-return
- **Milestone bonus** — logic `Math.floor(newPoints / 100) - Math.floor(previousPoints / 100)` เหมือนกับ manual score action ทุกประการ; ทุก milestone ที่ข้ามจะ +2 house points พร้อม `score_history` row
- **`src/modules/events/scanner.service.ts`** — เพิ่ม private static method `awardAttendanceIndividualPoints(tx, params)` (+68 บรรทัด) และ 3 call sites ใน each attended-transition block; ไม่มีการเปลี่ยน logic check-in เดิม; transaction boundary ไม่เปลี่ยน

### Admin / Events — Create / Edit Form

- **`src/app/admin/events/page.tsx`** — field "House Points Awarded" และ "Individual Points Awarded" แสดงเป็น 2 input แยกกัน พร้อม hint text อธิบาย mechanic ของแต่ละอย่าง; event card แสดง badge `+N EACH` เมื่อ `individualPointsAwarded > 0`; i18n key ใหม่ wire แล้วในทุก call site
- **`src/app/api/admin/events/route.ts` (POST) + `src/app/api/admin/events/[id]/route.ts` (PUT)** — รับและ persist field `individualPointsAwarded` ใหม่; validation ผ่าน zod schema เดิม

### Mobile/UI — i18n

- **`src/lib/i18n.ts`** — เพิ่ม key ใหม่ครบ 4 ภาษา (EN/TH/MM/CN) สำหรับ label และ hint ของ Individual Points field, badge `+N EACH`, และ tooltip อธิบาย mechanic

### DB / Migration

- **`src/db/schema.ts`** — เพิ่มคอลัมน์ `individualPointsAwarded` (`integer DEFAULT 0`) บนตาราง `events`; comment อธิบาย mechanic ของ house points vs individual points ไว้ครบแล้วในไฟล์; ไม่มีการเปลี่ยน column เดิม
- **`src/db/migrate.ts`** — step 45: `ALTER TABLE events ADD COLUMN IF NOT EXISTS individual_points_awarded integer DEFAULT 0`; additive, idempotent, non-destructive; existing events default เป็น 0 (ไม่มีผลกระทบต่อกิจกรรมที่มีอยู่เดิม)
- **`drizzle/0014_event_individual_points.sql`** — snapshot migration สำหรับ Drizzle Kit; ไม่ได้ใช้บน self-hosted (ใช้ `migrate.ts` โดยตรง) แต่อัปเดตเพื่อให้ Drizzle schema state ตรงกัน
- **ต้องรัน `npm run db:migrate:container` จาก Portainer console ก่อน deploy image ใหม่** — code ที่อ่าน `individualPointsAwarded` พร้อม deploy ได้หลัง migration เสร็จเท่านั้น; migration non-destructive ทุกขั้นตอน

### PDPA / สิทธิ์เข้าถึง

- **ไม่มีการเปลี่ยน access control, PDPA gating หรือ audit log สำหรับข้อมูลทางการแพทย์** — `awardAttendanceIndividualPoints` ไม่ได้อ่านหรือเปิดเผยข้อมูล medical detail; การอ่าน `hasMedicalCondition` และ audit log check-in ยังทำงานเหมือนเดิมทุกประการ; gate ของ field ใหม่ใน Create/Edit Event อยู่หลัง admin auth เช่นเดิม
