## ฝั่งทีม (technical changelog)

### Infra / Edge Proxy — Retired Vercel Deploy Fix

- **Root cause** — `src/proxy.ts` (edge middleware) import `@/auth` ที่ module-load time; `auth.ts` มี fail-fast guard ที่ throw `Error` เมื่อ `AUTH_URL` ไม่ถูกตั้งค่าในโหมด production; Vercel deploy ที่ retire ไปแล้วไม่มี env var นี้ → module crash ก่อนที่ request จะถึง `MovedNotice` gate ใน `layout.tsx`; Next.js catch edge runtime error และส่ง bare HTTP 500 กลับโดยไม่มี `Content-Type` header + `nosniff` → เบราว์เซอร์ treat เป็นไฟล์และดาวน์โหลดทันที

- **`src/lib/site-moved.ts`** (ไฟล์ใหม่, +22 บรรทัด) — helper เดียวที่ share ข้าม proxy / auth / layout: `isSiteMoved(): boolean`; อ่านเฉพาะ `process.env` ไม่ import อะไรเลย (edge-safe และ dependency-free โดยเจตนา); logic: `NEXT_PUBLIC_SITE_MOVED === "1"` หรือ (`VERCEL === "1"` และ `NEXT_PUBLIC_SITE_MOVED !== "0"`); ไม่มี hardcoded fallback — self-hosted Docker ไม่มี `VERCEL` env จึง return `false` เสมอ

- **`src/auth.ts`** (+7 บรรทัด) — เพิ่มเงื่อนไข `!isSiteMoved()` ใน fail-fast guard ของ `AUTH_URL`; guard ยังคง enforce ครบถ้วนบน self-hosted production เหมือนเดิม; bypass เฉพาะบน retired deploy เพื่อให้ module โหลดได้โดยไม่ crash

- **`src/proxy.ts`** (+9 บรรทัด) — short-circuit ก่อนทำงานใด ๆ เมื่อ `isSiteMoved()`: return `NextResponse.next()` ทันที ไม่เรียก `auth()` หรือเข้าถึง DB → request ผ่านต่อไปถึง `layout.tsx` ซึ่งแสดง `<MovedNotice />` แทน; ทุก route รวมถึง `/admin` จะได้รับหน้า notice แทนที่จะเจอ dead DB connection

- **`src/app/layout.tsx`** (−8 +6 บรรทัด net) — ลบ inline flag definition ออกแล้วเปลี่ยนเป็น `import { isSiteMoved }` จาก `@/lib/site-moved`; logic ไม่เปลี่ยน แต่ตอนนี้ใช้ source of truth เดียวกันกับ proxy และ auth

### DB / Migration

- **ไม่มี DB migration ใหม่** — ไม่มีการเพิ่มหรือเปลี่ยนคอลัมน์; ไม่ต้องรัน `db:migrate:container`

### PDPA / สิทธิ์เข้าถึง

- **ไม่มีการเปลี่ยน access control, PDPA gating หรือ audit log** — fix แตะเฉพาะ edge proxy bootstrap logic บน retired deploy; ไม่มี read path ใหม่สำหรับ medical detail; gate ทุกชั้นบน self-hosted ยังคงอยู่ครบถ้วน

## อัปเดตรอบที่ 2 — fix(uploads): compress รูปในเบราว์เซอร์ก่อนส่ง รับมือ reverse-proxy body cap (413)

ช่วง: 28 มิ.ย. 69 · สรุปไว้สำหรับลง Discord และให้ทีมที่เกี่ยวข้องอ่าน

## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **แก้บั๊ก: อัปโหลดสลิปชำระเงินในร้านค้า (dashboard/shop) ที่แสดง "Internal Server Error" มาตลอด** — รูปถ่ายจากโทรศัพท์ขนาด 2–5MB ถูก reverse proxy ของเซิร์ฟเวอร์มหาวิทยาลัยปฏิเสธด้วย 413 ก่อนที่ request จะถึง Next.js เลย — ทำให้ไม่มีรอยใน log และไม่มีการแจ้งเตือนใน Discord แต่อย่างใด; ตอนนี้รูปถูก compress ให้เล็กลงในเบราว์เซอร์ก่อนส่งอัตโนมัติ — สลิปอัปโหลดได้ตามปกติ
- **การอัปโหลดรูปทุกจุดในระบบปลอดภัยจาก 413 แล้ว** — รูปโปรไฟล์, รูปสมัครสมาชิก (onboarding), รูปสินค้าและ QR ชำระเงิน (Admin shop), และไฟล์รูปภาพแนบในฟอร์มกิจกรรม ทั้งหมดผ่านการ compress ก่อนส่ง — ลด latent bug เดิมที่มีอยู่ทุกจุด (รูปโปสเตอร์กิจกรรม admin ทำงานได้เพราะมี compress ก่อนแล้ว; ที่เหลือไม่มี = bug แฝง)
- **ถ้ารูปยังใหญ่เกินไปหลัง compress ระบบแสดงข้อความที่อ่านเข้าใจได้** — "รูปภาพมีขนาดใหญ่เกินไป กรุณาเลือกรูปที่เล็กลง" ครบ 4 ภาษา (TH/EN ทุกหน้า; TH/CN/MM/EN เพิ่มเติมบนหน้า forms) แทนที่จะแสดง error ลึกลับที่ไม่ได้บอกว่าเกิดอะไรขึ้น

## ฝั่งทีม (technical changelog)

### Uploads / Client-side Compression

- **`src/lib/compress-image.ts`** (ไฟล์ใหม่, +78 บรรทัด) — shared browser-side helper: `compressImageFile(file, { maxDim, quality })` downscale ด้วย Canvas → re-encode เป็น WebP; ไม่ throw เลย (fallback คืน original file เสมอ เพื่อไม่ให้ upload path พัง); ข้ามไฟล์ที่ไม่ใช่ raster image และ animated GIF (canvas loop จะทำลาย animation); mirrors pipeline ที่ admin event-poster uploader ใช้อยู่แล้วและพิสูจน์แล้วว่าใช้งานได้

  - **`src/app/dashboard/shop/ShopClient.tsx`** (+19, −7 บรรทัด) — payment slip upload → `maxDim 1600`; ข้อความบนสลิปและ bank QR ยังอ่านได้/สแกนได้

  - **`src/app/admin/shop/AdminShopClient.tsx`** (+12, −5 บรรทัด) — product image upload + payment QR image upload → `maxDim 1600`; QR ต้องการ resolution พอสแกนได้ (ตรวจสอบเชิงประจักษ์ก่อน ship)
  - **`src/app/onboarding/OnboardingClient.tsx`** (+19, −8 บรรทัด) — signup photo → `maxDim 512`
  - **`src/app/dashboard/history/page.tsx`** (+12, −5 บรรทัด) — form "file" answer upload → `maxDim 1600`; PDF pass through โดยไม่แตะ (canvas compress PDF ไม่ได้ — PDF ขนาดใหญ่ยังอาจ 413 แยกต่างหากถ้า proxy cap ต่ำมาก แต่เป็น separate concern)

### Uploads / Client Error Handling

- **Guard `res.json()` บน proxy rejection** — proxy 413 คืน HTML body ไม่ใช่ JSON; `res.json()` เดิมจะ throw `SyntaxError: Unexpected token '<'` ซึ่ง surface เป็น error ลึกลับ; แก้โดย check `res.ok` ก่อน parse และ branch บน `res.status === 413` เพื่อ throw message "image too large" ที่อ่านเข้าใจได้ทุก call site ที่แก้ไข; กระทบทุก image upload path ที่ listed ด้านบน
- **i18n** — ข้อความ "image too large" ใน TH/EN ทุกหน้า; หน้า history (forms) เพิ่ม CN/MM ด้วยเพราะหน้านี้รองรับ 4 ภาษาอยู่แล้ว

### DB / Migration

- **ไม่มี DB migration ใหม่** — pure client-side utility + error handling; ไม่มีการเพิ่มหรือเปลี่ยนคอลัมน์; ไม่ต้องรัน `db:migrate:container`

### PDPA / สิทธิ์เข้าถึง

- **ไม่มีการเปลี่ยน access control, PDPA gating หรือ audit log** — server ยังคง re-encode และ harden ทุก upload ผ่าน `src/lib/image-upload.ts` เหมือนเดิม; stored image quality ไม่เปลี่ยน (server cap 1600px/q0.8 อยู่แล้ว); นี่คือ size/UX fix ไม่ใช่ security boundary; ไม่มี read path ใหม่สำหรับ medical detail

- **`src/app/admin/shop/AdminShopClient.tsx`** (+102, −9 บรรทัด) — เพิ่ม pagination 10 รายการต่อหน้าบนทั้งรายการสินค้า (Products list) และรายการคำสั่งซื้อ (Orders list); ปุ่ม Prev / Next พร้อม indicator หน้าปัจจุบัน — ซ่อนอัตโนมัติเมื่อมีแค่ 1 หน้า; ใช้เทคนิค derived `safePage` clamp + adjust-state-during-render เพื่อ reset หน้าที่ 1 โดยอัตโนมัติเมื่อเปลี่ยน filter หรือ tab — ไม่ใช้ `useEffect` (สอดคล้องกับ lint rule ของ project ที่ห้าม set-state-in-effect)

### Shop / Admin — Orders Filters

- **Filter dropdown สินค้า** — ทีมงานเลือกดูเฉพาะคำสั่งซื้อที่มีสินค้ารายการที่เลือกอยู่ใน order (inclusive match กับทุก line item ของ order); ข้อมูล dropdown ดึงจาก product list ที่ fetch ไว้แล้วในหน้าเดิม — ไม่มี API endpoint ใหม่
- **Filter ช่วงวันที่/เวลา (From / To)** — กรอง `created_at` ของ order แบบ inclusive ทั้งสองปลาย; input type `datetime-local`; ถ้าไม่กรอก From หรือ To ถือว่าไม่จำกัดด้านนั้น — สามารถใช้ปลายเดียวก็ได้
- **ปุ่ม "Clear filters"** — แสดงเฉพาะเมื่อมี product filter หรือ date filter ที่ active อยู่; กดครั้งเดียว reset dropdown และ date inputs ทั้งสองพร้อมกัน
- **Product + date filters ทำงานก่อน status tabs** — count badge ใน tab (Pending / Approved / Rejected / All) สะท้อนจำนวน order ที่ผ่าน filter ที่ตั้งไว้ ไม่ใช่จำนวนทั้งหมด; ทีมงานเห็นทันทีว่ากรอง product X ออกมาแล้วมี order pending อีกกี่รายการ
- Filter ทั้งหมดทำงาน client-side บน order list ที่ fetch ไว้แล้ว — ไม่มี query DB ใหม่ ไม่มี API ใหม่

### DB / Migration

- **ไม่มี DB migration ใหม่** — pure UI / client-side change; ไม่มีการเพิ่มหรือเปลี่ยนคอลัมน์; ไม่ต้องรัน `db:migrate:container`

### PDPA / สิทธิ์เข้าถึง

- **ไม่มีการเปลี่ยน access control, PDPA gating หรือ audit log** — feature อยู่ใน admin shop UI layer; admin auth gate ยังคงเป็น requirement เดิมครบถ้วน; ไม่มี read path ใหม่สำหรับ medical detail; audit log สำหรับ admin อ่าน order ที่เพิ่มใน PR #115 ยังทำงานครบโดยไม่เปลี่ยนแปลง

## ฝั่งทีม (technical changelog)

### Shop / Admin — Orders Product Filter UX

- **`src/app/admin/shop/AdminShopClient.tsx`** (+79, −5 บรรทัด) — แทนที่ `<select>` native ของ product filter ใน Orders tab ด้วย component `FilterDropdown` ที่เขียนไว้ inline ในไฟล์เดียวกัน; visual เข้ากันกับ input style และ chip ที่ใช้ในหน้า Admin Shop อยู่แล้ว
- **`FilterDropdown` component** มีฟีเจอร์ดังนี้:
  - **chevron หมุน** — ไอคอน `ChevronDown` หมุน 180° เมื่อ dropdown เปิด (CSS transition 0.2 s) บ่งบอก state เปิด/ปิดชัดเจน
  - **styled dropdown panel** — แสดงด้านล่าง trigger button, `zIndex 50`, shadow, `borderRadius` ตาม design token `--radius-md`; แต่ละ option row มี hover highlight (`--bg-elevated`) และ checkmark icon (`Check` size 15) บน option ที่ active อยู่ — option active แสดงสีและ font-weight ต่างจากที่เหลือ
  - **ปิดเมื่อคลิกนอก dropdown (outside-click)** — `mousedown` listener บน `document` ร่วมกับ `ref` ของ container; listener register เฉพาะเมื่อ dropdown เปิด (cleanup ใน `useEffect` return)
  - **ปิดด้วยปุ่ม Escape** — `keydown` listener คู่ขนานกับ outside-click, cleanup เหมือนกัน
  - **scroll + truncate ชื่อสินค้ายาว** — panel มี `maxHeight 280px` + `overflowY: auto`; ชื่อ option ตัดด้วย `textOverflow: ellipsis` ทั้งใน trigger และใน list
- **พฤติกรรมการ filter ไม่เปลี่ยน** — option list ยังเป็น "ทุกสินค้า / All products" + ชื่อสินค้า distinct เหมือนเดิม; การเลือกยัง reset page กลับหน้า 1 เหมือนเดิม; ดึงข้อมูลจาก product list ที่ fetch ไว้แล้ว ไม่มี API call ใหม่
