## ฝั่งทีม (technical changelog)

### Admin / Scanner — Live Event Reminder (แก้ incident สแกนผิดกิจกรรม)

- **Context / root cause** — เคยเกิดเหตุจริงที่เจ้าหน้าที่สแกนเช็คชื่อโดยไม่ทันสังเกตว่า dropdown เลือกกิจกรรมค้างอยู่ที่กิจกรรมผิด ทำให้ attendance ถูกบันทึกเข้ากิจกรรมผิดตัว — PR นี้แก้ที่ต้นตอ UX ของหน้า scanner ทั้งหมด
- **`src/app/admin/scanner/page.tsx`** (+156, −22 บรรทัด) เป็นไฟล์เดียวที่แก้:
  - **`isEventLive(e)`** helper ใหม่ — เช็คว่า `now` อยู่ในช่วง `startTime`–`endTime` ของกิจกรรมหรือไม่ (mirror logic เดียวกับที่หน้า `admin/events` ใช้อยู่แล้ว); type `Event` เพิ่ม field `startTime`/`endTime` เพื่อใช้เช็คนี้
  - **Default event ตอนโหลดหน้า** — เปลี่ยนจาก default = กิจกรรมแรกใน list (newest) มาเป็น `events.find(isEventLive) ?? d[0]` — ถ้ามีกิจกรรม live อยู่ตอนนี้ scanner จะเปิดมาเลือกกิจกรรมนั้นให้อัตโนมัติ
  - **`sortedForScanner(events)`** — จัดกิจกรรมที่ live ให้ลอยขึ้นบนสุดของทั้งปุ่ม selector และ dropdown list (ไม่กระทบ order ของกิจกรรมที่ไม่ live ต่อกัน — คง server's desc-startTime order เดิม)
  - **`LiveBadge`** component ใหม่ — ป้าย "LIVE" สีเขียวพร้อมจุดกะพริบ (`animate-pulse`); reuse ทั้งบนปุ่มเลือกกิจกรรม, แต่ละแถวใน dropdown list, และ overlay บนกล้อง
  - **Warning card เตือนเปลี่ยนกิจกรรม** — ถ้ากิจกรรมที่เลือกอยู่ไม่ live แต่มีกิจกรรมอื่น live อยู่ ณ ขณะนั้น (`otherLiveEvent`) จะโชว์การ์ดสีเหลืองอำพันพร้อมข้อความเตือนครบ 4 ภาษา (TH/EN/CN/MM) และปุ่ม "เปลี่ยนไปกิจกรรมที่กำลังจัดอยู่" กดครั้งเดียวสลับ `eventId` + `sessionId` ให้ทันที
  - **Overlay บนกล้องขณะสแกน** — เดิม overlay ตอน `isScanning` โชว์แค่ badge "SCANNING" มุมซ้ายบน; ตอนนี้เปลี่ยนเป็นแถบเต็มความกว้างด้านบนกล้อง แสดงชื่อกิจกรรมที่เลือกอยู่ (+ label วัน/session เมื่อเป็นกิจกรรมหลายวัน) ตลอดเวลาที่กล้องเปิดอยู่ — สีเข้ม (`rgba(0,0,0,0.55)`) เมื่อกิจกรรมนั้น live, สีเหลืองอำพัน (`rgba(180,83,9,0.75)`) เมื่อไม่ live — ทำให้ตัวเตือนอยู่ในสายตาตลอดตอนที่เจ้าหน้าที่มองกล้อง ไม่ใช่แค่ตอนมองที่ dropdown ด้านบนเท่านั้น
  - แก้ bug เล็กในตัว: fetch events `useEffect` ที่รันครั้งเดียวตอน mount ลบการ `setLoadingEvents(true)` / `setEventsError(null)` ที่ไม่จำเป็นออก (ค่าตั้งต้นเป็นแบบนั้นอยู่แล้ว)

### DB / Migration

- **ไม่มี DB migration ใหม่** — ใช้ `startTime`/`endTime` ที่มีอยู่แล้วในข้อมูลกิจกรรม (payload จาก `/api/admin/events` เดิม); ไม่มีการเพิ่มหรือเปลี่ยนคอลัมน์; ไม่ต้องรัน `db:migrate:container`

### PDPA / สิทธิ์เข้าถึง

- **ไม่มีการเปลี่ยน access control, PDPA gating หรือ audit log** — feature เป็น UI/UX layer ล้วนบนหน้า scanner ที่มี auth gate เดิมอยู่แล้ว (scanner-only `smo` role เข้าได้เฉพาะ `/admin/scanner` ตาม `admin-access.ts`); ไม่มี API ใหม่ ไม่มี read path ใหม่สำหรับ medical detail

## ฝั่งทีม (technical changelog)

### Admin / Scanner — แก้ภาพกล้องกลับด้านตอน fallback เป็นกล้องหน้า

- **Context / root cause** — หน้า scanner ขอใช้กล้องหลัง (`facingMode: "environment"`) เสมอ ซึ่งเบราว์เซอร์ส่งภาพแบบไม่กลับด้านอยู่แล้วตามปกติ — อุปกรณ์จริงที่ใช้สแกนหน้างาน (มือถือ/แท็บเล็ตที่มีกล้องหลัง) จึงไม่เคยเจอปัญหานี้ แต่บนอุปกรณ์ที่ไม่มีกล้องหลัง (เช่น admin เปิดทดสอบบนโน้ตบุ๊ก) เบราว์เซอร์จะ fallback ไปใช้กล้องหน้าเงียบ ๆ โดยไม่แจ้งเตือน และภาพจากกล้องหน้าตามธรรมเนียมเบราว์เซอร์จะถูกส่งมาแบบกลับด้านอยู่แล้ว (เหมือนกระจกเซลฟี่) — ทำให้ admin ที่ทดสอบสแกน QR สับสนว่ากล้องแสดงผลผิด
- **`src/app/admin/scanner/page.tsx`** (+16 บรรทัด) เป็นไฟล์เดียวที่แก้ — หลังกล้องเริ่มทำงาน เรียก `scanner.getRunningTrackSettings()` เช็คว่า `facingMode` ที่ได้จริงเป็น `"user"` หรือไม่ (แปลว่าเจอ fallback เป็นกล้องหน้า) ถ้าใช่ จะ set CSS `transform: scaleX(-1)` บน `<video>` element ของกล้อง เพื่อพลิกภาพกลับมาเป็นทิศทางธรรมชาติ (ไม่กลับด้าน) — ครอบ logic ด้วย `try/catch` และ `console.error` เผื่อ browser ไม่รองรับ `getRunningTrackSettings()`
- **ไม่กระทบการใช้งานจริงหน้างาน** — อุปกรณ์ที่ใช้กล้องหลังจริงตามปกติจะไม่เข้าเงื่อนไขนี้เลย ภาพยังคงแสดงผลเหมือนเดิมทุกประการ; แก้เฉพาะกรณี fallback เป็นกล้องหน้าเท่านั้น

### DB / Migration

- **ไม่มี DB migration ใหม่** — pure client-side video element style fix; ไม่มีการเพิ่มหรือเปลี่ยนคอลัมน์; ไม่ต้องรัน `db:migrate:container`

### PDPA / สิทธิ์เข้าถึง

- **ไม่มีการเปลี่ยน access control, PDPA gating หรือ audit log** — fix แตะเฉพาะการแสดงผลภาพกล้องบนหน้า scanner ที่มี auth gate เดิมอยู่แล้ว; ไม่มี API ใหม่ ไม่มี read path ใหม่สำหรับ medical detail

## ฝั่งทีม (technical changelog)

### Admin / Scanner — แก้เงื่อนไขตรวจจับกล้อง fallback ให้ครอบคลุมขึ้น

- **Context / root cause** — หลังจากปล่อยรอบที่ 2 (PR #134) เจอว่ายังไม่แก้ปัญหาจริง: ทดสอบบนโน้ตบุ๊กแล้วเอียงหน้าไปทางขวา ภาพบนจอยังกลับกลายเป็นเอียงซ้ายเหมือนเดิม ต้นตอคือค่า `facingMode` ที่ได้จาก `scanner.getRunningTrackSettings()` บนกล้องเดสก์ท็อป/โน้ตบุ๊กมักไม่รายงานเป็น `"user"` ตรง ๆ ตามที่คาดไว้ — หลายเบราว์เซอร์คืนค่าอื่นหรือคืน `undefined` เลย (ขึ้นกับ browser/driver ของกล้องนั้น ๆ) ทำให้เงื่อนไขเดิมที่เช็ค exact-match `=== "user"` แทบไม่เคยเป็นจริง กล้อง fallback จึงไม่ถูกพลิกภาพ และยังโชว์ภาพแบบกลับด้านธรรมชาติ (เหมือนรูปที่คนอื่นถ่ายหน้าเรา ซ้าย-ขวาสลับกัน) อยู่เหมือนเดิม
- **`src/app/admin/scanner/page.tsx`** (+14, −9 บรรทัด) เป็นไฟล์เดียวที่แก้ — เปลี่ยนเงื่อนไขจาก "พลิกภาพเมื่อ `facingMode === 'user'`" เป็น "พลิกภาพทุกครั้งที่ยืนยันได้ว่า `facingMode` ไม่ใช่ `'environment'`" (`transform: facingMode === "environment" ? "none" : "scaleX(-1)"`) — กลับตรรกะจาก whitelist (ต้องแม่นว่าเป็นกล้องหน้า) มาเป็น whitelist ฝั่งกล้องหลังแทน (ต้องแม่นว่าเป็นกล้องหลังเท่านั้นถึงจะไม่พลิก) ทำให้ครอบคลุมทั้งกล้องหน้า, กล้องเดสก์ท็อปที่ไม่ report facingMode ชัดเจน ให้พลิกภาพสม่ำเสมอเหมือนกันหมด
- **ผลลัพธ์** — ทิศทางการเอียงตรงกันแล้วทั้งกรณีใช้กล้องหลังจริง (deploy จริงหน้างาน) และกรณี fallback เป็นกล้องหน้า/กล้องเดสก์ท็อป (ทดสอบบนโน้ตบุ๊ก); อุปกรณ์ที่ report `facingMode: "environment"` ถูกต้องยังคงไม่ถูกพลิกภาพเหมือนเดิม ไม่กระทบการใช้งานจริงหน้างาน

### DB / Migration

- **ไม่มี DB migration ใหม่** — pure client-side video element style logic เหมือนรอบที่ 2; ไม่มีการเพิ่มหรือเปลี่ยนคอลัมน์; ไม่ต้องรัน `db:migrate:container`

### PDPA / สิทธิ์เข้าถึง

- **ไม่มีการเปลี่ยน access control, PDPA gating หรือ audit log** — fix แตะเฉพาะการแสดงผลภาพกล้องบนหน้า scanner ที่มี auth gate เดิมอยู่แล้ว; ไม่มี API ใหม่ ไม่มี read path ใหม่สำหรับ medical detail
