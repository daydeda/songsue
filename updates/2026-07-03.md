## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **ไม่กระทบหน้าของนักศึกษาโดยตรง** — การเปลี่ยนแปลงนี้อยู่ในหน้า Admin → Events ที่ประธานชมรม/ประธานสาขาใช้จัดการกิจกรรมของตัวเอง; หน้า Dashboard ฝั่งนักศึกษาไม่เปลี่ยนแปลงใด ๆ
- **สำหรับประธานชมรม/ประธานสาขา** — เดิมกดปุ่ม "Edit" บนกิจกรรมของตัวเองไม่ได้เลย (ปุ่มถูกซ่อนไว้ ให้ดูอย่างเดียว/สแกน/export ได้เท่านั้น); ตอนนี้แก้ไขรายละเอียดกิจกรรมของตัวเองได้แล้ว — ชื่อกิจกรรม คำอธิบาย วัน-เวลา สถานที่ และจำนวนที่รับ โดยไม่ต้องรอให้ staff แก้ให้
- **ยังแก้ไม่ได้เหมือนเดิม (ช่องเป็นสีจาง + มีคำเตือนกำกับ)** — สิทธิ์เข้าร่วมตามบทบาท, สิทธิ์เข้าร่วมตามสาขา, ผู้ดูแลกิจกรรม (ประธาน), คะแนนบ้านที่ได้รับ, และคะแนนรายบุคคลที่ได้รับ — ส่วนเหล่านี้ยังต้องขอให้ admin/ทะเบียน/ผู้จัดกิจกรรมแก้ให้เหมือนเดิม
- **แก้ไขได้เฉพาะกิจกรรมของตัวเองเท่านั้น** — เหมือนกับที่เห็นแค่กิจกรรมตัวเองในรายการอยู่แล้ว ไม่สามารถแก้กิจกรรมของฝ่ายอื่นได้
- **การสร้างกิจกรรมใหม่และการลบกิจกรรมยังคงเป็นสิทธิ์ staff เท่านั้น ไม่เปลี่ยน** — ประธานชมรม/ประธานสาขาสร้างหรือลบกิจกรรมไม่ได้ แก้ได้แค่รายละเอียดของกิจกรรมที่มีอยู่แล้วและตัวเองดูแล
- **แก้บั๊ก: ลงทะเบียนกิจกรรมจากในหน้ารายละเอียด (event preview) แล้วป๊อปอัพ "ต้องทำ pre-test ก่อน" หายไป** — เดิมถ้ากดลงทะเบียนจากตรงป๊อปอัพรายละเอียดกิจกรรมที่เปิดค้างอยู่ (ไม่ใช่กดจากการ์ด/รายการ) ป๊อปอัพแจ้งเตือนหลังลงทะเบียน (ต้องทำ pre-test / ลงทะเบียนสำเร็จ / มีข้อผิดพลาด / ยืนยันยกเลิกลงทะเบียน) จะขึ้นจริงแต่ถูกบังซ่อนอยู่หลังป๊อปอัพรายละเอียดที่เปิดค้าง ต้องปิดป๊อปอัพรายละเอียดก่อนถึงจะเห็น ทำให้พลาดข้อความ "ต้องทำ pre-test ก่อน" ได้ง่าย — ตอนนี้แก้แล้ว ป๊อปอัพแจ้งเตือนจะขึ้นทับด้านบนทันทีไม่ว่าจะลงทะเบียนจากตรงไหน

## ฝั่งทีม (technical changelog)

### Admin / Events — เปิดสิทธิ์แก้ไขรายละเอียดกิจกรรมแบบ scoped ให้ club_president/major_president

- **`src/app/admin/events/page.tsx`** — เพิ่ม `isPresidentRole` (role ใน `["club_president", "major_president"]`) และ `canEditEventDetails = !isAttendanceOnly || isPresidentRole` ควบคุมการแสดงปุ่ม "Edit" — เดิมปุ่มนี้ซ่อนทั้งหมดสำหรับ role กลุ่ม `isAttendanceOnly` (scanner-only: smo/club_president/major_president) ตอนนี้ president เห็นปุ่ม Edit แล้วแต่ปุ่ม Delete ยังคงกันด้วย `!isAttendanceOnly` เดิม — president ไม่มีทางลบกิจกรรมได้
- **`canEditRestrictedFields = !isAttendanceOnly`** ใช้ disable 5 ส่วนที่ sensitive กว่าในฟอร์ม: Role-Based Access Control, Major-Based Access Control, Managed By (President), House Points Awarded, Individual Points Awarded — ทำเป็น opacity 0.5 + `pointerEvents: none` (สำหรับ selector 3 ส่วน) และ `disabled` attribute (สำหรับ input ตัวเลข 2 ช่อง) พร้อมข้อความเตือนสีส้มตัวหนา (i18n key ใหม่ `eventStaffOnlyFieldHint`) แทนที่ hint ปกติเมื่อ user ไม่ใช่ staff
- **`PUT /api/admin/events/[id]/route.ts`** — เดิม gate รับเฉพาะ `super_admin`/`admin`/`registration`/`organizer`; เพิ่ม `isPresidentRole` ให้ผ่าน initial 401 check ได้ด้วย (เช็คจาก role set ผ่าน `effectiveRoles()` ไม่ใช่ primary role เดี่ยว — pattern เดียวกับที่ใช้ทั่วโปรเจกต์)
  - **Ownership verify สำหรับ non-staff** — fetch `event.ownerClubIds`/`ownerMajors` แล้วเช็คผ่าน `EventScopeService.getPresidentScope` + `isEventManagedByScope` (ฟังก์ชัน ownership-scoping ตัวเดียวกับที่ attendance/export/report ใช้อยู่แล้ว) — ไม่ตรงกับกิจกรรมที่ตัวเองดูแล → 401 ทันที
  - **Server strip 5 field ที่ restricted ก่อนเขียนลง DB จริง** — `pointsAwarded`, `individualPointsAwarded`, `allowedRoles`, `allowedMajors`, `managedByRoles`, `ownerClubIds`, `ownerMajors` ถูกตั้งเป็น `undefined` สำหรับทุก caller ที่ไม่ใช่ staff โดยไม่สนว่า client ส่งอะไรมาในตัว payload — นี่คือ security boundary ตัวจริง ไม่ใช่แค่ disable ที่ UI
- **`src/lib/i18n.ts`** — เพิ่ม key `eventStaffOnlyFieldHint` ครบทั้ง 4 ภาษา (EN/TH/MM/CN): "Editable by admin, registration, and organizer roles only." และคำแปล

### PDPA / สิทธิ์เข้าถึง

- **เปิดสิทธิ์เขียนใหม่ (edit event details) ให้ role ที่ต่ำกว่า staff** — คู่ขนานกับที่เปิดสิทธิ์จัดการสมาชิกชมรมให้ `club_president` ไปก่อนหน้านี้ (PR #140); scope แน่นด้วย ownership check เดียวกับที่ attendance/export/report ใช้อยู่แล้ว ไม่ใช่ check ใหม่ที่ยังไม่ผ่านการ review
- **Field ที่ sensitive กว่า (คะแนน, สิทธิ์เข้าร่วมตามบทบาท/สาขา, ownership/managed-by) ยังคง staff-only ทั้งสองชั้น** — ทั้ง UI (disabled/grayed out) และ server (strip ออกจาก payload ก่อนเขียน DB) เป็น defense-in-depth ไม่ใช่แค่ซ่อนที่ client ฝั่งเดียว
- **ไม่มีการเปลี่ยนแปลงใด ๆ เกี่ยวกับข้อมูลทางการแพทย์/PDPA-sensitive field** — events ไม่มี medical data อยู่แล้ว การเปลี่ยนแปลงนี้กระทบแค่ field รายละเอียดกิจกรรมและ scope สิทธิ์การแก้ไข

### DB / Migration

- **ไม่มี DB migration ใหม่** — ใช้ column และตารางที่มีอยู่แล้วทั้งหมด (`events.ownerClubIds`, `ownerMajors`, `managedByRoles` ฯลฯ); ไม่มีการเพิ่มหรือเปลี่ยนคอลัมน์; ไม่ต้องรัน `db:migrate:container`

### Dashboard / Mobile-UI — แก้ z-index ป๊อปอัพหลังลงทะเบียนให้ลอยเหนือ event preview

- **`src/app/dashboard/DashboardClient.tsx`** — ป๊อปอัพ event detail/preview modal (เปิดผ่าน `?event=<id>`) เรนเดอร์ที่ `zIndex: 1999` ส่วนป๊อปอัพอีก 4 ตัว ได้แก่ pre-test-required, generic registration success, error, และ unregister-confirmation เดิมเรนเดอร์ที่ `zIndex: 1350` ซึ่งต่ำกว่า — เวลาลงทะเบียนจากในป๊อปอัพ preview ที่เปิดค้างอยู่ ป๊อปอัพแจ้งเตือนเหล่านี้จะซ้อนอยู่หลัง preview (มองไม่เห็น) ทั้งที่เรนเดอร์อยู่จริง
- **Fix:** เปลี่ยน z-index ของทั้ง 4 modal จาก `1350` → `2050` (สูงกว่า preview modal ที่ `1999`) ทำให้ป๊อปอัพเหล่านี้ขึ้นทับด้านบนเสมอ ไม่ว่าจะลงทะเบียนจากการ์ด/รายการ หรือจากในป๊อปอัพ event preview — เป็นการแก้ stacking-order ฝั่ง client ล้วน ๆ ไม่มีการเปลี่ยน schema/API
