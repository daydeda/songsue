# ActiveCAMT — อัปเดต 6 ก.ค. 69

## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **ไม่กระทบหน้าของนักศึกษาโดยตรง** — การเปลี่ยนแปลงนี้อยู่ในหน้า Admin → Events (แผงรายชื่อผู้เข้าร่วม/attendance roster) ที่ staff ใช้ดูรายชื่อผู้เข้าร่วมกิจกรรมเท่านั้น
- **สำหรับ staff (registration/organizer/admin)** — เพิ่มปุ่มกรอง "Filter: Master & Ph.D Degree" ในแถบตัวกรองของรายชื่อผู้เข้าร่วม อยู่ข้าง ๆ ตัวกรองเดิม (ชั้นปี, นักศึกษาไทย/ต่างชาติ, เฉพาะนักศึกษา, มีเงื่อนไขทางการแพทย์, ยังไม่เช็คอิน) — กดแล้วจะกรองให้เห็นเฉพาะนักศึกษาปริญญาโท/เอกในกิจกรรมนั้น ๆ ทันที
- **อัปเดตในวันเดียวกัน** — แยกปุ่มกรองข้างต้นออกเป็น 2 ปุ่มอิสระ "Filter: Master's Degree" กับ "Filter: Ph.D Degree" แทนปุ่มรวมเดิม staff เลือกดูเฉพาะปริญญาโทอย่างเดียว, เฉพาะปริญญาเอกอย่างเดียว, หรือเปิดทั้งสองปุ่มพร้อมกันเพื่อดูทั้งคู่ก็ได้ — ยืดหยุ่นกว่าเดิมที่กรองได้แค่แบบรวมสองกลุ่มเป็นก้อนเดียว

- ตอนนี้ใครก็ตามที่เข้าเว็บที่หน้าแรกจะถูกพาไป `/dashboard` ทันทีเหมือนเดิมทุกประการ (ก่อน 5 ก.ค.) — เนื่องจาก `/dashboard` เปิดให้ผู้ที่ยังไม่ล็อกอินดูตัวอย่างได้อยู่แล้วตั้งแต่ 9 มิ.ย. (พร้อมปุ่มสมัคร/ลงทะเบียนของตัวเอง) คนที่ยังไม่ล็อกอินจึงเห็นประสบการณ์เดิมเป๊ะ ๆ กับก่อนวันที่ 5 ก.ค. — เป็นการ revert กลับ ไม่ใช่ฟีเจอร์ใหม่ ไม่มีอะไรหายไปจากมุมมองนักศึกษา

### แก้บัคสำคัญ: บ้านของนักศึกษาสุ่มเปลี่ยนทุกครั้งที่ทีมอัปเดตเว็บ (PR #154)

- **แก้แล้ว: "บ้าน" ของนักศึกษาจะไม่เปลี่ยนเองอีกต่อไปหลังทีม deploy เว็บใหม่** — ก่อนหน้านี้มีบัคที่ทำให้บ้านของนักศึกษา (เช่น "หมอ") เปลี่ยนไปเป็นบ้านอื่น (เช่น "หลวง") แบบสุ่ม โดยจะไปสังเกตเห็นตอนเช็คอินกิจกรรมครั้งต่อไปด้วย QR แล้วเจอว่าอยู่บ้านอื่นไปแล้ว — สาเหตุมาจากสคริปต์ฐานข้อมูลเบื้องหลังที่รันซ้ำทุกครั้งที่ทีม deploy เว็บ ตอนนี้แก้จุดนั้นแล้ว บ้านของนักศึกษาจะเซ็ตครั้งเดียวและไม่ถูกรีเซ็ตอีกจากการ deploy ในอนาคต
- **หมายเหตุ:** การแก้นี้ป้องกันแค่ "ไม่ให้บัคนี้เกิดซ้ำในอนาคต" เท่านั้น ส่วนบ้านของนักศึกษาที่เคยถูกสลับผิดไปแล้วจากบัคนี้ก่อนหน้า ทีมงานต้องรันสคริปต์กู้คืนแยกอีกครั้ง (คนละงานจากการแก้บัคนี้ ยังไม่ได้ทำ)

### DB / Migration — root cause: unconditional `UPDATE users SET house_id = NULL` รันซ้ำทุก deploy (PR #154)

- **สาเหตุ:** `src/db/migrate.ts` **ไม่มี migrations-tracking table** — ทุกครั้งที่รัน `db:migrate` / `db:migrate:container` สคริปต์จะ replay ทุก step จากบรรทัดแรกใหม่หมดทุกครั้ง (ไม่ใช่รันแค่ step ที่ยังไม่เคยรัน)
- Step 65 เดิม (leftover จากฟีเจอร์ four-faculty-houses ที่ revert ไปแล้วใน PR #152 / commit `6168501`) มีบรรทัด `UPDATE users SET house_id = NULL` แบบไม่มีเงื่อนไขใด ๆ กำกับ — ตั้งใจให้รันครั้งเดียวตอน cutover เปลี่ยนโมเดลบ้าน แต่เพราะสคริปต์ replay ทุก step ทุกครั้ง มันจึงรันซ้ำไปเรื่อย ๆ ทุก deploy แม้หลังจาก feature ทั้งฟีเจอร์ถูก revert ไปแล้วก็ตาม (ไม่มีใครไปลบบรรทัดนี้ทิ้งตอน revert)
- ผลคือ `house_id` ของผู้ใช้ทุกคนถูกรีเซ็ตเป็น `NULL` ทุกครั้งที่ deploy แล้วนักศึกษาจะถูก re-derive เข้าบ้านใหม่แบบสุ่ม/balancing ตอนเช็คอินกิจกรรมครั้งต่อไปผ่าน `ScannerService.ensureHouseAssigned` — คือที่มาของบัค "บ้านเปลี่ยนเองหลัง deploy"
- **แก้ใน `src/db/migrate.ts`:** ลบบรรทัด `UPDATE users SET house_id = NULL` ออกจาก step 65 ทิ้ง เหลือไว้แค่ `UPDATE users SET faculty = 'CAMT' WHERE faculty IS NULL` (backfill คอลัมน์ faculty เดิม ซึ่ง idempotent และไม่มีผลข้างเคียง) พร้อมคอมเมนต์เตือนไว้ในโค้ดว่าห้ามใส่ unconditional house_id reset กลับมาอีก เพราะจะเกิดบัคเดิมซ้ำทันทีในการ deploy ครั้งต่อไป
- **ยังไม่ปิดจบ 100%:** การแก้นี้หยุดแค่ "ไม่ให้บัคเกิดซ้ำในอนาคต" — ข้อมูล `house_id` ที่ถูกรีเซ็ตไปแล้วจากบัคนี้ในอดีตยังคงต้องกู้คืนด้วยสคริปต์ที่มีอยู่แล้ว (`scripts/restore-houses-from-supabase.mjs` จาก snapshot ก่อนรีเซ็ต, หรือ `scripts/backfill-user-houses.mjs` สำหรับคนที่ไม่มีบ้านเลย) โดยต้องรันแยกอีกครั้งจาก Portainer console — เป็น manual follow-up ที่ยังไม่ได้ทำ ไม่รวมอยู่ใน PR #154 นี้
- ไม่มีการเปลี่ยน schema/สร้าง column ใหม่ — เป็นการแก้ logic ในสคริปต์ migration เท่านั้น

#### PDPA / สิทธิ์เข้าถึง (เฉพาะ PR #154)

- **ไม่มีการเปลี่ยนแปลง** — `house_id` ไม่ใช่ข้อมูลทางการแพทย์หรือข้อมูลอ่อนไหวตาม PDPA ไม่เข้าเงื่อนไข audit log; การแก้เป็นการลบ side-effect ที่ผิดพลาดออกจากสคริปต์ migration เท่านั้น ไม่กระทบสิทธิ์เข้าถึงข้อมูลใด ๆ
