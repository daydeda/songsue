## ฝั่งทีม (technical changelog)

### DB / Infra — เพิ่มระบบสำรองฐานข้อมูล Postgres รายวันขึ้น Google Drive อัตโนมัติ (PR #155)

- **ปัญหาเดิม:** ตั้งแต่ย้ายมา self-host Postgres บนเซิร์ฟเวอร์มหาวิทยาลัย (24 มิ.ย. 69) ฐานข้อมูล production (`activecamt-db`) ยังไม่มีระบบ backup ใด ๆ เลย — ถ้าเครื่อง/volume มีปัญหาข้อมูลทั้งหมดหายถาวร
- **เพิ่ม service ใหม่ `backup` ใน `docker-stack.yml`** — รันทุกวันแบบอัตโนมัติ: `pg_dump` ฐานข้อมูล → gzip → อัปโหลดขึ้นโฟลเดอร์ Google Drive ที่กำหนด → ลบไฟล์ dump ในเครื่องทิ้งทันทีหลังอัปโหลดสำเร็จ → ล้าง backup เก่าบน Drive ที่เกิน `BACKUP_RETENTION_DAYS` (default 30 วัน) ทิ้งอัตโนมัติ
- **`scripts/backup-db.mjs`** — สคริปต์ job หลักที่รันทุกวันตามด้านบน; ตัว dump ไม่แตะ persistent volume ของเซิร์ฟเวอร์เลย (ไม่ใช้พื้นที่ดิสก์เพิ่มถาวร) และไม่มีค่าใช้จ่ายเพิ่ม ($0 — ใช้โควตา Google Drive ปกติของบัญชี)
- **เลือกใช้ OAuth บัญชี Google ส่วนตัวของ operator แทน service account** — เหตุผล: service account ไม่มีโควตาพื้นที่ Drive เป็นของตัวเอง และไม่สามารถเป็นเจ้าของไฟล์นอก Shared Drive ได้ (Shared Drive ไม่มีในทุกแผน Google/Workspace) — จึงต้องใช้ Desktop app OAuth client + refresh token แทน
- **`scripts/gdrive-get-refresh-token.mjs`** — สคริปต์ one-time สำหรับรันในเครื่อง local เพื่อขอ refresh token ผ่าน browser consent flow แบบ loopback (ใช้ตั้งค่าครั้งแรกเท่านั้น ไม่ใช่ส่วนของ job รายวัน)
- **`Dockerfile`** — เพิ่มติดตั้ง `postgresql16-client` ใน image เพื่อให้มี `pg_dump` ใช้งานได้ในคอนเทนเนอร์
- **`docs/supabase-to-university-server-cutover.md`** — เพิ่ม section "Backups" อธิบายขั้นตอน one-time setup ทั้งหมด (สร้าง OAuth client, ขอ refresh token, ตั้งค่า env vars ที่เกี่ยวข้อง)
- **`.env.production.example`** — เพิ่มตัวแปร env ที่เกี่ยวข้องกับ backup (Google OAuth credentials, Drive folder ID, retention days ฯลฯ) พร้อมคำอธิบาย
- **ตรวจสอบแล้วก่อนขึ้นจริง** — ทดสอบ end-to-end กับ Google Drive folder จริงในเครื่อง local ครบ 3 ขั้น (auth ผ่าน, มองเห็นโฟลเดอร์ปลายทางถูกต้อง, อัปโหลด/ลบไฟล์ทำงานถูกต้อง) ก่อน merge

### PDPA / สิทธิ์เข้าถึง

- **สำคัญ — backup มีข้อมูลทั้งฐานข้อมูล รวมถึงข้อมูลอ่อนไหวตาม PDPA (ข้อมูลทางการแพทย์, emergency contact ฯลฯ) อยู่ในไฟล์ dump เดียวกันทั้งหมด** เพราะเป็น `pg_dump` ระดับฐานข้อมูล ไม่ได้กรองแยกตาราง — ตัวไฟล์ backup บน Google Drive จึงเป็นจุดที่ต้องระวังเรื่องสิทธิ์เข้าถึงเทียบเท่ากับตัวฐานข้อมูลจริง (ใครมีสิทธิ์เข้าถึงโฟลเดอร์ Drive ปลายทาง = เข้าถึงข้อมูลอ่อนไหวทั้งหมดได้)
- ปัจจุบันอัปโหลดผ่านบัญชี Google ส่วนตัวของ operator (ไม่ใช่ service account ที่แยก ownership ได้ชัดเจนกว่า) — ทีมควรจำกัดว่าใครมีสิทธิ์เข้าถึงโฟลเดอร์ Drive ปลายทางนี้ให้แคบที่สุด และพิจารณาทบทวนแนวทาง ownership/access ของโฟลเดอร์นี้ต่อไปเป็น follow-up
- ไม่มีการเพิ่ม audit log สำหรับ job นี้ (เป็น automated job เบื้องหลัง ไม่ใช่การเข้าถึงข้อมูลผ่าน UI/API โดย user คนใดคนหนึ่ง) — ไม่กระทบ audit log ของการเข้าถึงข้อมูลทางการแพทย์ที่มีอยู่เดิม

### DB / Migration

- **ไม่มี DB migration ใหม่** — ไม่มีการเปลี่ยน schema ใด ๆ ไม่ต้องรัน `db:migrate:container`; เป็นการเพิ่ม service/สคริปต์ backup เท่านั้น
- **ต้องทำ one-time setup บนเซิร์ฟเวอร์จริงก่อนใช้งาน** — สร้าง OAuth client + ขอ refresh token (ผ่าน `scripts/gdrive-get-refresh-token.mjs`) แล้วตั้งค่า env vars ที่เกี่ยวข้องใน production ตามที่ระบุไว้ใน `docs/supabase-to-university-server-cutover.md` ก่อน service `backup` จะทำงานได้จริงบน production
