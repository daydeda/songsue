## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **ระบบสำรองฐานข้อมูล (ด้านล่าง) ไม่กระทบหน้าตาหรือการใช้งานของนักศึกษาเลย** — เป็นงานเบื้องหลังฝั่ง infrastructure ล้วน ๆ ไม่มีหน้าจอ ปุ่ม หรือ flow ใดที่นักศึกษาเห็นเปลี่ยนไป
- **ฟีเจอร์ใหม่: ระบบ "ตัดสิทธิ์" นักศึกษาที่ลงทะเบียนแต่ไม่มาเช็คอิน (No-show Strike-out)** — ถ้าลงทะเบียนกิจกรรมล่วงหน้าไว้แล้วไม่ไปเช็คอินเลยสักครั้ง (staff ต้องกดยืนยันหลังกิจกรรมจบก่อน ไม่ใช่ระบบตัดอัตโนมัติทันทีที่กิจกรรมปิด) จะถูกนับเป็น "strike" 1 ครั้ง สะสมครบ 3 ครั้งจะถูกระงับสิทธิ์ **ลงทะเบียนกิจกรรมใหม่** ชั่วคราว จนกว่า staff จะปลดล็อกให้ (ไม่กระทบกิจกรรมที่ลงทะเบียน/เช็คอินไปแล้วก่อนหน้า)
- **เห็นยอดสะสมได้ทันทีจากหน้า Dashboard** — ตั้งแต่ strike แรกจะมีป้ายเตือน "ไม่มาเช็คอิน: X/3" ขึ้นให้เห็น (สีเหลืองเตือนล่วงหน้า แล้วเปลี่ยนเป็นสีแดงตั้งแต่เหลืออีก 1 ครั้งจะโดนบล็อก) พร้อมคำอธิบายสั้น ๆ ว่าจะเกิดอะไรขึ้นถ้าสะสมครบ 3 ครั้ง
- **โดนหักคะแนนบุคคลทุกครั้งที่โดน strike** — ปกติครั้งละ 10 คะแนน (staff ผู้กดยืนยันปรับได้ในช่วง 1–50 คะแนนต่อครั้ง แล้วแต่ความร้ายแรง)
- **ถูกบล็อกแล้ว ไม่ต้องรอ staff ติดต่อกลับเอง** — มีปุ่ม "ยื่นอุทธรณ์" บนป้ายเตือนเดียวกัน พิมพ์คำชี้แจง (อย่างน้อย 10 ตัวอักษร) ส่งให้ staff พิจารณาได้เลยจากหน้า Dashboard — ยื่นได้ครั้งละ 1 คำร้องเท่านั้น (ต้องรอผลคำร้องเดิมก่อนถึงจะยื่นใหม่ได้) และจะเห็นสถานะ "รอตรวจสอบ" หรือเหตุผลที่ staff ปฏิเสธ (ถ้ามี) อัปเดตอยู่ตรงป้ายเดิมโดยไม่ต้องรีเฟรชเอง (poll ทุก 60 วินาที)

## ฝั่งทีม (technical changelog)

### DB / Infra — เพิ่มระบบสำรองฐานข้อมูล Postgres รายวันขึ้น Google Drive อัตโนมัติ (PR #155)

- **ปัญหาเดิม:** ตั้งแต่ย้ายมา self-host Postgres บนเซิร์ฟเวอร์มหาวิทยาลัย (24 มิ.ย. 69) ฐานข้อมูล production (`activecamt-db`) ยังไม่มีระบบ backup ใด ๆ เลย — ถ้าเครื่อง/volume มีปัญหาข้อมูลทั้งหมดหายถาวร
- **เพิ่ม service ใหม่ `backup` ใน `docker-stack.yml`** — รันทุกวันแบบอัตโนมัติ: `pg_dump` ฐานข้อมูล → gzip → อัปโหลดขึ้นโฟลเดอร์ Google Drive ที่กำหนด → ลบไฟล์ dump ในเครื่องทิ้งทันทีหลังอัปโหลดสำเร็จ → ล้าง backup เก่าบน Drive ที่เกิน `BACKUP_RETENTION_DAYS` (default 30 วัน) ทิ้งอัตโนมัติ
- **`scripts/backup-db.mjs`** — สคริปต์ job หลักที่รันทุกวันตามด้านบน; ตัว dump ไม่แตะ persistent volume ของเซิร์ฟเวอร์เลย (ไม่ใช้พื้นที่ดิสก์เพิ่มถาวร) และไม่มีค่าใช้จ่ายเพิ่ม ($0 — ใช้โควตา Google Drive ปกติของบัญชี)
- **เลือกใช้ OAuth บัญชี Google ส่วนตัวของ operator แทน service account** — เหตุผล: service account ไม่มีโควตาพื้นที่ Drive เป็นของตัวเอง และไม่สามารถเป็นเจ้าของไฟล์นอก Shared Drive ได้ (Shared Drive ไม่มีในทุกแผน Google/Workspace) — จึงต้องใช้ Desktop app OAuth client + refresh token แทน
- **`scripts/gdrive-get-refresh-token.mjs`** — สคริปต์ one-time สำหรับรันในเครื่อง local เพื่อขอ refresh token ผ่าน browser consent flow แบบ loopback (ใช้ตั้งค่าครั้งแรกเท่านั้น ไม่ใช่ส่วนของ job รายวัน)
- **`Dockerfile`** — เพิ่มติดตั้ง `postgresql16-client` ใน image เพื่อให้มี `pg_dump` ใช้งานได้ในคอนเทนเนอร์
- **`docs/supabase-to-university-server-cutover.md`** — เพิ่ม section "Backups" อธิบายขั้นตอน one-time setup ทั้งหมด (สร้าง OAuth client, ขอ refresh token, ตั้งค่า env vars ที่เกี่ยวข้อง)
- **`.env.production.example`** — เพิ่มตัวแปร env ที่เกี่ยวข้องกับ backup (Google OAuth credentials, Drive folder ID, retention days ฯลฯ) พร้อมคำอธิบาย
- **ตรวจสอบแล้วก่อนขึ้นจริง** — ทดสอบ end-to-end กับ Google Drive folder จริงในเครื่อง local ครบ 3 ขั้น (auth ผ่าน, มองเห็นโฟลเดอร์ปลายทางถูกต้อง, อัปโหลด/ลบไฟล์ทำงานถูกต้อง) ก่อน merge

### PDPA / สิทธิ์เข้าถึง

- **สำคัญ — backup มีข้อมูลทั้งฐานข้อมูล รวมถึงข้อมูลอ่อนไหวตาม PDPA (ข้อมูลทางการแพทย์, emergency contact ฯลฯ) อยู่ในไฟล์ dump เดียวกันทั้งหมด** เพราะเป็น `pg_dump` ระดับฐานข้อมูล ไม่ได้กรองแยกตาราง — ตัวไฟล์ backup บน Google Drive จึงเป็นจุดที่ต้องระวังเรื่องสิทธิ์เข้าถึงเทียบเท่ากับตัวฐานข้อมูลจริง (ใครมีสิทธิ์เข้าถึงโฟลเดอร์ Drive ปลายทาง = เข้าถึงข้อมูลอ่อนไหวทั้งหมดได้)
- ปัจจุบันอัปโหลดผ่านบัญชี Google ส่วนตัวของ operator (ไม่ใช่ service account ที่แยก ownership ได้ชัดเจนกว่า) — ทีมควรจำกัดว่าใครมีสิทธิ์เข้าถึงโฟลเดอร์ Drive ปลายทางนี้ให้แคบที่สุด และพิจารณาทบทวนแนวทาง ownership/access ของโฟลเดอร์นี้ต่อไปเป็น follow-up
- ไม่มีการเพิ่ม audit log สำหรับ job นี้ (เป็น automated job เบื้องหลัง ไม่ใช่การเข้าถึงข้อมูลผ่าน UI/API โดย user คนใดคนหนึ่ง) — ไม่กระทบ audit log ของการเข้าถึงข้อมูลทางการแพทย์ที่มีอยู่เดิม

### DB / Migration

- **ไม่มี DB migration ใหม่** — ไม่มีการเปลี่ยน schema ใด ๆ ไม่ต้องรัน `db:migrate:container`; เป็นการเพิ่ม service/สคริปต์ backup เท่านั้น
- **ต้องทำ one-time setup บนเซิร์ฟเวอร์จริงก่อนใช้งาน** — สร้าง OAuth client + ขอ refresh token (ผ่าน `scripts/gdrive-get-refresh-token.mjs`) แล้วตั้งค่า env vars ที่เกี่ยวข้องใน production ตามที่ระบุไว้ใน `docs/supabase-to-university-server-cutover.md` ก่อน service `backup` จะทำงานได้จริงบน production

### No-show Strike-out + Appeals (PR #156, US-STRI-15a/b/c)

- **`src/lib/strikes.ts`** (ใหม่) — รวม constant ทั้งหมดไว้จุดเดียว: `NO_SHOW_STRIKE_THRESHOLD = 3`, `NO_SHOW_PENALTY_POINTS = 10` (default), `NO_SHOW_PENALTY_MIN/MAX = 1/50` (ขอบเขตที่ override ได้), `APPLY_STRIKES_ROLES = [super_admin, admin, organizer]`, `RESET_STRIKES_ROLES = [super_admin, admin]` (แคบกว่า apply เพราะ reset คือการล้างบทลงโทษ)
- **`POST /api/admin/events/[id]/apply-strikes`** (ใหม่, gate `APPLY_STRIKES_ROLES`) — staff กดยืนยันหลังกิจกรรมจบ (block ถ้ากิจกรรมยังไม่จบ) ระบบดึงรายชื่อที่ `status='registered'` แต่ไม่มีแถว `attended` ในกิจกรรมนั้น แล้ว flip เป็น `no_show` ทีละคนด้วย guarded UPDATE (กันโดน strike ซ้ำถ้ามีคนกดพร้อมกัน/สแกนช้าเข้ามาทีหลัง) เพิ่ม `users.noShowCount` และตั้ง `registrationBlocked = true` เมื่อถึง 3 — จำนวนคะแนนที่หักส่งมาทาง body (`points`, default 10) ถูก validate ว่าอยู่ในช่วง 1–50 ฝั่ง server ก่อนเสมอ; มี `GET` คู่กันไว้พรีวิวรายชื่อก่อนกดยืนยันจริง
- **`POST /api/admin/students/[id]/strikes/reset`** (ใหม่, gate `RESET_STRIKES_ROLES` เท่านั้น) — ล้าง `noShowCount`/`registrationBlocked` ให้นักศึกษารายคน **ไม่คืนคะแนนที่หักไปแล้ว** (ถือเป็นบทลงโทษที่ใช้ไปแล้ว แยกจากการปลดบล็อก)
- **`POST /api/events/[id]/register`** — เพิ่มเช็ค `profile.registrationBlocked` ก่อนให้ลงทะเบียน ถ้าบล็อกอยู่ตอบ 403 พร้อม `noShowCount` ปัจจุบันกลับไปให้ frontend ใช้ขึ้นข้อความ
- **Appeals — `src/app/api/appeals/route.ts`** (ใหม่) — `POST` ให้นักศึกษาที่ถูกบล็อกส่งคำร้อง (message 10–1000 ตัวอักษร), กันยื่นซ้ำด้วย pre-check + partial unique index `no_show_appeals_one_pending_per_user` (กันคำร้อง pending ซ้อนสองใบจริง ๆ ที่ระดับ DB ไม่ใช่แค่ pre-check ฝั่ง app); `noShowCountAtAppeal` snapshot ค่าตอนยื่นไว้ในแถว เพื่อให้ admin เห็น context ตอนพิจารณาแม้ค่าจริงจะเปลี่ยนไปแล้ว; `GET` คืนคำร้องล่าสุดของตัวเองให้ dashboard เช็คสถานะ
- **`src/app/api/admin/appeals/route.ts` + `.../[id]/route.ts`** (ใหม่, gate `RESET_STRIKES_ROLES`) — `GET` list คำร้องทั้งหมด (filter ตาม status ได้), `PATCH` approve/reject — **approve จะ reset strikes ให้ในทรานแซกชันเดียวกัน** (ไม่มีทางเกิดสภาพ "อนุมัติแล้วแต่ยังโดนบล็อกอยู่") และ re-assert `status='pending'` ในทรานแซกชันกันสองแอดมินกด resolve ซ้อนกันบนคำร้องเดียว
- **`src/app/admin/appeals/`** (ใหม่: `page.tsx` + `AppealsClient.tsx`) — แท็บใหม่ในหน้า Admin ให้ staff ดูคิวคำร้อง กรองตามสถานะ (pending/approved/rejected/all) และ approve/reject พร้อมใส่ note ได้; เพิ่มลิงก์ในเมนู **`AdminNav.tsx`** (gate เท่ากับ `RESET_STRIKES_ROLES` — เห็นแค่ admin/super_admin)
- **`src/app/admin/students/page.tsx`** — เพิ่ม badge "No-show X/3 (· Blocked)" ต่อแถวนักศึกษา และปุ่ม "Reset Strikes" (เฉพาะ role ที่แก้/ลบนักศึกษาได้อยู่แล้ว)
- **`src/app/admin/events/page.tsx`** — เพิ่มปุ่ม "Strike No-shows" ในหน้ารายชื่อผู้เข้าร่วม (โผล่เฉพาะ role ใน `APPLY_STRIKES_ROLES` และกิจกรรมต้องจบแล้ว) เปิด modal พรีวิวรายชื่อ no-show, ให้ปรับจำนวนคะแนนที่จะหักก่อนกดยืนยันจริง, สรุปผลหลังยืนยัน (จำนวนคนที่โดน / คะแนนรวมที่หัก / จำนวนคนที่เพิ่งถูกบล็อก)
- **`src/app/dashboard/DashboardClient.tsx`** (+192 บรรทัด) — banner แจ้ง strike + modal ยื่นอุทธรณ์ตามที่อธิบายไว้ฝั่งนักศึกษาด้านบน, poll strikes/appeal ทุก 60 วินาทีคู่กับ events/houses/announcement เดิม
- **`src/lib/i18n.ts`** (+136 บรรทัด) — คำแปลครบ 4 ภาษา (EN/TH/MM/CN) สำหรับ banner, modal อุทธรณ์, ข้อความบล็อกตอนลงทะเบียน, เมนู/หน้า Admin Appeals

### กลไกการหักคะแนน — หักเฉพาะคะแนนบุคคล ไม่แตะคะแนนบ้าน + ⚠️ เอกสารไม่ตรงกับโค้ดที่ shipped จริง

- **`src/lib/award-individual-points.ts`** — เพิ่ม `deductIndividualPoints()` หัก `users.points` (คะแนนบุคคล) ของนักศึกษาเท่านั้น **ไม่แตะคะแนนรวมของบ้าน** — มีการ insert แถวลง `scoreHistory` ไว้ด้วยแต่ตั้ง `delta: 0` (ไว้เป็น log ให้เห็นในประวัติเฉย ๆ ไม่ใช่การหักคะแนนบ้านจริง) เหตุผลตามคอมเมนต์ในโค้ด: การ strike ไม่ควรไปเรียกคืนโบนัสบ้าน 100 แต้มที่นักศึกษาได้จากการเข้าร่วมกิจกรรมจริงแยกต่างหาก
- **คะแนนบุคคลติดลบได้แล้ว ไม่ floor ที่ 0 อีกต่อไป** — `deductIndividualPoints()` หักตรง ๆ ไม่มี `GREATEST(0, ...)` กำกับ และ PR นี้ยังไปแก้ **`src/modules/events/scanner.service.ts`** (การหักคะแนนแบบ manual score จากหน้า scanner) ให้เอา `GREATEST(0, ...)` เดิมออกด้วยเช่นกัน — ตั้งใจให้ `users.points` มีพฤติกรรมเหมือน `houses.points` ที่ไม่เคย floor ที่ 0 อยู่แล้ว (ตามคอมเมนต์ในโค้ด)
- **⚠️ พบว่าเอกสารที่แก้ในพีอาร์เดียวกันนี้เขียนไว้ไม่ตรงกับโค้ดจริง** — `docs/scoring-rules.md` §6 ข้อ 5 และ `docs/agile/01-product-backlog.md` (US-STRI-15b) ต่างระบุว่าการหักคะแนน no-show "floored at 0 (`GREATEST(0, ...)`)" แต่โค้ดที่ merge จริงตัดการ floor นี้ออกทั้งคู่ (ทั้งจุดใหม่และจุดเดิมใน scanner.service.ts) — ควรตรวจสอบว่าตั้งใจให้ติดลบได้จริงหรือเป็นบั๊ก แล้วแก้เอกสารหรือโค้ดให้ตรงกันเป็น follow-up
- **`src/lib/leaderboard-cache.ts`** (ใหม่) — รวม logic bust cache ของ leaderboard (`house-standings`, `individual-standings`) เป็นฟังก์ชันเดียว `revalidateLeaderboards()` แทนที่โค้ด `revalidateTag` ที่เคยกระจายซ้ำ ๆ หลายจุด (houses/points, events/form, apply-strikes, scan, award-points event/form payout) — เรียกทุกจุดที่มีการเขียนคะแนนเพื่อให้ leaderboard อัปเดตทันทีแทนที่จะรอ cache TTL เดิม (สูงสุด 15 วิ); `individual-standings` cache เพิ่ม tag ให้ bust ได้ด้วย (เดิมไม่มี tag)

### PDPA / สิทธิ์เข้าถึง

- **ไม่มีการอ่าน/เพิ่มข้อมูลทางการแพทย์ใหม่ในฟีเจอร์นี้** — ข้อมูลที่เกี่ยวข้องมีแค่ `noShowCount`/`registrationBlocked`/ข้อความอุทธรณ์ ไม่ใช่ข้อมูลอ่อนไหวตาม PDPA
- **สิทธิ์แบ่งสองชั้นตามความรุนแรงของการกระทำ** — ยิง strike ได้ (`APPLY_STRIKES_ROLES`: super_admin/admin/organizer, จำกัดเฉพาะกิจกรรมที่จบแล้ว) เปิดกว้างกว่าการ reset/อนุมัติอุทธรณ์ (`RESET_STRIKES_ROLES`: super_admin/admin เท่านั้น) เพราะการ reset คือการล้างบทลงโทษทั้งหมด ควรเป็นการตัดสินใจของ staff ระดับสูงกว่า
- **ทุก apply-strikes / reset / approve-reject อุทธรณ์ เขียน audit log ครบ** — มี `targetId` เป็นนักศึกษาเป้าหมาย, จำนวนคะแนนที่หักจริง, และค่า noShowCount ก่อน/หลัง ผ่าน `AuditService.logActionInternal` ในทรานแซกชันเดียวกับการเขียนข้อมูลเสมอ

### DB / Migration — **ยังไม่ได้รันบน production**

- **Schema เพิ่ม 2 จุด, additive + idempotent + non-destructive ทั้งคู่** — `users.no_show_count` (integer, default 0) / `users.registration_blocked` (boolean, default false) ผ่าน `ADD COLUMN IF NOT EXISTS` (migrate.ts step 67, mirror ใน `drizzle/0023_users_no_show_strikeout.sql`) และตารางใหม่ `no_show_appeals` พร้อม index/partial-unique-index ผ่าน `CREATE TABLE/INDEX IF NOT EXISTS` (step 68, mirror ใน `drizzle/0024_no_show_appeals.sql`)
- **รีเฮิร์สกับ local DB ผ่านเรียบร้อยแล้ว แต่ยังไม่ได้รันกับ production จริง** — โค้ดที่เพิ่งขึ้น main อ่านคอลัมน์/ตารางใหม่นี้ (dashboard, admin students, admin events, appeals ทั้งชุด) ดังนั้น **ก่อนจะ recreate container `activecamt-app` รอบหน้า ต้องรัน `npm run db:migrate:container` จาก Portainer console ก่อนเสมอ** ตาม `/safe-deploy` — ไม่ทำตามลำดับนี้จะพัง (โค้ดใหม่จะพยายามอ่าน/เขียนคอลัมน์ที่ยังไม่มีอยู่จริงใน production)
