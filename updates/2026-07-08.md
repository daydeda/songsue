# ActiveCAMT — อัปเดต 8 ก.ค. 69

ช่วง: 8 ก.ค. 69 · สรุปไว้สำหรับลง Discord และให้ทีมที่เกี่ยวข้องอ่าน

## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **อุทธรณ์ no-show strike เปลี่ยนจาก "อุทธรณ์รวมทั้งบัญชี" เป็น "อุทธรณ์แยกรายกิจกรรม"** — ถ้ามีหลาย strike จากหลายกิจกรรม ตอนนี้ยื่นอุทธรณ์แยกทีละกิจกรรมได้ ไม่ต้องรวมทุกครั้งไว้ในคำร้องเดียวเหมือนเดิม
- **ยื่นอุทธรณ์ได้เร็วขึ้น — ไม่ต้องรอให้ถูกบล็อกครบ 3 ครั้งก่อน** เดิมต้องรอ registration ถูกระงับ (3/3) ถึงจะยื่นได้ ตอนนี้ยื่นอุทธรณ์ของกิจกรรมนั้น ๆ ได้ตั้งแต่โดน strike แรก
- **ป้ายเตือน "ไม่มาเช็คอิน" บน Dashboard แสดงรายชื่อกิจกรรมที่ทำให้โดน strike แยกเป็นรายการ** แต่ละกิจกรรมมีปุ่ม "ยื่นอุทธรณ์" และสถานะ "รอตรวจสอบ" / "ถูกปฏิเสธ" ของตัวเอง ไม่ปนกับกิจกรรมอื่น
- **staff อนุมัติอุทธรณ์แล้ว จะลบ strike เฉพาะกิจกรรมนั้นกิจกรรมเดียว** เช่น 3/3 → 2/3 (ไม่ใช่รีเซ็ตกลับเป็น 0/3 ทั้งบัญชีเหมือนเดิม) และปลดบล็อกให้อัตโนมัติทันทีที่ตัวเลขลดต่ำกว่าเกณฑ์
- **แก้ปัญหาชื่อกิจกรรมยาวล้นกรอบบนมือถือ** ทั้งที่ป้ายเตือน "ไม่มาเช็คอิน" หน้า Dashboard, หัวข้อ modal ยื่นอุทธรณ์ ("Appealing no-show for: …") และ modal ยืนยันอนุมัติ/ปฏิเสธฝั่งแอดมิน — ชื่อกิจกรรมยาวจะตัดบรรทัดพอดีกรอบแทนที่จะทะลุออกไป

## ฝั่งทีม (technical changelog)

### No-show Appeals — เปลี่ยนจากต่อบัญชี เป็นต่อกิจกรรม (PR #157, ต่อจาก PR #156)

- **`src/db/schema.ts`** — `noShowAppeals` เพิ่มคอลัมน์ `eventId` (uuid, nullable, FK → `events` ON DELETE CASCADE) — nullable เพื่อให้ ADD COLUMN ยังเป็น additive กับแถวเก่าที่มีอยู่ แต่ app layer บังคับให้ทุกคำร้องใหม่ต้องมี eventId เสมอ; unique index เดิม `no_show_appeals_one_pending_per_user` (1 pending ต่อ user) ถูกแทนที่ด้วย `no_show_appeals_one_pending_per_user_event` (scope เป็น user+event) — นักศึกษาที่มีหลาย strike จึงเปิดคำร้อง pending พร้อมกันได้หลายใบ (คนละกิจกรรม) แต่กิจกรรมเดียวกันซ้ำไม่ได้
- **`src/app/api/appeals/route.ts`** — `GET` เปลี่ยนจากคืน appeal ล่าสุดใบเดียว เป็นคืนรายการ `noShowEvents` (ทุกกิจกรรมที่มี attendance status `no_show` ของ user นั้น) แต่ละอันแนบ appeal ล่าสุดของกิจกรรมนั้น (ถ้ามี); กิจกรรมที่อุทธรณ์ผ่านแล้วจะหายจากลิสต์เองเพราะ attendance ถูกเปลี่ยนเป็น `excused`. `POST` เปลี่ยนเงื่อนไขจาก "ต้องถูก `registrationBlocked` อยู่ก่อน" เป็น "ต้องมี strike (`noShowCount > 0`) และมี attendance แถว `no_show` ของ **กิจกรรมนั้น** อยู่จริง" — บังคับส่ง `eventId` มาด้วยเสมอ (zod `uuid()`); กันยื่นซ้ำด้วย pre-check + unique index ใหม่ที่ scope เป็น (user, event)
- **`src/app/api/admin/appeals/[id]/route.ts`** — `PATCH` approve เปลี่ยนจาก "reset `noShowCount` เป็น 0 ทั้งบัญชี" เป็น "ลด `noShowCount` ลง 1" (floor ที่ 0) และปลดบล็อก (`registrationBlocked = false`) เฉพาะเมื่อค่าที่เหลือต่ำกว่าเกณฑ์ (`NO_SHOW_STRIKE_THRESHOLD`) — พร้อมอัปเดต attendance row(s) ของกิจกรรมนั้นจาก `no_show` เป็น `excused` ในทรานแซกชันเดียวกัน ไม่แตะ strike จากกิจกรรมอื่นเลย; audit log message ปรับให้ระบุกิจกรรมและค่า before/after ของ `noShowCount`; เพิ่มเงื่อนไข reject กรณี appeal เก่าที่ไม่มี `eventId` ผูกอยู่ (แถวก่อนไมเกรชันนี้) จะ approve อัตโนมัติไม่ได้ ต้อง handle แยก
- **`src/app/api/admin/appeals/route.ts`** — `GET` list เพิ่ม `with: { event: {...} }` ให้ staff เห็นชื่อ/เวลากิจกรรมของแต่ละคำร้องโดยตรง แทนที่จะต้อง match เดาจาก attendance
- **`src/app/admin/appeals/AppealsClient.tsx`** — การ์ดคำร้องแสดงชื่อกิจกรรมกำกับไว้ด้วย; modal ยืนยันอนุมัติเปลี่ยนข้อความจาก "resets to 0/{max}" เป็นระบุกิจกรรม + ค่า strike ก่อน/หลัง ({before}/{max} → {after}/{max}) ให้แอดมินเห็นผลจริงก่อนกดยืนยัน
- **`src/app/dashboard/DashboardClient.tsx`** (+99/-25 บรรทัด) — banner เปลี่ยนจาก object `appeal` เดี่ยว เป็น list `noShowEvents[]` แต่ละอันมีปุ่ม "ยื่นอุทธรณ์"/ป้ายสถานะของตัวเอง; ปุ่มอุทธรณ์แสดงตั้งแต่ strike แรก (ไม่ผูกกับ `registrationBlocked` แล้ว) ต่างจากเดิมที่โผล่เฉพาะตอนถูกบล็อกครบ 3 ครั้ง
- **`src/lib/i18n.ts`** — คำแปลครบ 4 ภาษาที่เกี่ยวข้อง ปรับข้อความอุทธรณ์/อนุมัติให้พูดถึง "กิจกรรม" แทน "การระงับสิทธิ์" ทั้งบัญชี, เพิ่มคีย์ label ชื่อกิจกรรม/สถานะ pending/rejected รายกิจกรรม

### PDPA / สิทธิ์เข้าถึง

- **ไม่มีข้อมูลอ่อนไหวใหม่ในฟีเจอร์นี้** — ยังคงเป็นแค่ `noShowCount` / `registrationBlocked` / ข้อความอุทธรณ์ / ความสัมพันธ์กับกิจกรรม ไม่ใช่ข้อมูลทางการแพทย์หรือ PDPA-sensitive
- **audit log ยังคงเขียนครบทุกครั้งที่ approve/reject** ผ่าน `AuditService.logActionInternal` ในทรานแซกชันเดียวกับการเขียนข้อมูล — ข้อความ log ปรับให้ระบุกิจกรรมที่เกี่ยวข้องและค่า `noShowCount` ก่อน/หลังด้วย
- **สิทธิ์ยิง strike กว้างขึ้น (ดูหัวข้อด้านบน)** — เพิ่ม `smo`/`club_president`/`major_president` เข้ามาใน `APPLY_STRIKES_ROLES` เป็นครั้งแรก โดย `club_president`/`major_president` ถูก scope ที่ server ให้ทำได้เฉพาะกิจกรรมตัวเอง (ไม่ใช่ทุกกิจกรรมเหมือน staff/`smo`) — ควรตรวจสอบบน production ว่า scope ทำงานถูกต้องจริง เพราะเป็นการให้สิทธิ์เชิงลงโทษ (หักคะแนน + block registration) แก่ role ที่ปกติเป็น scanner-only/scoped-only ครั้งแรก

### DB / Migration — **ยังไม่ได้รันบน production** (ต่อจาก step 67/68 เมื่อวาน)

- **Schema เพิ่ม 1 จุด, additive + idempotent + non-destructive** — `no_show_appeals.event_id` (uuid, nullable, FK → `events` ON DELETE CASCADE) ผ่าน `ADD COLUMN IF NOT EXISTS` (migrate.ts **step 69**, mirror ใน `drizzle/0025_no_show_appeals_per_event.sql`); สร้าง index `no_show_appeals_event_idx`, ลบ index เก่า `no_show_appeals_one_pending_per_user` ด้วย `DROP INDEX IF EXISTS`, แล้วสร้าง `no_show_appeals_one_pending_per_user_event` ใหม่แทน — ทุกขั้นตอน idempotent (`IF NOT EXISTS`/`IF EXISTS`) ปลอดภัยรันซ้ำได้
- **รีเฮิร์สกับ local DB ผ่านเรียบร้อยแล้ว แต่ยังไม่ได้รันกับ production** — โค้ดที่เพิ่งขึ้น main (dashboard, appeals API ทั้งชุด, admin appeals UI) อ่าน/เขียนคอลัมน์ `event_id` นี้โดยตรง รวมถึงยังพึ่ง step 67/68 จาก PR #156 เมื่อวานที่ก็ยังไม่ได้รันบน production เช่นกัน — **ก่อน recreate container `activecamt-app` รอบหน้า ต้องรัน `npm run db:migrate:container` จาก Portainer console ก่อนเสมอ** (จะครอบคลุมทั้ง step 67–69 ในรอบเดียว) ตาม `/safe-deploy` — ไม่ทำตามลำดับนี้จะพัง

### สิทธิ์เข้าถึงคิว Appeals — เปิดให้ smo ดูได้ (ไม่ใช่แค่ staff), club/major president อนุมัติได้เฉพาะกิจกรรมตัวเอง (PR #159, ต่อจาก PR #157/#156)

- **`src/lib/strikes.ts`** — แยกสิทธิ์ appeal ออกเป็น 2 role list ใหม่ แทนที่การใช้ `RESET_STRIKES_ROLES` ตัวเดียวร่วมกันทั้งดูและอนุมัติเหมือนเดิม: `VIEW_APPEALS_ROLES` = `[super_admin, admin, smo, club_president, major_president]` (ใครเข้าคิว `/admin/appeals` ได้บ้าง) และ `RESOLVE_APPEALS_ROLES` = `[super_admin, admin, club_president, major_president]` (ตัด `smo` ออก — ดูได้แต่กดอนุมัติ/ปฏิเสธไม่ได้)
- **`src/app/api/admin/appeals/route.ts`** (`GET`) — gate เปลี่ยนจาก `RESET_STRIKES_ROLES` เป็น `VIEW_APPEALS_ROLES`; ผลลัพธ์ของ `club_president`/`major_president` ถูก scope ที่ server ให้เห็นเฉพาะ appeal ของกิจกรรมที่ตัวเองเป็นเจ้าของ ผ่าน `EventScopeService.getPresidentScope` + `isEventManagedByScope` (ตัว scoping helper เดียวกับที่ apply-strikes และหน้า events list ใช้อยู่แล้ว) ส่วน `super_admin`/`admin`/`smo` เห็นทุก appeal แบบไม่ถูก scope; ฟิลด์ `ownerClubIds`/`ownerMajors` ของกิจกรรมถูกดึงมาแค่เพื่อกรอง scope แล้วตัดออกจาก response ก่อนส่งกลับ ไม่กระทบ shape ที่ `AppealsClient` ใช้อยู่
- **`src/app/api/admin/appeals/[id]/route.ts`** (`PATCH`) — gate เปลี่ยนจาก `RESET_STRIKES_ROLES` เป็น `RESOLVE_APPEALS_ROLES` (`smo` ดูคิวได้แต่ยิง `PATCH` ถูก reject ที่นี่); สำหรับ `club_president`/`major_president` (non-staff) เพิ่มการโหลด `ownerClubIds`/`ownerMajors` ของกิจกรรมที่ผูกกับ appeal แล้วเช็ค scope ด้วย `EventScopeService.getPresidentScope` + `isEventManagedByScope` — ถ้ากิจกรรมไม่อยู่ใน scope ของตัวเอง จะได้ 403 Forbidden ก่อนอนุมัติ/ปฏิเสธ
- **`src/app/admin/appeals/page.tsx`** — server component gate ที่ใช้ redirect เปลี่ยนจาก `RESET_STRIKES_ROLES` เป็น `VIEW_APPEALS_ROLES`; ปลายทาง redirect กรณีไม่มีสิทธิ์เปลี่ยนจาก hardcode `/admin/dashboard` เป็น `adminLandingHrefForRoles(myRoles)` (ไปหน้า landing ตาม role จริงของผู้ใช้ แทนที่จะไป dashboard เสมอ)
- **`src/app/admin/appeals/AppealsClient.tsx`** — อ่าน role ของผู้ใช้ที่ login อยู่ฝั่ง client (`useSession` + `effectiveRoles`) แล้วแสดงปุ่มอนุมัติ/ปฏิเสธบนการ์ด appeal ที่ยัง pending เฉพาะเมื่อ role ของผู้ใช้ตัดกับ `RESOLVE_APPEALS_ROLES` — `smo` เห็นการ์ดทุกใบแต่ไม่เห็นปุ่มกด (server ยัง enforce ซ้ำที่ `PATCH` อยู่ดี เช็คฝั่ง client เป็นแค่ UI ไม่ใช่ security boundary)
- **`src/lib/admin-access.ts`** — `SCANNER_ONLY_PAGES` (allowlist หน้าที่ scanner-only role อย่าง `smo` เข้าได้) เพิ่ม `/admin/appeals` เข้าไป (เดิม `smo` เข้าหน้านี้ไม่ได้เลย)
- **`src/components/admin/AdminNav.tsx`** — ลิงก์เมนู "Appeals" แสดงให้ scanner-only roles (`smo`, `club_president`, `major_president`) เห็นด้วย ไม่ใช่แค่ full-admin role เหมือนเดิม (ยังซ่อนสำหรับ `organizer`/`registration` เหมือนเดิม เพราะไม่เคยมี `VIEW_APPEALS_ROLES`)
- **`src/app/admin/students/page.tsx`** (แก้บั๊กเล็กที่ติดมาในพีอาร์เดียวกัน ไม่เกี่ยวกับ appeals โดยตรง) — `canEditOrDelete` (คุมปุ่ม edit/delete/reset-strikes บนหน้ารายชื่อนักศึกษา) เดิมเช็คแค่ `role` หลักตัวเดียว (`userRole === "super_admin" || userRole === "admin"`) เปลี่ยนเป็นเช็ค role ที่ effective ทั้งหมด (`effectiveRoles(session.user.role, session.user.roles).some(r => RESET_STRIKES_ROLES.includes(r))`) — ผู้ใช้ที่ได้สิทธิ์ admin/super_admin เป็น role รอง (ไม่ใช่ primary `role`) จะเห็นปุ่มพวกนี้ด้วย เป็นแค่การแก้บั๊ก ไม่มีข้อมูล PDPA ใหม่ถูกเปิด และ `RESET_STRIKES_ROLES` เองไม่เปลี่ยน (ยังเป็น super_admin/admin เท่านั้น) — เปลี่ยนแค่ว่าใคร "ถูกนับว่ามี" role นั้น

**ผลลัพธ์ที่ผู้ใช้จะเห็น**: `smo` (staff สแกน QR) ตอนนี้เห็นคิว appeal ไม่มาเช็คอินได้ เอาไว้เป็นข้อมูลประกอบตอนสแกนนักศึกษา แต่กดอนุมัติ/ปฏิเสธไม่ได้; `club_president`/`major_president` ซึ่งยิง strike ให้กิจกรรมของตัวเองได้อยู่แล้ว ตอนนี้ตรวจสอบและอนุมัติ/ปฏิเสธ appeal ของกิจกรรมเดียวกันได้ด้วย โดยไม่ต้องขอสิทธิ์ admin/organizer เต็ม — เป็นครั้งแรกที่ role ที่ไม่ใช่ admin/organizer ได้เข้าถึงคิว appeals เลย

### PDPA / สิทธิ์เข้าถึง (PR #159)

- **ไม่มีข้อมูล PDPA/การแพทย์ใหม่ถูกแตะต้อง** — appeal ยังคงเกี่ยวกับแค่ `noShowCount`/`registrationBlocked`/ข้อความอุทธรณ์/ความสัมพันธ์กับกิจกรรมเหมือนเดิม
- **audit log ยังเขียนครบทุกครั้งที่ approve/reject** ผ่าน `AuditService` ในทรานแซกชันเดียวกับการเขียนข้อมูล ไม่เปลี่ยนจากเดิม
- **นี่เป็นการขยายสิทธิ์ครั้งที่ 2 ในวันเดียวกันให้ role ที่ scoped/scanner-only** (ต่อจากการขยาย `APPLY_STRIKES_ROLES` ที่ทำไปก่อนหน้านี้ในวันเดียวกัน) — ควรตรวจสอบบน production ด้วย caveat เดียวกัน เพราะการอนุมัติ appeal คือ action เชิงลงโทษ(-ย้อนกลับ) (ลบ strike ที่ตั้งไปแล้ว) ที่เพิ่งให้สิทธิ์แก่ role ที่แต่ก่อนถูกจำกัดสิทธิ์มากกว่านี้เป็นครั้งแรก

### สิทธิ์ registration ในงาน appeals/strikes + filter no-show ใหม่บน roster (PR #161, ต่อจาก PR #159)

- **`src/lib/strikes.ts`** — เพิ่ม role `registration` เข้าไปใน 3 list พร้อมกัน จนตอนนี้สิทธิ์เท่า `admin` ในทุกจุดที่เกี่ยวกับ no-show:
  - `APPLY_STRIKES_ROLES` — เดิมมี `smo` อยู่ด้วย ตอนนี้ **เอา `smo` ออก แล้วใส่ `registration` แทน** (`smo` ดูได้อย่างเดียว ยิง strike ไม่ได้อีกต่อไป)
  - `VIEW_APPEALS_ROLES` — เพิ่ม `registration`
  - `RESOLVE_APPEALS_ROLES` — เพิ่ม `registration` (`smo` ไม่เคยอยู่ใน list นี้อยู่แล้ว ยังคงดูได้แต่กดอนุมัติ/ปฏิเสธไม่ได้เหมือนเดิม)

- **`src/app/api/admin/appeals/[id]/route.ts`** (`PATCH`) — `isStaff` เพิ่ม `registration` — อนุมัติ/ปฏิเสธ appeal ได้เหมือน `admin` ทุกกิจกรรมไม่ถูก scope
- **`src/app/api/admin/events/[id]/apply-strikes/route.ts`** — `isUnscopedStaff` เปลี่ยนจาก `[super_admin, admin, organizer, smo]` เป็น `[super_admin, admin, organizer, registration]` — `smo` ไม่ผ่านเงื่อนไขนี้อีกแล้ว แต่จริง ๆ ไม่มีทางมาถึง route นี้เลยเพราะถูกตัดออกจาก `APPLY_STRIKES_ROLES` ที่ gate ชั้นนอกไปแล้ว
- **`src/app/admin/events/page.tsx`** — `canApplyStrikes` (คุมปุ่ม "Strike No-shows" ที่ UI) sync ให้ตรงกับ `APPLY_STRIKES_ROLES` ใหม่: มี `registration`, ไม่มี `smo`
- **สรุปผลลัพธ์**: staff ฝ่าย registration (role `registration`) ตอนนี้เข้า `/admin/appeals` ดูและอนุมัติ/ปฏิเสธ appeal ได้ และกดปุ่ม "Strike No-shows" บน `/admin/events` ได้ — สิทธิ์เท่า admin ในสองจุดนี้ ไม่ถูก scope ตามกิจกรรมเหมือน club/major president ส่วน `smo` ถูกริบสิทธิ์ยิง strike ออกไป (จากที่เคยได้ตั้งแต่ PR #157) เหลือแค่ดูคิว appeals/roster ได้เฉยๆ

### Filter ใหม่บน attendance roster — "Has Strike History" / "No-Show This Event"

- **`src/app/api/admin/events/[id]/attendance/route.ts`** — เพิ่ม `noShowCount` เข้าไปทั้งใน `FULL_USER_COLUMNS` และ `THIN_USER_COLUMNS` (ที่ role แบบ `smo`/`club_president`/`major_president` ใช้) พร้อม comment ระบุชัดว่า strike tally ไม่ใช่ PDPA-sensitive จึงส่งให้ทุก role ที่เข้าถึง roster ได้ ต่างจาก emergency contact/medical detail ที่ยังจำกัดเฉพาะ `FULL_USER_COLUMNS` (admin/super_admin เท่านั้น)
- **`src/app/admin/events/page.tsx`** — เพิ่ม 2 ปุ่ม filter บน roster ของแต่ละกิจกรรม (`/admin/events` → เปิดกิจกรรม → attendance):
  - **"Has Strike History"** (`filterStrikeHistory`) — กรองเฉพาะนักศึกษาที่มี `noShowCount > 0` แบบทั้งบัญชี (สะสมจากทุกกิจกรรมในอดีต ไม่ใช่แค่กิจกรรมนี้)
  - **"No-Show This Event"** (`filterEventNoShow`) — กรองเฉพาะนักศึกษาที่ลงทะเบียนกิจกรรมนี้ แต่ไม่เคยเช็คอินเข้าเซสชันใดของกิจกรรมนี้เลย คำนวณจาก `eventNoShowIds` (useMemo ใหม่) ที่ mirror ตรรกะเดียวกับ `findNoShowStudentIds` ใน apply-strikes route — ให้ผลตรงกับรายชื่อที่จะโดน strike จริงถ้ากด "Strike No-shows"
  - ทั้งสองปุ่มไม่ถูกซ่อนสำหรับ thin-roster role (`smo`/`club_president`/`major_president`) เหมือนปุ่ม medical filter — เพราะ strike count ไม่ใช่ข้อมูลอ่อนไหวแบบ PDPA
- **ผลลัพธ์ที่ผู้ใช้จะเห็น**: staff/organizer/registration/president ที่เปิด roster กิจกรรมไหนก็ได้ กดกรองดูได้ทันทีว่าใครเคยโดน strike มาก่อน (ทั้งบัญชี) หรือใครไม่มาเช็คอินกิจกรรมนี้เลย โดยไม่ต้องไล่เช็คทีละคน — ใช้ประกอบก่อนกดปุ่ม "Strike No-shows" ได้

### PDPA / สิทธิ์เข้าถึง (PR #161)

- **ไม่มีข้อมูลทางการแพทย์ใหม่ถูกเปิด** — `noShowCount` ที่เพิ่มเข้า roster เป็นแค่ตัวเลขนับ strike ไม่ใช่ medical detail/emergency contact ซึ่งยังคงจำกัดเฉพาะ `FULL_USER_COLUMNS`/admin เท่านั้นเหมือนเดิม
- **ขยายสิทธิ์ครั้งที่ 3 ในรอบสัปดาห์นี้ให้ role ที่เคย scoped/จำกัดกว่า** — `registration` (เดิมเป็น scanner/roster-facing เท่านั้น ไม่เคยแตะ action เชิงลงโทษ) ได้สิทธิ์ apply strike + resolve appeal แบบไม่ถูก scope เท่า admin เป็นครั้งแรก ส่วน `smo` ถูกลดสิทธิ์ลง (ยิง strike ไม่ได้แล้ว) — ควรตรวจสอบบน production ว่า `registration` ใช้สิทธิ์นี้ตามที่ตั้งใจจริง เพราะ apply-strikes/resolve-appeal เป็น action ที่กระทบ `registrationBlocked`/`noShowCount` ของนักศึกษาโดยตรง
- **audit log ยังเขียนครบทุกครั้งที่ apply strike / approve / reject** ไม่เปลี่ยนจากเดิม (ผ่าน `AuditService` ในทรานแซกชันเดียวกับการเขียนข้อมูล)
