## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **(PR #162) ไม่กระทบนักศึกษาโดยตรง** — เป็นการแก้ UI ฝั่ง staff/admin ล้วน ๆ (หน้าจัดการรายชื่อผู้เข้าร่วมกิจกรรม) ไม่มีหน้าจอหรือ flow ใดที่นักศึกษาใช้เปลี่ยนไป
- **(PR #164) กิจกรรมที่มีโควตาจำกัด จะไม่ถูก "ทีมงาน" แย่งที่นั่งอีกต่อไป** — เดิมถ้า staff/organizer ลงทะเบียนกิจกรรมตัวเองเพื่อเข้าไปสแกน QR ก็นับรวมเป็น 1 ที่นั่งในโควตาเหมือนนักศึกษาทั่วไป ตอนนี้แอดมินกำหนดได้ว่าใครเป็น "ทีมงานของกิจกรรมนี้" ล่วงหน้า คนกลุ่มนี้ลงทะเบียน/เช็คอินได้โดยไม่กินโควตาที่นั่งของผู้เข้าร่วมจริง (รวมถึงโควตาแยกไทย/นานาชาติด้วย)
- **(PR #164) ทีมงานที่คุมกิจกรรมจะไม่ถูกตัดคะแนน/ขึ้น strike "ไม่มาเช็คอิน"** เหมือนที่เคยเกิดขึ้นได้ — เพราะปกติคนที่ยืนสแกนบัตรจะไม่มีใครมาสแกนให้ตัวเอง เดิมกรณีนี้เข้าเงื่อนไข no-show ผิด ๆ ตอนนี้ระบบรู้แล้วว่าเป็นทีมงานจึงไม่ตัดคะแนนให้
- **(PR #164) เช็คอินสายหลังโดน strike ไปแล้วยังทำได้** (แก้บั๊กเล็กที่ติดมาในพีอาร์เดียวกัน) — เดิมถ้ากิจกรรมกด "Strike No-shows" ไปแล้ว คนที่แถวถูกเปลี่ยนเป็น `no_show` จะสแกน QR เช็คอินไม่ได้อีกเลยแม้จะมาสายจริง (ระบบขึ้น "already checked in" ให้ผิด ๆ) ตอนนี้สแกนเช็คอินได้ตามปกติ (คะแนนที่ถูกตัดไปแล้วไม่ได้คืนอัตโนมัติ ต้องให้ organizer พิจารณาเป็นกรณีไป)

### กำหนด "ทีมงานกิจกรรม" รายบุคคล + ยกเว้นโควตา/no-show strike + ปรับ export (PR #164)

- **ที่มา:** เดิมระบบไม่มีคอนเซปต์ "staff ของกิจกรรมนี้" แบบชัดเจน — เดาจากใครบังเอิญเป็นคนถือเครื่องสแกน QR เท่านั้น ทำให้ (1) staff ที่ลงทะเบียนกิจกรรมตัวเองไปกินโควตาที่นั่งของผู้เข้าร่วมจริง และ (2) staff ที่ไม่มีใครสแกนให้ตัวเองเสี่ยงโดน strike "ไม่มาเช็คอิน" ทั้งที่อยู่ในงานตลอด — PR นี้แก้ทั้งคู่ด้วยการให้แอดมิน "assign" คนเป็นทีมงานล่วงหน้าแบบ explicit ต่อกิจกรรม แทนการเดา

- **`src/db/schema.ts`**
  - `events.staffUserIds` — คอลัมน์ใหม่ `jsonb` (`string[]`, nullable/ไม่มี default) เก็บ user id ของคนที่แอดมัน assign เป็นทีมงาน "กิจกรรมนี้โดยเฉพาะ" แยกจาก `managedByRoles`/`ownerClubIds`/`ownerMajors` เดิมที่ตอบคำถาม "ใครมีสิทธิ์จัดการ config กิจกรรม" — ตัวใหม่นี้ตอบคำถาม "ใครทำงานหน้างานจริง" (เช่น ยืนสแกน QR)
  - `attendance.isStaff` — คอลัมน์ใหม่ `boolean not null default false` เป็น **snapshot ตอน insert** ว่าตอนลงทะเบียน/เช็คอินคนนี้อยู่ใน `staffUserIds` ของกิจกรรมหรือไม่ — ตั้งใจไม่ผูกกับ `staffUserIds` แบบ live เพื่อให้ประวัติการเช็คอินเก่าไม่เปลี่ยนความหมายย้อนหลังถ้าแอดมินแก้ลิสต์ทีมงานทีหลัง

- **`src/app/api/events/[id]/register/route.ts`** — คำนวณ `isEventStaff` จาก `event.staffUserIds.includes(userId)` แล้วข้ามการเช็คโควตาทั้ง 3 ชุด (โควตารวม, โควตาไทย, โควตานานาชาติ) ทั้งหมดเมื่อเป็นทีมงาน โควตารวมเองก็เปลี่ยนไปนับเฉพาะแถวที่ `isStaff = false` ด้วย (กันไม่ให้ staff ที่ลงทะเบียนไปแล้วไปเบียดที่นั่งคนอื่นทางอ้อม); แถว `attendance` ที่ insert เขียน `isStaff: isEventStaff` ติดไปด้วย

- **`src/modules/events/scanner.service.ts`**
  - เพิ่ม `isEventStaff` (เช็คจาก `event.staffUserIds` เทียบกับ `student.id`) แล้วส่ง `isStaff: isEventStaff` ติดไปกับทุก path ที่ insert แถว `attendance` ใหม่ (ทั้ง pre-registered confirm และ walk-in)
  - walk-in ยกเว้นการเช็คโควตาเช่นเดียวกัน (`!isEventStaff && (event.quota !== null || sessionWalkIn !== null)`) และการนับจำนวนที่ใช้เช็ค cap (`totalCap`) กับ sub-limit ของ walk-in เปลี่ยนไปนับเฉพาะ `isStaff = false`
  - **บั๊กฟิกที่ติดมาในพีอาร์เดียวกัน (ไม่เกี่ยวกับ staff โดยตรง):** เงื่อนไข atomic update ตอน `action === "confirm"` เดิม `WHERE status = 'registered'` เปลี่ยนเป็น `WHERE status IN ('registered', 'no_show')` — เพราะ apply-strikes จะพลิกแถวที่ไม่มาเช็คอินเป็น `no_show` ถ้าไม่ปรับเงื่อนไขนี้ คนที่โดน strike ไปแล้วแต่มาสายจะสแกนเช็คอินไม่ได้อีกเลย (โดนบอก "already checked in" ผิด ๆ) — คะแนนที่ตัดไปแล้วจาก strike **ไม่ถูกคืนอัตโนมัติ** เป็น action แยกที่ organizer ต้องพิจารณาเอง

- **`src/app/api/admin/events/[id]/apply-strikes/route.ts`** — `findNoShowStudentIds` เพิ่มเงื่อนไข `eq(attendance.isStaff, false)` ตอน query แถว `registered` ที่ยังไม่ได้เช็คอิน — ทีมงานที่ลงทะเบียนกิจกรรมตัวเองแต่ไม่มีใครสแกนให้จะไม่ถูกนับเป็น no-show และไม่โดนตัดคะแนน/บล็อก registration

- **`src/app/api/admin/events/route.ts`** — `getAttendeeCounts` (ตัวเลข "X registered" ที่โชว์บนการ์ดกิจกรรมแต่ละใบในหน้า `/admin/events`) เพิ่ม `.where(eq(attendance.isStaff, false))` ก่อน group by — ตัวเลขที่แอดมินเห็นตอนนี้คือจำนวนผู้เข้าร่วมจริง ไม่รวมทีมงาน; endpoint สร้าง/แก้กิจกรรม (`POST`/`PUT` ทั้งสองไฟล์ `route.ts` และ `[id]/route.ts`) รับ/บันทึก `staffUserIds` เพิ่มเข้า zod schema และ payload ที่ insert/update

- **`src/app/api/admin/events/[id]/export/route.ts`** (Excel export) — ทำเพื่อให้ไฟล์ export ใช้เป็นหลักฐานจำนวนผู้เข้าร่วมจริงประกอบเอกสารปิดโครงการได้ โดยแยกทีมงานออกจากผู้เข้าร่วมชัดเจน:
  - เพิ่มคอลัมน์ **"Staff"** ต่อคน (Yes/ว่าง) ในชีตรายชื่อ ถัดจาก House
  - Summary sheet เปลี่ยนคอลัมน์ "Distinct Students" เป็น 2 คอลัมน์แยก: **"Staff"** (จำนวนทีมงานที่นับ distinct) และ **"Attendees (excl. Staff)"** (`distinct - staff` = ผู้เข้าร่วมจริงล้วน ๆ) ทั้งในตารางแยกวัน (multi-day) และแถวรวมทั้งกิจกรรม

- **`src/app/admin/events/page.tsx`** (+540/-… บรรทัด, ไฟล์ใหญ่ที่สุดในพีอาร์นี้)
  - **Event editor:** เพิ่มฟิลด์ใหม่ "Event Staff" — person-picker ค้นหาด้วยชื่อ/รหัสนักศึกษา เลือกได้หลายคน (mirror UI pattern เดียวกับ assignee-picker ของ S-form ที่มีอยู่แล้ว ใช้ people-directory ร่วมกันผ่าน `ensureAssigneeUsersLoaded()` ตัวใหม่ที่ dedup การโหลดซ้ำ) เป็นฟิลด์ staff-only เหมือน "Managed By" เดิม (ล็อกไม่ให้แก้ถ้า `canEditRestrictedFields` เป็น false)

### PDPA / สิทธิ์เข้าถึง (PR #164)

- **ไม่มีข้อมูลทางการแพทย์ใหม่ถูกเปิด** — `staffUserIds`/`isStaff` เป็นแค่ user id list และ boolean flag บอกบทบาทในกิจกรรม ไม่ใช่ medical detail/emergency contact ซึ่งยังคงจำกัดเฉพาะ `FULL_USER_COLUMNS`/admin เหมือนเดิม
- **field `staffUserIds` เป็น staff-only** เหมือน `managedByRoles`/`ownerClubIds`/`ownerMajors` เดิม — server (`eventUpdateSchema`/`eventSchema` gate ผ่าน `canEditRestrictedFields` เดิม) ยังเป็น source of truth ตัวจริง ไม่ใช่แค่ disable ปุ่มฝั่ง UI
- **ต้องรัน migration บน production ก่อน deploy โค้ดนี้** — โค้ดที่อ่าน/เขียน `events.staff_user_ids` และ `attendance.is_staff` (register route, scanner service, apply-strikes, admin events route/export, event editor UI) จะพังทันทีถ้าคอลัมน์ยังไม่มีบน production — ต้องรัน `npm run db:migrate:container` จาก Portainer console ก่อน recreate container ตาม `/safe-deploy` เสมอ; migration idempotent/non-destructive ตามกติกาเดิม (`ADD COLUMN IF NOT EXISTS`, ไม่มี `DELETE`)
