## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **สมัครใช้งานเสร็จ (กรอกโปรไฟล์ครบ) ก็ได้บ้านสี (แดง/เขียว/เหลือง/น้ำเงิน) ทันที** ไม่ต้องรอไปเช็คอินกิจกรรมแรกก่อนเหมือนเดิม — เดิมถ้าใครสมัครเสร็จแล้วยังไม่เคยไปสแกน QR เข้าร่วมกิจกรรมเลย จะไม่มีบ้านค้างอยู่แบบนั้นไปเรื่อย ๆ (หน้า dashboard/profile/leaderboard โชว์บ้านว่างเปล่า) เพราะยิ่งวันผ่านไปคนสมัครใหม่ยิ่งเยอะกว่าคนไปเช็คอินจริง กลุ่มคนไม่มีบ้านจึงพอกพูนขึ้นทุกวัน ตอนนี้แก้ที่ต้นตอแล้ว
- บ้านที่ได้ยังคงเลือกแบบเป็นธรรมเหมือนเดิม คือสีที่มีคนอยู่น้อยที่สุดในคณะเดียวกันของนักศึกษาคนนั้น (ไม่ใช่สุ่ม/ไม่ใช่เลือกเอง) — แค่ย้ายจังหวะที่ตัดสินใจให้เร็วขึ้น

## ฝั่งทีม (technical changelog)

### ย้ายจุดจัดสรรบ้านสี จาก "เช็คอินครั้งแรก" มาเป็น "onboarding เสร็จ" (PR #167)

- **ที่มา:** ตั้งแต่ migration รีเซ็ต `house_id` ของทุกคนเป็น NULL (step 65) การจัดสรรบ้านออกแบบให้ทำตอน `ScannerService.ensureHouseAssigned` (ตอนสแกน QR เช็คอินครั้งแรก) เท่านั้น — คนที่สมัครเสร็จแต่ไม่เคยไปงานเลยจึงไม่มีบ้านค้างอยู่ไม่มีกำหนด และกลุ่มนี้โตขึ้นเรื่อย ๆ เพราะอัตราสมัครสูงกว่าอัตราเข้าร่วมกิจกรรม PR นี้ย้าย logic หลักมาที่ onboarding แทน
- **`src/app/api/profile/route.ts`** (`POST`, onboarding submit) — ในทรานแซกชันเดียวกับที่ตั้ง `profileCompleted = true` และเขียน audit log: ล็อกด้วย `pg_advisory_xact_lock(HOUSE_BALANCE_LOCK_KEY)` (ล็อกตัวเดียวกับที่ `UsersService.provisionStaffBypass` ใช้อยู่แล้วตอน provision บัญชี staff) ก่อนเรียก `HousesService.pickBalancedHouseIdForFaculty(faculty, tx)` แล้วเซ็ต `houseId: sql\`COALESCE(${users.houseId}, ${pickedHouseId})\`` — ใช้ `COALESCE` กันไม่ให้ทับ `houseId` ที่แอดมินอาจตั้งมือไว้ก่อนหน้าแล้ว
- **`src/modules/houses/houses.service.ts`** — `pickBalancedHouseIdForFaculty` เพิ่ม parameter `executor: DbOrTx = db` (เดิม hardcode `db` ตรง ๆ) ให้เรียกจากใน transaction ของ route ข้างบนได้ เพื่อให้การนับจำนวนสมาชิกต่อบ้าน + การเขียน houseId ถูก serialize ภายใต้ advisory lock เดียวกัน กันสองคนสมัครพร้อมกันแล้วอ่าน "บ้านนี้คนน้อยสุด" ตรงกันทั้งคู่แล้วยัดเข้าบ้านเดียวกันพร้อมกัน (race)
- **`src/modules/events/scanner.service.ts`** (`ensureHouseAssigned`) — เก็บโค้ดเดิมไว้ทั้งหมด แต่ปรับเป็น **fallback safety net เท่านั้น**: ทำงานจริงเฉพาะกรณีนักศึกษาที่ onboard ไปแล้วก่อนพีอาร์นี้ หรือถูกแอดมินเคลียร์ `houseId` ทีหลัง ยังไม่ได้ event check-in ปกติของคนส่วนใหญ่ตอนนี้จึง no-op เพราะมีบ้านอยู่แล้วตั้งแต่ onboarding
- **`scripts/backfill-user-houses.mjs`** — query เดิม `WHERE house_id IS NULL` เปลี่ยนเป็น **`WHERE house_id IS NULL AND profile_completed = true`** — สคริปต์นี้มีไว้กวาดคนที่ onboard เสร็จไปแล้วภายใต้ระบบเก่าแต่ตกหล่นไม่มีบ้าน ถ้าไม่กรองด้วย `profile_completed = true` จะไปแตะคนที่ยังกรอกโปรไฟล์ไม่เสร็จ (ยังไม่ได้เลือกคณะ) ซึ่งจะเดาคณะ default เป็น CAMT แล้วล็อกบ้านผิดคณะให้ก่อนที่เจ้าตัวจะเลือกจริงด้วยซ้ำ — คนกลุ่มนี้ปล่อยไว้ ให้ตอน submit onboarding จัดสรรให้เองแบบสมดุลถูกต้อง

### PDPA / สิทธิ์เข้าถึง / DB (PR #167)

- **ไม่มีข้อมูล PDPA/การแพทย์เกี่ยวข้อง และไม่มี DB migration** — เป็นการย้ายจังหวะ/ปรับ logic การจัดสรร `houseId` (ซึ่งไม่ใช่ข้อมูลอ่อนไหว) ล้วน ๆ ไม่มีคอลัมน์ใหม่ ไม่แตะ role/gate ใด ๆ
- **audit log ของ onboarding เดิมยังคงเขียนในทรานแซกชันเดียวกัน** ไม่เปลี่ยน — แค่เพิ่มขั้นตอน advisory lock + house pick เข้าไปในทรานแซกชันเดิมนั้นด้วย
