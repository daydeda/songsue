## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **ประธานชมรม (club_president) เสนอกิจกรรมของชมรมตัวเองให้ทีมงานพิจารณาได้เลยจากหน้า "Clubs"** — กรอกชื่อ เวลาเริ่ม/จบ สถานที่ โควตา รูปโปสเตอร์ ช่วงเวลาเปิด/ปิดรับสมัคร กลุ่มเป้าหมาย (ไทย/นานาชาติ) เปิด/ปิด walk-in หรือ "รับเฉพาะ walk-in" โควตาแยกไทย/นานาชาติ จำกัดเฉพาะปี 1 เสนอชื่อทีมงานที่ช่วยดูแลงาน และตั้งได้หลายวัน (multi-session) — ส่งแล้วดูสถานะ "ข้อเสนอของฉัน" ได้เอง (รอตรวจสอบ/อนุมัติแล้ว/ปฏิเสธแล้ว/ถอนแล้ว) และถอนข้อเสนอเองได้ถ้ายังไม่ถูกตรวจสอบ — **ประธานชมรมยังไม่มีสิทธิ์สร้างกิจกรรมจริงเอง** ทุกอย่างเป็นแค่ "คำขอ" ที่ทีมงาน admin/registration/organizer ต้องรีวิวแล้วสร้างกิจกรรมจริงให้อีกที (ไม่รวม SMO — SMO มีสิทธิ์แค่หน้าสแกน QR เท่านั้น)
- ประธานชมรมเห็น/เสนอได้เฉพาะชมรมที่ตัวเองเป็นประธานจริงเท่านั้น (ระบบเช็คให้อัตโนมัติ ไม่ใช่เลือกเอง)
- **กิจกรรมที่ทีมงานตั้งเป็น "Walk-in เท่านั้น"** จะไม่มีปุ่ม "Register Now" ให้กดล่วงหน้าอีกต่อไป — หน้ารายละเอียดกิจกรรมโชว์ป้าย "Walk-in เท่านั้น" พร้อมข้อความอธิบายว่ามาถึงแล้วสแกนเข้าร่วมได้เลย ไม่ต้องลงทะเบียนล่วงหน้า
- เมนูฝั่งแอดมิน/staff (ที่มองเห็นตามสิทธิ์ของแต่ละคน) ถูกจัดกลุ่มใหม่ให้อ่านง่ายขึ้น เป็นหมวด Events / Community / Content / System แทนรายการยาวเรียงเดียว — ไม่กระทบสิทธิ์เข้าถึงเดิม แค่จัดหมวดหมู่
- **หน้า "Manage Events" ของแอดมิน/staff (`/admin/events`) เรียงลำดับใหม่ให้ดูง่ายขึ้น** — กิจกรรมที่กำลังจัดอยู่ (live) ขึ้นก่อน ตามด้วยกิจกรรมที่ยังไม่ถึง (upcoming) ส่วนกิจกรรมที่จบไปแล้ว (past) จะจมไปอยู่ท้ายลิสต์เสมอ ไม่ปนกับกิจกรรมที่ยังใช้งานอยู่แบบเมื่อก่อน — ในกลุ่ม live/upcoming เรียงตามเวลาที่ใกล้เริ่มที่สุดก่อน ส่วนกลุ่ม past เรียงตามกิจกรรมที่เพิ่งจบล่าสุดก่อน
- **การ์ดกิจกรรมที่จบไปแล้วถูกทำให้จางลงและออกโทนขาวดำ (grayscale)** ให้ต่างจากกิจกรรม live/upcoming อย่างชัดเจน แค่กวาดตาก็รู้ทันทีว่ากิจกรรมไหนจบไปแล้ว ไม่ต้องเข้าไปเช็คทีละใบเหมือนก่อน

## ฝั่งทีม (technical changelog)

### Event Proposals — ประธานชมรมเสนอกิจกรรม ทีมงานรีวิว/สร้างจริง (PR #168)

- **ตารางใหม่ `event_proposals`** (`src/db/schema.ts`) — เก็บคำขอกิจกรรมของ club_president แยกจาก `events` โดยสิ้นเชิง ฟิลด์ mirror ของ `events` เกือบทั้งหมด (title/description/start-end/registration window/location/quota/poster/walkInsEnabled/walkInsOnly/quotaWalkIn/registrationMode/sessions/targetThai/targetInternational/quotaThai/quotaInternational/firstYearOnly/staffUserIds) แต่ **ค่าที่เสนอมาทั้งหมดไม่ผูกมัด (non-binding)** — `pointsAwarded`, `allowedRoles`, `allowedMajors`, `managedByRoles`, `ownerClubIds` staff ต้องกำหนดเองตอนสร้างกิจกรรมจริง (mirror หลักการ field-strip เดิมของการแก้ไขกิจกรรมโดยประธาน)
  - lifecycle: `status` = `pending` → `approved` / `rejected` / `withdrawn`; `reviewedBy` ไม่มี FK (mirror `noShowAppeals.reviewedBy`); `clubId` cascade delete (ลบชมรม = ลบประวัติข้อเสนอ, audit log ยังเก็บ trail แยกไว้); `resultingEventId` เป็น `set null` กันไม่ให้การลบกิจกรรมจริงทีหลัง block การลบด้วย FK
  - คอลัมน์ใหม่บน `events`: **`walk_ins_only`** (boolean, default false) — กิจกรรมที่ปิดการลงทะเบียนล่วงหน้าทั้งหมด รับเฉพาะ walk-in หน้างาน
- **`src/db/migrate.ts`** เพิ่ม step 72–74: step 72 `CREATE TABLE IF NOT EXISTS event_proposals` + 3 index (club_id, status, proposed_by), step 73 เติมคอลัมน์ poster/registration-window/walk-ins/audience/staff ทีละคอลัมน์ด้วย `ADD COLUMN IF NOT EXISTS`, step 74 `events.walk_ins_only` — ทั้งหมด additive/idempotent ตามกติกาเดิม ไม่มี `DELETE`/lossy step ใด ๆ (migration files `drizzle/0012`–`0015`)
- **`src/lib/event-proposals.ts`** (ใหม่) — role constants: `SUBMIT_PROPOSAL_ROLES = ["club_president"]` (ไม่รวม major_president เพราะ scope ผูกกับ clubId เท่านั้น), `REVIEW_PROPOSAL_ROLES = ["super_admin","admin","registration","organizer"]` (เซตเดียวกับที่สร้างกิจกรรมจริงได้อยู่แล้ว)
- **`src/modules/events/event-proposals.service.ts`** (ใหม่) — `getSubmittableClubIds`/`isClubInScope` ดึง scope ของประธานชมรมจาก `EventScopeService.getPresidentScope` เดิม (แหล่งเดียวกับที่ event-ownership ใช้) **ไม่เคยเชื่อ clubId ที่ส่งมาจาก client**; `listForViewer` staff เห็นทุกข้อเสนอ (filter ตาม status ได้), ประธานชมรมเห็นเฉพาะของชมรมตัวเอง — คำนวณ scope ใหม่จาก userId/roles ของ viewer เองเสมอ ไม่รับ scope จาก client
- **API ใหม่:**
  - `GET/POST /api/admin/event-proposals` — GET list (staff เห็นทั้งหมด + filter `?status=`, ประธานชมรมเห็นเฉพาะของตัวเอง); POST submit ข้อเสนอ — เช็ค `isClubInScope` ก่อนเสมอ (403 ถ้าไม่ใช่ชมรมตัวเอง), เช็คชมรมไม่ถูก archive, และ **`staffUserIds` ที่เสนอมาถูกกรองให้เหลือเฉพาะคนที่เป็นสมาชิกชมรมนั้นจริง** (query จาก `ClubsService.getClubMembers` ไม่เชื่อ id ที่ client ส่งมา) — เขียน audit log ทุกครั้งที่ submit
  - `GET/PATCH /api/admin/event-proposals/[id]` — GET รายละเอียด (staff-only, ใช้ prefill ตอนสร้างกิจกรรมจากข้อเสนอ); PATCH รองรับ `reject` (staff เท่านั้น) กับ `withdraw` (เจ้าของข้อเสนอเท่านั้น, ต้องเป็น proposal ของตัวเอง) — ทั้งคู่ re-assert `status='pending'` ใน `UPDATE...WHERE` เดียวกับตอนอัปเดต (optimistic guard แบบเดียวกับ `/api/admin/appeals/[id]`) กัน race เช่นกดปฏิเสธซ้ำ/กดพร้อมกับ approve
  - **ไม่มี action "approve" แยก** — การอนุมัติเป็น side effect ของการที่ staff สร้างกิจกรรมจริงจากข้อเสนอ (ดูข้อถัดไป) ไม่ใช่ mutation เดี่ยว ๆ
- **`POST /api/admin/events`** — รับ `proposalId` (optional) เป็น linkage marker เฉยๆ: staff ยังต้องกรอกทุกฟิลด์ของกิจกรรมจริงเองครบ (รวมฟิลด์อ่อนไหวที่ประธานชมรมไม่มีสิทธิ์ตั้ง เช่น `pointsAwarded`, `allowedRoles`, `allowedMajors`, `managedByRoles`, `ownerClubIds`, `staffUserIds`) — ถ้ามี `proposalId` มาด้วย จะ `UPDATE event_proposals SET status='approved', resultingEventId=<กิจกรรมใหม่> WHERE id=proposalId AND status='pending'` **ในทรานแซกชันเดียวกับการสร้างกิจกรรม** (ไม่ใช่ endpoint แยก) ถ้าข้อเสนอถูกจัดการไปแล้วพร้อมกัน (double-submit/race) จะไม่ error แค่ข้าม การสร้างกิจกรรมยังสำเร็จตามปกติ — audit log บันทึกทั้งสองกรณี; role check เปลี่ยนจาก `session.user.role` เดี่ยว ๆ เป็น `effectiveRoles` (กัน staff ที่มี role รองเป็น admin-set แต่ role หลักไม่ใช่ ถูกปฏิเสธผิด ๆ เหมือนที่ `PUT /[id]` แก้ไปแล้วก่อนหน้า)
- **UI ใหม่:** `src/app/admin/clubs/ProposeEventSection.tsx` (ฟอร์มส่งข้อเสนอ + "ข้อเสนอของฉัน" ของประธานชมรม, ต่อจากหน้า `/admin/clubs`) และ `src/app/admin/proposals/` (`page.tsx` + `EventProposalsClient.tsx` — คิวรีวิวของ staff, filter ตามสถานะ, ปุ่ม Reject พร้อม modal ใส่เหตุผล, ปุ่ม "Create Event from Proposal" ที่พาไปหน้าสร้างกิจกรรมพร้อม prefill)
- **Walk-ins-only (`walkInsOnly`):** ฟีเจอร์ย่อยที่มากับพีอาร์นี้ (ใช้ได้ทั้งกิจกรรมจริงและข้อเสนอ) — `POST /api/events/[id]/register` ปฏิเสธการลงทะเบียนล่วงหน้าทั้งหมดถ้า `event.walkInsOnly = true` (ยกเว้น staff/event-staff ที่ต้องลงทะเบียนเพื่อเข้าไปทำงาน) ด้วย 403; ฝั่ง client (`DashboardClient.tsx`) ปุ่ม Register ถูกปิดใช้งานและแสดงป้าย "Walk-in เท่านั้น" แทน พร้อมข้อความอธิบายใต้การ์ดกิจกรรม; ตั้งค่าได้จากตัวแก้ไขกิจกรรมของ staff (toggle ใหม่ auto เปิด `walkInsEnabled` ให้ด้วยเสมอเมื่อเปิด `walkInsOnly`) และเป็นหนึ่งในฟิลด์ที่ประธานชมรมเสนอได้ในข้อเสนอ
- **`src/components/admin/AdminNav.tsx`** — จัดกลุ่ม nav list เดิม (flat array) เป็น `NAV_GROUPS` แบบมีหัวข้อย่อย: Overview (ไม่มีหัวข้อ) → Events (Manage Events, QR Scanner, **Proposals ใหม่**, Appeals) → Community (Clubs, Students) → Content (Announcement, Shop) → System (Audit Trails) — กลุ่มไหนไม่มี item ที่ role ปัจจุบันมีสิทธิ์เห็นเลยจะถูกซ่อนทั้งกลุ่ม; เมนู "Proposals" gate ด้วย `REVIEW_PROPOSAL_ROLES` เดียวกับ API (organizer/registration เห็นได้ ต่างจาก Appeals ที่จำกัดแค่ admin/super_admin) — เพิ่ม i18n key ใหม่ 4 คีย์ (`navGroupEvents/Community/Content/System`) ครบทั้ง 4 ภาษา (EN/TH/MM/CN) ใน `src/lib/i18n.ts`
- **QA:** ตรวจสอบเพิ่มด้วย route-gate-auditor subagent (ผลลัพธ์: PASS ไม่พบช่องโหว่สิทธิ์เข้าถึง) และ manual end-to-end smoke test บน local dev DB จริง (submit ข้อเสนอ → ยืนยันว่า role นักศึกษาทั่วไปเห็นคิวรีวิวไม่ได้ 401 → staff แปลงข้อเสนอเป็นกิจกรรมจริง → ยืนยันแถวข้อเสนอพลิกเป็น `status='approved'` พร้อม `resultingEventId` ถูกต้อง)

### PDPA / สิทธิ์เข้าถึง / DB (PR #168)

- **ไม่มีข้อมูลทางการแพทย์เกี่ยวข้อง** — ฟีเจอร์นี้เป็น event logistics ล้วน ๆ (ชื่อ/เวลา/สถานที่/โควตา/โปสเตอร์ ฯลฯ) ไม่แตะ medical detail/emergency contact ใด ๆ
- **scope ของประธานชมรมคำนวณฝั่ง server เสมอ ไม่เชื่อ client** — ทั้ง `clubId` ที่ส่งเสนอ และ `staffUserIds` ที่แนะนำ ถูก re-verify กับข้อมูลจริง (`EventScopeService.getPresidentScope`, `ClubsService.getClubMembers`) ทุกครั้งก่อนเขียนลง DB
- **คิวรีวิวและปุ่ม reject ถูก gate ด้วย `REVIEW_PROPOSAL_ROLES` ฝั่ง server จริง** ไม่ใช่แค่ซ่อนปุ่มฝั่ง UI — ฟิลด์อ่อนไหวของกิจกรรมจริง (points/allowed roles-majors/managed-by/owner clubs/staff) ยังคงเป็นสิ่งที่ประธานชมรมตั้งเองไม่ได้ ต้องให้ staff กรอกตอนแปลงข้อเสนอเป็นกิจกรรมจริงเท่านั้น (mirror field-strip precedent เดิม)
- **ต้องรัน migration บน production ก่อน deploy โค้ดนี้** — โค้ดที่อ่าน/เขียน `event_proposals` และ `events.walk_ins_only` จะพังทันทีถ้าคอลัมน์/ตารางยังไม่มีบน production — รัน `npm run db:migrate:container` จาก Portainer console ก่อน recreate container ตาม `/safe-deploy` เสมอ; migration เป็น additive/idempotent ตามกติกาเดิม (`CREATE TABLE IF NOT EXISTS`, `ADD COLUMN IF NOT EXISTS`, ไม่มี `DELETE`)

### Admin Events list — เรียงตามสถานะ + จางกิจกรรมที่จบแล้ว (PR #169)

- **`src/app/admin/events/page.tsx`** — comparator ใน `useMemo` ที่ใช้เรียงลิสต์กิจกรรมเปลี่ยนเป็นเรียงตาม bucket สถานะก่อน: `live` → `upcoming` → `past` (กิจกรรม past จะไม่ถูกเรียงปนกับ live/upcoming อีกต่อไป) จากนั้นค่อยเรียงในแต่ละ bucket — `live`/`upcoming` เรียงตาม `startTime` ใกล้สุดก่อน (soonest-first), `past` เรียงตาม `endTime` เพิ่งจบล่าสุดก่อน (most-recently-ended-first)
- การ์ดกิจกรรมที่อยู่ใน bucket `past` เพิ่ม inline style แบบมีเงื่อนไข: `filter: grayscale(0.85)` + `opacity: 0.6` และเปลี่ยนพื้นหลังจาก `--bg-surface` เป็น `--bg-elevated` (จางลงเล็กน้อย) — แก้ปัญหา UX เดิมที่การ์ดกิจกรรมจบแล้วกับกิจกรรมที่ยังใช้งานอยู่หน้าตาเหมือนกันทุกประการ ทำให้แอดมิน/staff สับสนว่ากิจกรรมไหนจบไปแล้ว
- เป็นการเปลี่ยนแปลงฝั่ง client ล้วน ๆ (sort comparator + conditional inline style) — **ไม่มีการแก้ schema, ไม่มี API ใหม่, ไม่กระทบ access control หรือ PDPA**
- ตรวจสอบด้วย `npm run lint` (ผ่าน มี warning เดิมที่ไม่เกี่ยวข้องเท่านั้น) และ `npm run build` (ผ่าน)
