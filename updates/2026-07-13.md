## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **ประธานสาขาวิชา (major_president) เสนอกิจกรรมของสาขาตัวเองได้แล้ว** จากหน้า "Majors" ใหม่ (เหมือนที่ประธานชมรมเสนอกิจกรรมได้อยู่แล้วที่หน้า "Clubs") — กรอกรายละเอียดกิจกรรมแบบเดียวกัน แถมเลือก "แนะนำ" กลุ่มเป้าหมายที่เข้าร่วมได้ (ตามบทบาท/สาขา/ชมรม) ให้ staff พิจารณาประกอบตอนสร้างกิจกรรมจริง — ค่าที่เสนอมายังไม่ผูกมัด staff ยังเป็นคนตั้งค่าจริงเองเสมอ
- **ข้อเสนอกิจกรรมที่ถูกปฏิเสธ แก้ไขแล้วส่งใหม่ได้เลยในหน้าเดิม** ไม่ต้องสร้างข้อเสนอใหม่ทั้งใบ (ใช้ได้ทั้งประธานชมรมและประธานสาขา)
- **แบบฟอร์มประเมิน/feedback บางแบบอาจโชว์ชื่อ-รหัสนักศึกษา-เบอร์ติดต่อผู้ตอบให้ทีมงาน (registration/organizer) เห็นได้** ถ้าเจ้าของฟอร์มเปิดไว้เท่านั้น — **ค่าเริ่มต้นคือไม่โชว์ตัวตนผู้ตอบเสมอ (anonymized)** สำหรับทุกแบบฟอร์ม ทีมงานระดับ admin เห็นได้อยู่แล้วไม่ว่าตั้งค่าอย่างไร
- **ประธานชมรม/สาขา แก้ไขรายละเอียดกิจกรรมหรือฟอร์มของตัวเองได้เสมอ** แม้ staff จะอนุมัติไปแล้วก็ตาม (ไม่ถูกล็อกห้ามแก้) แต่ทุกครั้งที่แก้ไข ระบบจะส่งกลับไปให้ staff ตรวจสอบใหม่โดยอัตโนมัติ (มีข้อความแจ้งเตือนตอนบันทึกให้รู้)
- **ทีมงาน (staff) มีหน้า "Pending Reviews" ใหม่** รวมทุกอย่างที่รอตรวจสอบไว้ที่เดียว (ทั้งกิจกรรมที่ประธานแก้ไข และฟอร์มที่ประธานสร้าง/แก้ไข) ไม่ต้องไล่เปิดทีละกิจกรรมเหมือนก่อน
- **แก้บั๊ก: staff ที่เพิ่มเข้ากิจกรรมทีหลัง (หลังคนนั้นลงทะเบียน/เช็คอินไปแล้ว) ตอนนี้ขึ้นในหน้า Attendance ฝั่ง staff ถูกต้องแล้ว** — ก่อนหน้านี้จะยังค้างอยู่ในลิสต์นักศึกษาปกติ (เสี่ยงโดน strike ว่าขาดงานทั้งที่เป็น staff)

## ฝั่งทีม (technical changelog)

### Respondent identity toggle + major_president proposals + staff review workflow (PR #176)

- **`forms.showRespondentIdentity`** (`src/db/schema.ts`, boolean, default `false`) — ควบคุมว่าผู้ดู submissions ที่ไม่ใช่ admin (registration/organizer และในอนาคต smo/club_president/major_president) จะเห็นชื่อ/รหัสนักศึกษา/ชื่อเล่น/สาขา/เบอร์/ช่องทางติดต่อของผู้ตอบหรือไม่ — `src/lib/form-access.ts` เพิ่ม `canSeeRespondentIdentity(role, showRespondentIdentity, reviewStatus)`: manager (super_admin/admin) เห็นเสมอ; ตอนที่ `reviewStatus === "pending"` จะซ่อนตัวตนให้ทุกคนยกเว้น manager เสมอ **ปิดช่องโหว่ self-service** ที่ประธานอาจเปิด flag เองแล้วเห็นตัวตนผู้ตอบก่อน staff รีวิว — บังคับ mask ฝั่ง server จริงที่ `GET /api/admin/events/[id]/form` (response ไม่มี field เหล่านั้นเลยถ้าไม่ผ่านเงื่อนไข ไม่ใช่แค่ซ่อนใน UI) พร้อม audit log ทุกครั้งที่ดู (บันทึกจำนวนที่เห็น/ถูก mask ไม่ใช่ตัวข้อมูลจริง)
- **major_president เสนอกิจกรรมได้แล้ว** — `event_proposals.clubId` เปลี่ยนเป็น nullable, เพิ่ม `majorCode` (exactly-one-of บังคับด้วย zod refine ฝั่ง API ไม่ใช่ DB constraint) — `EventProposalsService.isMajorInScope`/`getSubmittableMajors` re-verify สาขาที่ประธานเสนอได้จริงฝั่ง server เสมอ (ไม่เชื่อ client) โดยใช้ scope object เดิม (`EventScopeService.getPresidentScope`) ที่ระบบ major_president ใช้อยู่แล้วสำหรับ events/appeals/strikes — ไม่ได้สร้าง scope ใหม่
- **ACL ที่ "แนะนำ" ใน proposal** — เพิ่ม `allowedRoles`/`allowedMajors`/`allowedClubs` (jsonb) mirror ฟิลด์เดียวกันบน `events` แต่เป็นแค่คำแนะนำ non-binding เหมือนฟิลด์อื่นในข้อเสนอ — staff ยังเป็นคนตั้งค่าจริงเองตอนสร้างกิจกรรมจากข้อเสนอเสมอ
- **Resubmit หลังถูกปฏิเสธ** — `PATCH /api/admin/event-proposals/[id]` เพิ่ม action `resubmit` (เจ้าของ proposal ที่ถูก reject เท่านั้น) — re-verify scope + สถานะ archived ของชมรมใหม่อีกครั้ง (เผื่อถูกถอนสิทธิ์/ชมรมถูก archive ระหว่างนั้น), guard ด้วย optimistic `UPDATE...WHERE status='rejected'` กัน race แบบเดียวกับ pattern เดิมของ `/api/admin/appeals/[id]`
- **Approve-then-auto-re-review สำหรับกิจกรรม/ฟอร์มที่ประธานแก้ไข** — `events.detailsReviewStatus` และ `forms.reviewStatus` (`pending`/`approved`) — การแก้ไขของประธานไม่ถูกบล็อกเลย (`PUT /api/admin/events/[id]`, `PATCH /api/admin/events/[id]/form`) แต่ทุกครั้งจะ reset กลับเป็น `pending` ให้ staff รีวิวใหม่เสมอ; ฟอร์มที่ยัง `pending` จะโชว์ให้นักศึกษาเห็นแต่ปิดรับคำตอบ (`getFormAvailability` คืนสถานะใหม่ `"pending_review"`) — staff แก้ไข/สร้างเองจะ auto-approve ทันที
- **`GET /api/admin/reviews`** (ใหม่) — รวมทุกฟอร์ม/กิจกรรมที่รอรีวิวไว้ที่เดียว, gate ด้วย `REVIEW_PROPOSAL_ROLES` ชุดเดียวกับที่รีวิว proposals — กรองกิจกรรมที่ staff สร้างเองออก (default `pending` แต่ไม่มีอะไรให้รีวิวจริง เพราะ approve ทันทีตอนสร้าง)
- **UI ใหม่:** `src/app/admin/majors/` (page + `MajorProposeEventSection.tsx`), `src/app/admin/reviews/` (page + `PendingReviewsClient.tsx`), `src/components/admin/EventFormBuilderModal.tsx`, `src/components/admin/EventFeedbackFormsShortcut.tsx` — `AdminNav.tsx` เพิ่มเมนู Majors (gate เฉพาะ major_president) และ Pending Reviews (gate เดียวกับ Proposals)
- **`src/db/migrate.ts`** เพิ่ม step 76–80 (mirror `drizzle/0026`–`0028`) ทั้งหมด additive/idempotent (`ADD COLUMN IF NOT EXISTS`, `ALTER COLUMN DROP NOT NULL` เป็น no-op ถ้าถูก drop ไปแล้ว, ไม่มี `DELETE`) — รันซ้ำกับ local Postgres แล้วได้แค่ `⚠️ already exists` NOTICE จบด้วย `✅ Migration complete!` ตามที่คาดไว้

### PDPA / สิทธิ์เข้าถึง (PR #176)

- **major_president ถูกเพิ่มครบทั้ง 4 ชั้นการ gate** (`admin-access.ts` ทุก predicate list รวม `SCANNER_ONLY_PAGES` ที่เพิ่ม `/admin/majors`, `AdminNav.tsx`, proxy ผ่าน predicate เดิม, ทุก API route) — ไม่มีการเพิ่มแบบไม่ครบชั้นแบบที่เคยเกิดปัญหา scanner-role มาก่อน
- **ทุก write path ของ president re-verify scope ฝั่ง server เสมอ** (`EventScopeService`/`isMajorInScope`) กัน IDOR — ไม่เชื่อ id ที่ client ส่งมาเลยไม่ว่ากรณีไหน
- **ตรวจสอบด้วย `/recheck`** (bugs/security/PDPA/perf) + sub-review เฉพาะทางจาก `pdpa-access-guard` agent — เจอจุดเดียวที่ comment ใน `schema.ts`/`migrate.ts`/route ไม่ตรงกับพฤติกรรมจริง (เขียนว่า "ล็อกห้ามแก้" แต่โค้ดจริงคือ auto-re-review ไม่บล็อก) แก้ comment ให้ตรงกับพฤติกรรมจริงแล้วในแพ็กเกจนี้ ไม่พบช่องโหว่ security/PDPA จริง
- **ต้องรัน migration บน production ก่อน deploy โค้ดนี้** — รัน `npm run db:migrate:container` จาก Portainer console ก่อน recreate container ตาม `/safe-deploy` เสมอ
- ตรวจสอบเพิ่มด้วย `npm run build`, `npm run lint`, `npx tsc --noEmit` (ผ่านทั้งหมด ไม่มี warning/error ใหม่จากแพ็กเกจนี้) และรัน migration รีเฮิร์สกับ local Postgres แล้ว
- ตัดแพ็กเกจนี้เป็น **tag/release v1.4.0**

### แก้บั๊ก: staff ที่เพิ่มทีหลังไม่ขึ้นในหน้า Attendance (PR #177)

- **root cause:** `attendance.isStaff` เป็น snapshot ที่ถูกกำหนดค่าครั้งเดียวตอนคนนั้นลงทะเบียน/เช็คอิน (`src/app/api/events/[id]/register/route.ts`) โดยตั้งใจไม่ re-derive จาก `events.staffUserIds` ใหม่ภายหลัง (comment เดิมใน `schema.ts`: "historical accuracy") — ถ้า admin เพิ่มคนเข้า staff **หลังจาก** คนนั้นมี attendance row อยู่แล้ว ค่า `isStaff` จะค้างเป็น `false` ตลอดไป ทำให้หน้า Attendance จัดกลุ่มผิด (ไปอยู่ฝั่งนักศึกษาแทนฝั่ง staff) และเสี่ยงโดนนับ quota/ตีตราขาดงานทั้งที่เป็น staff
- **`groupedAttendance`** (`src/app/admin/events/page.tsx`) เพิ่มเงื่อนไข: ถือว่าเป็น staff ด้วยถ้า user id อยู่ใน `staffUserIds` **ปัจจุบัน** ของกิจกรรม ไม่ใช่แค่เช็ค flag ที่ถูก freeze ไว้ตอน register — แก้ปัญหาการแสดงผลทันทีโดยไม่ต้องรอ backfill
- **`PUT /api/admin/events/[id]`** เมื่อบันทึก `staffUserIds` ใหม่ จะ diff กับค่าก่อนหน้าและ backfill `attendance.isStaff = true` ให้แถวที่มีอยู่แล้วของคนที่เพิ่งถูกเพิ่มเป็น staff — ทำแค่ทิศทางเดียว (false → true เท่านั้น ไม่ revert ตอนถอดคนออกจาก staff) เพื่อรักษา semantics เดิมที่กัน exploit เปิด quota คืนด้วยการถอด staff ออกภายหลัง พร้อมกันไปในทรานแซกชันเดียวกับการอัปเดตกิจกรรม
- ตรวจสอบด้วย `npm run lint`, `npx tsc --noEmit`, `npm run build` (ผ่านทั้งหมด ไม่มี error ใหม่จากแพ็กเกจนี้)
- ตัดแพ็กเกจนี้เป็น **tag/release v1.4.1**
