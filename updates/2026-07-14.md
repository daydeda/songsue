## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **แก้บั๊ก: ถอดคนออกจากทีมงาน (staff) ของกิจกรรมแล้ว ชื่อหายไปจากหน้า Attendance** — ก่อนหน้านี้ถ้าเคยกำหนดใครเป็น staff ของกิจกรรม (ทั้งที่คนนั้นลงทะเบียน/เช็คอินไปแล้ว) แล้วภายหลังถอดออกจากทีมงาน ระบบจะไม่คืนชื่อคนนั้นกลับไปอยู่ฝั่งนักศึกษาปกติ — ค้างอยู่ฝั่ง staff ตลอด และหายไปจากยอดนับต่าง ๆ (ลงทะเบียนล่วงหน้า/ขาดงาน/โควตา) ทำให้ดูเหมือนชื่อหายไปเลย ตอนนี้ถอดออกจากทีมงานแล้วชื่อจะกลับไปอยู่ฝั่งนักศึกษาและถูกนับตามปกติทันที
- **แก้บั๊ก: ยอดลงทะเบียนในหน้ากิจกรรมเกินโควตา เช่นขึ้น 17/16** — หน้ากิจกรรมฝั่งนักศึกษาเคยนับรวมทีมงาน (staff) ที่ลงทะเบียนกิจกรรมของตัวเองเข้าไปในยอด "ลงแล้ว/โควตา" ด้วย ทั้งที่ staff ไม่ถูกนับเข้าโควตาจริง ทำให้ตัวเลขที่แสดงดูเกินโควตาไปได้ ตอนนี้ยอดที่แสดงจะนับเฉพาะนักศึกษาจริงเหมือนกับการเช็คโควตาตอนลงทะเบียนแล้ว
- **ฟีเจอร์ใหม่: การแก้ไขกิจกรรมของประธานชมรม/สาขา ต้องรอทีมงานอนุมัติก่อนจึงจะมีผลจริง** — เดิมถ้าประธานชมรม/สาขาแก้ไขรายละเอียดกิจกรรมที่มีอยู่แล้ว (ชื่อ วันที่ โควตา สถานที่ ฯลฯ) การเปลี่ยนแปลงจะมีผลทันทีและนักศึกษาเห็นทันที ทั้งที่ทีมงานยังไม่ได้ตรวจสอบ ตอนนี้การแก้ไขจะถูกเก็บไว้รอตรวจสอบก่อน นักศึกษาจะยังเห็นข้อมูลเวอร์ชันล่าสุดที่อนุมัติแล้วจนกว่าทีมงานจะอนุมัติการเปลี่ยนแปลงนั้นจริง ๆ
- **หน้าข้อเสนอกิจกรรม (Proposals) ตอนนี้บอกด้วยว่าใครเป็นคนตรวจสอบ** ไม่ใช่แค่วันเวลาที่ตรวจสอบเหมือนเดิม
- **ปรับปรุง: ทีมงานเห็นได้ทันทีว่ากิจกรรมไหนมีการแก้ไขรออนุมัติ และเปลี่ยนอะไรไปบ้าง** — เดิมต้องกดเข้าไปแก้ไขกิจกรรมทีละอันถึงจะเห็นว่ามีข้อเสนอรออยู่ ตอนนี้การ์ดกิจกรรมในหน้า Manage Events จะขึ้นป้าย "President edited this event" พร้อมสรุป diff (ค่าเดิม → ค่าใหม่) และชื่อผู้เสนอ ให้เห็นตรง ๆ โดยไม่ต้องกดเข้าไปทีละกิจกรรม
- **แก้บั๊ก: ป้าย "President edited this event" ขึ้นทั้งที่ประธานไม่ได้แก้ไขอะไรเลย** — กิจกรรมเก่า 3 รายการที่มีอยู่ก่อนฟีเจอร์ตรวจสอบการแก้ไข (PR #183) ถูกระบบตราไว้ว่า "รอตรวจสอบ" ไปเองตั้งแต่ตอนเพิ่มฟีเจอร์ ทั้งที่ไม่มีใครแก้ไขจริง ทำให้ทีมงานเห็นป้ายเตือนค้างอยู่ถาวรโดยไม่มีอะไรให้ตรวจสอบ ตอนนี้แก้ไขข้อมูลย้อนหลังแล้วและป้องกันไม่ให้เกิดซ้ำ

## ฝั่งทีม (technical changelog)

### แก้บั๊ก: ถอด staff ออกแล้ว attendance.isStaff ไม่ revert (PR #180)

- **root cause:** `PUT /api/admin/events/[id]` (ตั้งแต่ PR #177/#179) backfill `attendance.isStaff` ทิศทางเดียวเท่านั้น (`false → true` ตอนเพิ่มเข้า `staffUserIds`) ไม่เคย revert ตอนถอดออก — คนที่เคยถูกตั้งเป็น staff แล้วมี attendance row อยู่ก่อน จะค้างเป็น `isStaff = true` ถาวรแม้ถูกถอดออกจาก `staffUserIds` ไปแล้ว ทำให้ค้างอยู่ฝั่ง Staff ในหน้า Attendance ตลอด และหลุดออกจากยอดนับฝั่งนักศึกษาทั้งหมด (quota/no-show/pre-registered ที่ filter `!isStaff`)
- **`PUT /api/admin/events/[id]`** (`src/app/api/admin/events/[id]/route.ts`) เพิ่ม logic สมมาตร: diff `staffUserIds` ทั้งสองทิศทาง — คนที่ถูกเพิ่มเข้ามาใหม่ยัง backfill `isStaff = true` เหมือนเดิม แต่ตอนนี้คนที่ถูก**ถอดออก**จะถูก set `isStaff = false` คืนด้วย (เฉพาะแถวของกิจกรรมนั้น เฉพาะที่ปัจจุบันเป็น `true`) ทำให้ assign/unassign เป็น action ที่ reversible จริง — ยอมรับ trade-off ที่ quota/no-show exemption จะไม่ถูกนับย้อนหลังอีกต่อไปหลังถอด staff ออก (ตรงข้ามกับที่ PR #177 เคยตั้งใจกันไว้) เพราะพฤติกรรมเดิมทำให้ดูเหมือนข้อมูลหายและสร้างความสับสนมากกว่า
- **`scripts/repair-stale-attendance-isstaff.mjs`** (ใหม่) — สคริปต์ one-off, idempotent สำหรับซ่อมแถวที่ค้าง `is_staff = true` อยู่แล้วบน production จากการถอด staff ที่เกิดขึ้นก่อนแก้บั๊กนี้ (mirror รูปแบบเดียวกับ `backfill-attendance-staff.mjs` แต่กลับทิศทาง) — ต้องรันบน prod ครั้งเดียวหลัง deploy ผ่าน Portainer console (`node scripts/repair-stale-attendance-isstaff.mjs`)
- ทดสอบจริงกับ local Postgres + dev-login: assign staff (isStaff → true) แล้ว unassign (ยืนยันว่า isStaff กลับเป็น false ถูกต้อง), รัน repair script กับแถวจำลองสถานะค้างเดิม ยืนยัน flip กลับ + idempotent (รันซ้ำแล้วไม่มีอะไรเปลี่ยน)
- ตรวจสอบด้วย `npm run lint`, `npm run build` (ผ่านทั้งหมด ไม่มี error ใหม่จากแพ็กเกจนี้ — error เดิมใน `battle-flow.test.ts` ไม่เกี่ยวข้องกับแพ็กเกจนี้)
- ไม่มีการเปลี่ยน schema/migration — เป็นแค่ logic fix + data repair script ที่ต้องรันครั้งเดียวบน prod
- ตัดแพ็กเกจนี้เป็น **tag/release v1.4.2**

### แก้บั๊ก: ยอดลงทะเบียนฝั่งนักศึกษานับรวม staff เข้าไปด้วย (PR #182)

- **root cause:** `getSeatCounts` ใน `GET /api/events` (`src/app/api/events/route.ts`) นับทุกแถวใน `attendance` ของกิจกรรมนั้นรวมเป็นยอด `registeredCount` โดยไม่กรอง `isStaff` ออก ต่างจาก `getAttendeeCounts` ฝั่ง admin roster (`src/app/api/admin/events/route.ts`) ที่กรองอยู่แล้ว และต่างจาก logic เช็คโควตาจริงตอนลงทะเบียน (`register/route.ts`) ที่ยกเว้น staff เช่นกัน
- เมื่อ staff ของกิจกรรมลงทะเบียนกิจกรรมของตัวเอง (ซึ่งตั้งใจให้ทำได้และไม่นับโควตา) ตัวเลขที่แสดงฝั่งนักศึกษาจะเกิน denominator เช่น `17/16` ทั้งที่ที่นั่งจริงยังพอ
- แก้โดยเพิ่ม `.where(eq(attendance.isStaff, false))` ใน `getSeatCounts` ให้ mirror กับ `getAttendeeCounts` — แก้เฉพาะตัวเลขที่แสดง ไม่กระทบ logic เช็คโควตาตอนลงทะเบียนซึ่งถูกต้องอยู่แล้ว
- ทดสอบจริงกับ local Postgres + dev-login: กิจกรรม quota=1, ตั้ง staff คนหนึ่งลงทะเบียน (isStaff=true) แล้วนักศึกษาจริงลงทะเบียนอีกคน — ก่อนแก้ขึ้น `2/1`, หลังแก้ขึ้น `1/1`
- ตรวจสอบด้วย `npm run lint`, `npm run build` (ผ่านทั้งหมด ไม่มี error ใหม่จากแพ็กเกจนี้ — error เดิมใน `battle-flow.test.ts` ไม่เกี่ยวข้องกับแพ็กเกจนี้)
- ไม่มีการเปลี่ยน schema/migration
- ตัดแพ็กเกจนี้เป็น **tag/release v1.4.3**

### ฟีเจอร์ใหม่: ระบบตรวจสอบการแก้ไขกิจกรรมของประธาน (hold-and-diff) + แสดงชื่อผู้ตรวจสอบข้อเสนอ (PR #183)

- **สิทธิ์เข้าถึง / access control:** เดิม `PUT /api/admin/events/[id]` เมื่อ club_president/major_president แก้ไขกิจกรรมที่มีอยู่แล้ว จะเขียนค่าลง column จริงของ `events` ทันที แล้วค่อย set `detailsReviewStatus = 'pending'` เป็นแค่ flag ให้ทีมงานรู้ทีหลัง — หน้ากิจกรรมฝั่งนักศึกษา (`GET /api/events`) จึงเห็นการเปลี่ยนแปลงที่ยังไม่ผ่านการตรวจสอบทันที และแบนเนอร์ที่ทีมงานเห็นก็บอกแค่ "มีการแก้ไข ให้ไปเช็คฟิลด์เอง" ไม่บอกว่าเปลี่ยนอะไรไปบ้าง
- **schema/migration (additive, idempotent):** เพิ่ม 3 column ใหม่บน `events` — `pendingDetailsChanges` (jsonb), `pendingDetailsSubmittedBy`, `pendingDetailsSubmittedAt` — เก็บค่าที่ประธานเสนอแก้ไข โดยไม่แตะ column จริงเลย จนกว่าทีมงานจะอนุมัติ (`migrate.ts` step 81, `drizzle/0029`) ต้องรัน `npm run db:migrate:container` ผ่าน Portainer console ก่อน deploy โค้ดนี้
- **`PUT /api/admin/events/[id]`:** แยก branch ชัดเจน — ประธาน (non-admin) เขียนได้แค่ field ที่อยู่ใน allowlist (`PRESIDENT_EDITABLE_FIELDS`: ชื่อ/รายละเอียด/วันเวลา/โควตา/สถานที่/รูป/ตารางวัน ฯลฯ) ลงไปที่ `pendingDetailsChanges` เท่านั้น ไม่แตะ column จริงเลย, ตรวจสิทธิ์ ownership (`EventScopeService.isEventManagedByScope`) เหมือนเดิมทุกครั้งก่อนรับการแก้ไข ทีมงาน (admin/registration/organizer) กด **Approve** เพื่อ apply diff ที่เก็บไว้ลง column จริง (ผ่าน field-mapper เดียวกับที่ staff แก้ไขตรง ๆ ใช้ ป้องกันไม่ให้ JSON ที่เก็บไว้แอบเขียน column ที่ไม่ได้ตั้งใจ) หรือกด **Discard** เพื่อทิ้งข้อเสนอโดยไม่กระทบข้อมูลจริงเลย
- **`GET /api/events`** (ฝั่งนักศึกษา) เพิ่ม `pendingDetailsChanges`/`pendingDetailsSubmittedBy`/`pendingDetailsSubmittedAt` เข้าไปใน strip list เดิม (คู่กับ `ownerClubIds`/`ownerMajors`) ไม่ให้ข้อเสนอที่ยังไม่อนุมัติหลุดไปถึงนักศึกษา
- **UI:** แบนเนอร์ฝั่งทีมงานตอนนี้แสดง diff แบบ field-by-field (ค่าเดิม → ค่าใหม่) พร้อมชื่อประธานที่เสนอและเวลาที่เสนอ, เปิดฟอร์มแก้ไขซ้ำในฐานะประธานจะเห็น draft ที่เสนอไว้แทนค่าจริงที่ยังไม่เปลี่ยน (ทีมงานเห็นค่าจริงเสมอ, diff แสดงแยกต่างหาก)
- **`event_proposals` (Task 2, ไม่มีการเปลี่ยน schema):** `EventProposalsService.listForViewer` และ `GET /api/admin/event-proposals/[id]` เพิ่ม join ความสัมพันธ์ `reviewer` ที่มีอยู่แล้วในฐานข้อมูล (column `reviewedBy` เดิม ถูก set ไว้ทุกครั้งที่ approve/reject/withdraw) เข้าไปในผลลัพธ์ — หน้า `/admin/proposals` เลยแสดงชื่อจริงแทนที่จะโชว์แค่วันเวลา
- ทดสอบจริงกับ local Postgres + dev-login: ประธานแก้ไขกิจกรรม → ยืนยันข้อมูลจริง/หน้านักศึกษาไม่เปลี่ยน, ทีมงานเห็น diff ถูกต้อง + ชื่อผู้เสนอ → Approve แล้วค่าจริงเปลี่ยนตาม (รวมตารางวัน), Discard แล้วค่าจริงไม่เปลี่ยนเลย, แก้ไขซ้ำโดยไม่เปลี่ยนอะไรจากข้อเสนอเดิมไม่ถูกนับเป็นการแก้ไขใหม่ (แก้บั๊กที่เจอระหว่างรีวิวเอง), และหน้า proposals แสดงชื่อผู้ตรวจสอบถูกต้อง
- ผ่าน `/recheck` เต็มรูปแบบ (bug/security/PDPA/perf/refactor) และรีวิวสิทธิ์เข้าถึง/PDPA แยกอีกรอบ — ไม่พบปัญหาระดับ crucial/moderate
- ตรวจสอบด้วย `npm run lint`, `npm run build` (ผ่านทั้งหมด ไม่มี error ใหม่จากแพ็กเกจนี้ — error เดิมใน `battle-flow.test.ts` ไม่เกี่ยวข้องกับแพ็กเกจนี้)
- ตัดแพ็กเกจนี้เป็น **tag/release v1.5.0**

### ปรับปรุง: แสดง diff การแก้ไขของประธานตรงบนการ์ดกิจกรรมเลย (PR #184)

- **root cause / gap ที่เจอ:** แบนเนอร์ diff จาก PR #183 (เช็คตรงบรรทัดที่ 38 ด้านบน) ถูก render เฉพาะตอนที่ทีมงานกดเปิดฟอร์มแก้ไขของกิจกรรมนั้นแล้วเท่านั้น — ในหน้ารายการ (`/admin/events`) ไม่มี indicator อะไรเลยว่ากิจกรรมไหนมีข้อเสนอค้างอยู่ ทีมงานเลยไม่มีทางรู้ได้นอกจากไล่กดเปิดแก้ไขทีละกิจกรรม
- **`src/app/admin/events/page.tsx`:** เพิ่มป้าย "President edited this event" (คลิกได้ กดแล้วพาไปยังฟอร์มแก้ไขพร้อม diff/Approve/Discard เดิม) และเพิ่ม diff panel แบบ read-only ตรงบนตัวการ์ดเลย (แสดง "Submitted by: ชื่อ" + รายการ field ที่เปลี่ยน ค่าเดิม → ค่าใหม่) — คำนวณจากข้อมูลที่ `GET /api/admin/events` ส่งมาอยู่แล้ว (`pendingDetailsChanges`/`pendingSubmitter`) ไม่มีการเปลี่ยน API/schema เพิ่ม
- ทดสอบจริงกับ local Postgres + dev-login (club_president + super_admin): ประธานแก้ชื่อ/โควตา/สถานที่ของกิจกรรมที่ดูแลอยู่ → ยืนยันค่าจริงไม่เปลี่ยน, การ์ดแสดง diff ถูกต้องครบ (`Event Title: A → B`, `Participant Quota: 50 → 75`, `Location: X → Y`, `Submitted by: ชื่อประธาน`) → กด Approve ผ่าน API แล้วค่าจริงอัปเดตตามและ pending state ถูกล้าง
- ตรวจสอบด้วย `npx tsc --noEmit`, `npm run lint`, `npm run build` (ผ่านทั้งหมด ไม่มี error ใหม่จากแพ็กเกจนี้ — error เดิมใน `battle-flow.test.ts` ไม่เกี่ยวข้องกับแพ็กเกจนี้)
- ไม่มีการเปลี่ยน schema/migration — เป็นแค่ client-side fix ที่ใช้ข้อมูลที่มีอยู่แล้ว

### แก้บั๊ก: ป้าย/คิวตรวจสอบ "President edited this event" ค้างอยู่ทั้งที่ไม่มีการแก้ไขจริง (PR #185)

- **root cause:** `drizzle/0027_slimy_nightcrawler.sql` (PR #183) เพิ่ม column `events.details_review_status` เป็น `DEFAULT 'pending' NOT NULL` โดยไม่ backfill — Postgres จึงตี stamp `'pending'` ย้อนหลังให้กิจกรรมทุกอันที่มีอยู่ก่อนหน้านั้นด้วย ทั้งที่ `pending_details_changes` ยังเป็น `NULL` (ไม่เคยมีประธานแก้ไขจริง) ขณะที่ฝั่ง UI (`src/app/admin/events/page.tsx`) และคิวตรวจสอบ (`GET /api/admin/reviews`) เช็คแค่ `detailsReviewStatus === 'pending'` อย่างเดียว ไม่ได้เช็คว่ามี diff จริงด้วย ทำให้กิจกรรมเก่า 3 รายการขึ้นป้าย/ค้างอยู่ในคิวถาวรโดยไม่มีอะไรให้ตรวจสอบ
- **migration (additive, idempotent, non-destructive):** เพิ่ม `drizzle/0030_backfill_details_review_status.sql` + `migrate.ts` step 82 — `UPDATE events SET details_review_status = 'approved' WHERE details_review_status = 'pending' AND pending_details_changes IS NULL` แก้เฉพาะแถวที่เป็น false positive เท่านั้น ข้อเสนอที่ยังรอตรวจสอบจริง (มี `pendingDetailsChanges`) ไม่ถูกแตะต้อง — รีเฮิร์สกับ local Postgres ผ่าน `.env.local` แล้ว 2 รอบ ไม่มี error
- **`src/app/admin/events/page.tsx`:** ป้ายบนการ์ด, แบนเนอร์ฝั่งทีมงาน (Approve/Discard), และแบนเนอร์ "รอตรวจสอบ" ที่ประธานเห็นเอง ทั้ง 3 จุดเพิ่มเงื่อนไข `pendingDetailsChanges` ต้องมีจริงคู่กับ `detailsReviewStatus === 'pending'` เสมอ ไม่ใช่เช็ค flag อย่างเดียว
- **`GET /api/admin/reviews`:** เพิ่มเงื่อนไข `isNotNull(events.pendingDetailsChanges)` เข้าไปในคิวรี ป้องกันไม่ให้กิจกรรมที่ไม่มี diff จริงหลุดเข้าคิวตรวจสอบของทีมงาน
- ทดสอบจริงกับ local Postgres: รัน migration แล้วยืนยันว่ากิจกรรมที่ไม่มี pending diff ถูกปรับกลับเป็น `approved` ถูกต้อง, รันซ้ำแล้ว idempotent (ไม่มีอะไรเปลี่ยนเพิ่ม)
- ตรวจสอบด้วย `npx tsc --noEmit`, `npm run lint`, `npm run build` (ผ่านทั้งหมด ไม่มี error ใหม่จากแพ็กเกจนี้ — error เดิมใน `battle-flow.test.ts` ไม่เกี่ยวข้องกับแพ็กเกจนี้)
- **ต้องรัน `npm run db:migrate:container` ผ่าน Portainer console บน prod** เพื่อแก้ข้อมูลกิจกรรม 3 รายการที่ได้รับผลกระทบจริง (โค้ดฝั่ง UI/API กันไม่ให้เห็นป้ายผิดได้เองอยู่แล้วแม้ยังไม่รัน migration แต่ migration คือตัวแก้ต้นตอที่ข้อมูล)
