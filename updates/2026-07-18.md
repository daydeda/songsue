## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **ฟีเจอร์ใหม่: ตำแหน่งงานในชมรม/สาขา (เช่น เลขา, การเงิน, ประชาสัมพันธ์ ฯลฯ) เข้าดูรายชื่อสมาชิกได้แล้ว** — เดิมมีแค่ "ประธาน" ชมรม/สาขาเท่านั้นที่เข้าเมนู "Clubs"/"Majors" ในหน้า admin ได้ ตอนนี้นักศึกษาที่ได้รับตำแหน่งอื่น ๆ ในชมรม/สาขาของตัวเอง (ไม่ใช่ประธาน) จะเห็นเมนูนี้ในแถบด้านข้างและเปิดดูรายชื่อสมาชิกในชมรม/สาขาตัวเองได้เช่นกัน — แต่เป็นแบบ "ดูอย่างเดียว" เห็นแค่ชื่อ รหัสนักศึกษา สาขา และบ้าน (house) เท่านั้น **ไม่เห็นเบอร์โทร/ช่องทางติดต่อ ไม่เห็นข้อมูลสุขภาพ และดาวน์โหลดไฟล์ export ไม่ได้** (สิทธิ์เหล่านี้ยังเป็นของประธานชมรม/สาขาและแอดมินเท่านั้น)
- **แก้บั๊ก: ตำแหน่งงานในชมรม/สาขา/สโมสรฯ (SMO)/อนุสโมสรฯ (ANUSMO) ไม่ทับกันแล้ว** — เดิมถ้านักศึกษาคนเดียวมีตำแหน่งอยู่หลายที่พร้อมกัน (เช่น เป็นเลขาชมรม A และฝ่ายการเงินของสาขา หรือเป็นทั้ง SMO และ ANUSMO) ระบบเก็บตำแหน่งไว้ที่เดียวรวมกัน พอตั้งตำแหน่งใหม่ที่หนึ่งจะไปทับตำแหน่งเดิมที่อื่นโดยไม่รู้ตัว ตอนนี้แต่ละที่ (ชมรม/สาขา/SMO/ANUSMO) แยกเก็บตำแหน่งของตัวเองแล้ว ตั้งที่ไหนก็ไม่กระทบที่อื่น และหน้า Students ของแอดมินก็แยกช่องตั้งตำแหน่ง SMO กับ ANUSMO ออกจากกันด้วย (คนเดียวเป็นได้ทั้งสองตำแหน่งพร้อมกัน)

## ฝั่งทีม (technical changelog)

### ฟีเจอร์ใหม่: สิทธิ์เข้าถึงตาม "ตำแหน่งงาน" (staff position) สำหรับรายชื่อสมาชิกชมรม/สาขา (PR #188)

- **`src/lib/positions.ts`** (ใหม่) — canonical enum `POSITION_IDS` (president, vice_president, secretary, finance, registration, coordination, pr, creative_production, project, venue, welfare, medical, club_affairs) เก็บเป็น id คงที่บน `users.position` แยกจาก `users.role`/`roles` (ระบบสิทธิ์) โดยสิ้นเชิง
- **สิทธิ์เข้าถึง (PDPA/access control):** เพิ่ม tier ที่ 3 ใน `GET /api/admin/clubs`(`/majors`) และ `.../members` — ผู้ถือ "ตำแหน่งงาน" ที่ไม่ใช่ประธาน จะเห็นเฉพาะชมรม/สาขาที่ตัวเองเป็นสมาชิกจริง (ตรวจสอบฝั่ง server ทุกครั้ง ไม่เชื่อ id จาก client) และได้ roster แบบจำกัด (`ClubsService.getClubMembersLimited` / เทียบเท่าฝั่งสาขา) — ไม่มี phone/contactChannels และไม่มีทางเรียก endpoint ข้อมูลสุขภาพ/export ได้เลย (route เหล่านั้นยังคง gate แค่ admin/president เหมือนเดิม ไม่ถูกแตะ)
- **`canEnterAdminAny`** (`src/lib/admin-access.ts`) ขยายจากเดิมที่อนุญาตเฉพาะ `position === "registration"` เป็นอนุญาตทุกตำแหน่งงาน (ยังคงถูกจำกัดเป็น scanner-only/confined เหมือนเดิมผ่าน `isScannerOnlyAny`)
- **bug fix ระหว่างทาง:** `AdminNav.tsx` เดิมเช็คแค่ `roles.includes("club_president")` ก่อนโชว์เมนู Clubs ในโหมด scanner-only ทำให้แม้ page/API จะรองรับผู้ถือตำแหน่งงานอื่นแล้ว แต่เมนูฝั่ง sidebar ไม่โผล่ให้กดเข้าเลย — แก้เป็นเช็ค `|| !!position` ด้วย
- เพิ่ม `GET .../members/[memberId]/medical` และ `GET .../members/export` (xlsx) สำหรับทั้งชมรมและสาขา — คงสิทธิ์เดิม (admin/president เท่านั้น) และ audit-log ทุกครั้งที่เข้าดู/export ตามข้อกำหนด PDPA
- **schema:** ประกาศ `users.position` ใน `src/db/schema.ts` (คอลัมน์นี้มีอยู่แล้วในทุกฐานข้อมูลจริงผ่าน `migrate.ts` step 6 ตั้งแต่ก่อนหน้านี้ — commit นี้แค่เพิ่มการประกาศ type ให้ตรงกับของจริง ไม่มี migration ใหม่ที่ต้องรันก่อน deploy)
- ตรวจสอบด้วย `npx tsc --noEmit`, `npm run lint`, `npm run build` (ผ่านทั้งหมด — error เดิมใน `battle-flow.test.ts` ไม่เกี่ยวข้องกับแพ็กเกจนี้)
- ตัดแพ็กเกจนี้เป็น **tag/release v1.7.0**

### แก้ CI: fixture ทดสอบเกม (battle/OX) ขาดคอลัมน์ `position` (PR #189)

- `stats-helper.test.ts` และ `battle-flow.test.ts` สร้างตาราง `users` แบบ raw SQL จำลองของจริงไว้ทดสอบ (ไม่ได้ใช้ schema จริง) — ตารางจำลองนี้เขียนไว้ก่อน PR #188 เพิ่มคอลัมน์ `position` จึงขาดคอลัมน์นี้ไป ทำให้ `db.insert(schema.users)` ที่ generate query จาก schema จริงพัง CI ทุกครั้ง (`column "position" of relation "users" does not exist`)
- เพิ่ม `position text` เข้าไปในตาราง fixture ทั้งสองไฟล์ให้ตรงกับ `schema.ts` — ไม่มีการแตะโค้ด runtime หรือ schema จริงเลย, ผลคือ CI ทั้ง 111 เทสต์ผ่าน
- ตัดแพ็กเกจนี้เป็น **tag/release v1.7.1**

### แยกเก็บ "ตำแหน่งงาน" (position) ตามชมรม/สาขา/SMO/ANUSMO แทนคอลัมน์เดียวรวมกัน (PR #192)

- **schema:** เพิ่ม `club_members.position` (ต่อชมรม), `users.majorPosition`, `users.smoPosition`, `users.anusmoPosition` (แยกอิสระต่อกัน — คนเดียวถือทั้ง smo และ anusmo พร้อมกันโดยมีตำแหน่งต่างกันได้) — `users.position` เดิมยังอยู่ในฐานข้อมูล (ไม่ลบ ไม่ backfill อัตโนมัติ เพราะถ้านักศึกษาอยู่หลาย context พร้อมกันจะเดาไม่ได้ว่าค่าเดิมเป็นของที่ไหน) ดู `scripts/list-legacy-positions.mjs` สำหรับ report (read-only) ใช้ reassign ตำแหน่งเองหลัง migrate
- **สิทธิ์เข้าถึง (access control):** ค่า `"registration"` ในคอลัมน์นี้เป็นทั้งชื่อตำแหน่งและสิทธิ์เข้าถึงระบบ (`isGlobalRegistrationPosition`, `canEnterAdminAny`, `EventScopeService`) จึงต้องแก้ session/token (`src/auth.ts`, `src/proxy.ts`, `src/lib/admin-access.ts`) และ API route ~20 ไฟล์ให้สอดคล้องกัน — จุดที่เสี่ยงที่สุดในโค้ดเบสตาม CLAUDE.md จึงให้ `pdpa-access-guard` รีวิวแยกต่างหาก
- **bug fix ระหว่างทาง:** `EventScopeService` เดิมถ้านักศึกษามีตำแหน่ง "registration" ในชมรมใดชมรมหนึ่ง จะได้สิทธิ์เห็น event ของ**ทุกชมรม**ที่ตัวเองเป็นสมาชิก (ไม่ใช่แค่ชมรมที่มีตำแหน่งนั้นจริง) — ตอนนี้ scope แคบลงให้ตรงเฉพาะชมรม/สาขาที่มีตำแหน่ง "registration" จริงเท่านั้น ยืนยันด้วยเทสต์จริง (สร้าง 2 ชมรม ตั้ง registration แค่ชมรมเดียว แล้วเช็คว่าเห็น event แค่ของชมรมนั้น)
- ตรวจสอบด้วย `npm run lint`/`npm run build` (ผ่านทั้งหมด), รีวิว `pdpa-access-guard` แยกต่างหาก (พบและแก้ 1 จุดที่ gate เข้มกว่าที่ควร), และ runtime `/verify` 25 เคสผ่าน (dev-bypass login ยิง API จริงกับ local DB ที่ migrate แล้ว)
- **⚠️ Prod ยังไม่ migrate ณ ตอน tag release นี้** — ต้องรัน `npm run db:migrate:container` ผ่าน Portainer console (`activecamt-app`) ก่อน deploy code เวอร์ชันนี้
- ตัดแพ็กเกจนี้เป็น **tag/release v1.8.0**
