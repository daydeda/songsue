### เพิ่มระบบ preview access token + ประตูล็อกก่อนวันเปิดลงทะเบียนจริง (PR #20)

- **`/preview` + ลิงก์ activation token ลับ** — แอดมินสร้าง/หมุน/ล้าง token ได้จากหน้าใหม่ `/admin/settings` (เก็บใน table ใหม่ `site_settings` ซึ่งเป็น singleton แบบเดียวกับ `announcements`) ใครก็ตามที่เปิดลิงก์ `/preview?token=...` ที่ถูกต้องจะถูก flip flag `users.previewAccess` เป็น true ทันที ทำให้ทีมทดสอบเข้าระบบและลงทะเบียนกิจกรรมได้ก่อนวันเปิดจริง โดยยังต้องผ่านเงื่อนไข quota / เวลาปิดรับ / walk-in-only / ปีหนึ่งเท่านั้น ตามปกติ (ข้ามได้แค่เงื่อนไข `registrationOpenTime` และ `allowedRoles`/`allowedMajors`/`allowedClubs` ของกิจกรรมนั้น ๆ)
- **จุดบังคับใช้จริงอยู่ที่ `src/proxy.ts` (edge middleware)** — ก่อนถึงวันเปิดลงทะเบียน (`REGISTRATION_OPENS_AT`, ค่า default 23 ก.ค. 69) มีแค่ผู้ที่มี `previewAccess` และสตาฟฟ์ที่เข้าหน้าแอดมินได้อยู่แล้วเท่านั้นที่จะเข้าหน้าที่ต้อง sign-in ได้ ผู้ใช้อื่นถูกกันไว้ที่หน้าแรกพร้อม countdown — ทั้ง countdown (cosmetic, `SongsueLanding.tsx`) และประตูบังคับจริงใน proxy ใช้ค่าคงที่เดียวกันจาก `src/lib/registration-window.ts` (edge-safe, อ่านแค่ `process.env`) ป้องกันไม่ให้ตัวเลขวันเปิดในหน้าเว็บกับตัวที่บังคับใช้จริงหลุดไม่ตรงกัน — รองรับ override เฉพาะ local ผ่าน `NEXT_PUBLIC_REGISTRATION_OPENS_AT` ใน `.env.local`

### จำกัดสิทธิ์แอดมินให้เห็นเฉพาะคณะตัวเอง (PR #21)

- **สตาฟฟ์ทุก role (ยกเว้น super_admin) เห็นเฉพาะคณะตัวเอง** — admin, registration, organizer, smo, anusmo, club_president, major_president ถูกจำกัดขอบเขตด้วยคอลัมน์ `users.faculty` เดิม (ไม่มีคอลัมน์ใหม่) ครอบคลุมหน้ารายชื่อนักศึกษา, สถิติ dashboard, การเช็คชื่อ/export/รายงานรายกิจกรรม และระบบสแกน QR เช็คอิน — เฉพาะ super_admin เท่านั้นที่ยังเห็นครบทั้ง 4 คณะ (CAMT/MASSCOM/ARCH/ARTS)
- **สแกนข้ามคณะถูกบล็อกทันที** — ถ้าสตาฟฟ์คณะหนึ่งสแกน QR ของนักศึกษาอีกคณะ ระบบจะขึ้น error ชัดเจนโดยไม่แสดงข้อมูลนักศึกษาใด ๆ เลย (สถานะใหม่ `wrong_faculty`); ถ้าบัญชีสตาฟฟ์ยังไม่ถูกกำหนดคณะ ก็ถูกกันไว้เช่นกัน (`no_faculty`) แทนที่จะเห็นข้อมูลทุกคณะโดยไม่ตั้งใจ
- **โมดูลกลาง `src/lib/faculty-scope.ts`** — จุดเดียวที่กำหนดกฎ scoping ทั้งหมด ทุก endpoint ที่แตะข้อมูลนักศึกษายังคง server-side role gate ของตัวเองแยกต่างหากตามปกติของโปรเจกต์ (faculty scoping เป็นชั้นเสริม ไม่ใช่ตัวแทนของ gate เดิม)
- **แก้บั๊กร้ายแรง 2 จุดที่เจอจากการ review แบบ adversarial:** (1) ก่อนแก้ ใครก็ตามที่เป็น admin/registration/organizer สามารถแก้ `faculty` ของตัวเองผ่าน `PATCH /api/profile` เพื่อสลับตัวเองเข้าคณะอื่นได้ — ปิดช่องนี้แล้ว (เปลี่ยน faculty ได้เฉพาะ super_admin ผ่าน route แอดมินเท่านั้น) (2) เนื่องจาก `faculty` เป็น concept ใหม่สำหรับบัญชีสตาฟฟ์ บัญชีสตาฟฟ์เดิมทุกบัญชีมีค่า `faculty = null` — ถ้าใช้ default เดิมที่แปลง null เป็น CAMT (ซึ่งถูกต้องสำหรับนักศึกษาเท่านั้น) จะทำให้แอดมินคณะ CAMT เห็น/แก้/ลบบัญชีสตาฟฟ์คณะอื่นที่ยังไม่ถูกกำหนดคณะได้ทั้งหมด แก้เป็น deny-safe: แถวที่ไม่ใช่นักศึกษาและ `faculty = null` จะไม่ match อะไรเลยจนกว่า super_admin จะกำหนดคณะให้
- **DB migration (step 85):** เพิ่ม index `idx_users_faculty` (`CREATE INDEX IF NOT EXISTS`, additive ล้วน) — rehearse ผ่าน local DB แล้ว **ยังไม่ได้รันกับ Supabase prod** ต้องรันผ่าน `/safe-deploy` ก่อน index จะมีผลจริงบน prod (โค้ดทำงานได้โดยไม่มี index อยู่แล้ว เพียงแต่ query จะช้ากว่าที่ควร)
- ตรวจสอบด้วย `npm run build` (ผ่าน), `npm run lint` (ไม่มี warning ใหม่), migration rehearsal + seed ข้อมูลทดสอบ 4 คณะบน local DB, ทดสอบ manual ผ่าน dev-bypass login ครบทุก role/คณะ (รวมกรณี super_admin เห็นครบ และบัญชีที่ยังไม่กำหนดคณะถูกกันไว้), และไล่ checklist `/scanner-verify` ครบ 6 เคสรวมเคสสแกนข้ามคณะใหม่
- ตัดแพ็กเกจนี้เป็น **tag/release v1.12.0**
