# ActiveCAMT (Songsue) — อัปเดต 21 ก.ค. 69

ช่วง: 21 ก.ค. 69 · สรุปไว้สำหรับลง Discord และให้ทีมที่เกี่ยวข้องอ่าน

## ฝั่งนักศึกษา (สิ่งที่ user จะเห็น)

- **เล่น P2P Battle ได้ทุกคนแล้ว** — เดิมเปิดให้เฉพาะ SMO/ANUSMO/แอดมิน ทดสอบบนระบบจริงก่อน ตอนนี้เปิดให้นักศึกษาทุกคนที่ล็อกอินเข้าเล่นได้แล้ว
- **กิจกรรมบางงานจะขึ้น "ลงทะเบียนที่ ActiveCAMT"** — สำหรับกิจกรรมที่จัดขึ้นฝั่ง ActiveCAMT แต่นับแต้มให้ Songsue ด้วย นักศึกษาจะเห็นการ์ดกิจกรรมโชว์สถานะพิเศษและต้องไปลงทะเบียน/เช็คอินที่ ActiveCAMT โดยตรง (กดลงทะเบียน/ยกเลิกในหน้า Songsue ไม่ได้สำหรับกิจกรรมกลุ่มนี้) แต้มบ้านจากการเข้าร่วมจะขึ้นในระบบ Songsue ให้เองอัตโนมัติ
- **ตอนนี้แอดมินเช็คอินกิจกรรมที่เชื่อมกับ ActiveCAMT จากฝั่งไหนก็ได้** — เดิมถ้ากิจกรรมถูก mirror มาจาก ActiveCAMT หน้าสแกนของ Songsue จะบล็อกไว้ทั้งหมด ต้องไปสแกนที่ ActiveCAMT เท่านั้น ตอนนี้สแกนที่ Songsue ได้เลย รวมถึงนักศึกษา walk-in ที่ไม่เคยลงทะเบียนในระบบไหนมาก่อนก็เช็คอินได้ (ระบบจะสร้างบัญชีให้อีกฝั่งอัตโนมัติ) — เฉพาะการให้คะแนนรายบุคคลเท่านั้นที่ยังทำได้แค่ฝั่ง Songsue
- **สีบ้านแสดงถูกต้องแล้วทุกที่** — บ้านสีเหลืองกับสีน้ำเงินเคยสลับกันแสดงผล (และบ้านสีเขียวเคยขึ้นเป็นสีเทา) ในบางหน้า ตอนนี้แก้ไขให้ตรงกับสีบ้านจริงทุกจุดแล้ว (แดชบอร์ดแอดมิน, หน้า Battle)
- **แก้บั๊กหน้าโปรไฟล์บางเคสไม่บันทึกการยินยอม PDPA** — ก่อนหน้านี้บางเงื่อนไขระบบไม่ได้บันทึกค่ายินยอม PDPA ที่นักศึกษากดไว้จริง ๆ ตอนนี้แก้ให้บันทึกถูกต้องแล้ว
- **หน้ากิจกรรมกรองตามคณะแล้ว** — นักศึกษาแต่ละคณะ (CAMT / MASSCOM / ARCH / ARTS) จะเห็นเฉพาะกิจกรรมของคณะตัวเองในหน้ารายการกิจกรรมและตอนลงทะเบียน ไม่ต้องไถผ่านกิจกรรมคณะอื่นอีกต่อไป (กิจกรรมที่ลงทะเบียน/เช็คอินไว้แล้วยังคงเห็นเหมือนเดิมไม่ว่าจะเป็นคณะไหน)

## ฝั่งทีม (technical changelog)

### ระบบ sync กิจกรรม/การลงทะเบียนจาก ActiveCAMT + เปิด P2P Battle เต็มรูปแบบ (PR #26)

- **Sync ทิศทางเดียวจาก ActiveCAMT → Songsue** — เมื่อแอดมิน ActiveCAMT ตั้งค่ากิจกรรมให้ "นับแต้มให้ Songsue ด้วย" และนักศึกษาลงทะเบียน/เช็คอิน QR ที่ฝั่ง ActiveCAMT ระบบใหม่ (`src/modules/integrations/activecamt-sync.service.ts` + `/api/integrations/activecamt/events`, `/register`) จะ mirror ข้อมูลกิจกรรมและสถานะการเข้าร่วมมาที่ตาราง `events`/`attendance` ของ Songsue ให้ cron แจกแต้มบ้าน (`checkAndAwardPastEventPoints`) เห็นและนับแต้มให้ถูกบ้าน — ActiveCAMT ยังคงเป็นเจ้าของการลงทะเบียน/โควตา/เช็คอิน QR เต็มรูปแบบเหมือนเดิม ฝั่ง Songsue ปิดกั้นการลงทะเบียน/ยกเลิก/สแกนสำหรับกิจกรรมที่ถูก mirror มาโดยเช็คคอลัมน์ใหม่ `events.externalSource` ในทุกจุด (register/cancel API, scan API, หน้าสแกนแอดมิน, การ์ดกิจกรรมบน dashboard)
- **บัญชีที่ sync มาเป็นแบบ PDPA-minimal** — สร้างบัญชี Songsue จริงให้นักศึกษาโดยอัตโนมัติที่ email เดียวกัน (ใช้ email เป็น join key ข้ามสองระบบ) แต่ `profileCompleted`/`pdpaConsent` เป็น false เสมอ และไม่แตะข้อมูลการแพทย์ใด ๆ เลย — ต้องให้นักศึกษายืนยัน PDPA consent ของ Songsue เองถ้าเข้าระบบตรง ๆ ในภายหลัง ส่วนบ้านจะถูกจัดให้แบบ balanced ตามระบบบ้านของ Songsue เอง (ไม่รับค่าบ้านจาก ActiveCAMT มาโดยตรง เพราะสองระบบใช้ house model คนละแบบ) และจัดให้ก็ต่อเมื่อ sync มาถึงสถานะ "attended" ครั้งแรกเท่านั้น ไม่ใช่ตอน "registered"
- **DB migration (step 88):** เพิ่มคอลัมน์ `events.external_source`/`events.external_id` + unique partial index `events_external_source_id_unique` (`WHERE external_id IS NOT NULL`) เป็น additive/idempotent ล้วน รันผ่าน `/safe-deploy` บน Supabase prod แล้วก่อน merge โค้ดที่อ่าน/เขียนคอลัมน์นี้
- **รีวิว PDPA/access-control** ผ่าน agent `pdpa-access-guard` ก่อน merge — ตรวจครบ: บัญชี sync ไม่แตะข้อมูลการแพทย์, audit log เขียนใน transaction เดียวกับการเปลี่ยนแปลงข้อมูลเสมอ, endpoint ใหม่ทั้งสองเช็ค bearer secret (`ACTIVECAMT_SYNC_SECRET`, constant-time, fail-closed) ก่อนแตะ DB ทุกครั้ง, และจุดกันการลงทะเบียน/สแกนซ้ำครอบคลุมทุก path จริง ไม่มี finding ระดับ crucial/moderate (มีแค่ 2 จุด low-severity เรื่อง reliability ที่ไม่กระทบความปลอดภัย)
- **เปิด P2P Battle ให้ทุก role ที่ล็อกอินแล้ว** — เดิมจำกัดไว้เฉพาะ SMO/ANUSMO/แอดมินสำหรับทดสอบบนระบบจริง (`src/lib/battle-access.ts`, `src/proxy.ts`, `src/app/battle/layout.tsx`) ตอนนี้เปิดให้ทุกคนแล้ว
- **แก้บั๊ก `GET /api/profile`** — เดิมถ้า session ยังใช้ได้แต่แถว user ในตารางถูกลบไปแล้ว (เช่นหลัง reset/reseed DB) จะโยน error "Value is not JSON serializable" แทนที่จะตอบ error ที่อ่านง่าย ตอนนี้ตอบ 404 ตรง ๆ แทน
- ตรวจสอบด้วย `npm run build` (ผ่าน, มี route ใหม่ทั้งสองขึ้นในผลลัพธ์), `npm run lint` (ไม่มี error/warning ใหม่ — error เดิมทั้งหมดยังอยู่ในไฟล์ที่ไม่เกี่ยวข้อง เช่น DoorCastle3D.tsx/SoftShadows.tsx), และ `npm test` (111/111 ผ่าน)
- **ยังไม่เสร็จ:** ฝั่ง ActiveCAMT (repo `smocamt-website` แยกต่างหาก) ที่ยิง sync มาหา endpoint พวกนี้ยังไม่ได้ทำ — ต้องตั้ง `ACTIVECAMT_SYNC_SECRET` ให้ตรงกันทั้งสองฝั่งบน Vercel ก่อนใช้งานจริงได้
- ตัดแพ็กเกจนี้เป็น **tag/release v1.14.0**

### ระบบ sync เช็คอินสองทิศทางเต็มรูปแบบ + แก้สีบ้าน + แก้บั๊ก auth/โปรไฟล์ (PR #27)

- **Sync เช็คอินสองทิศทาง (ต่อยอดจาก PR #26 ที่เป็นทิศทางเดียว)** — เดิม Songsue บล็อกการลงทะเบียน/ยกเลิก/สแกนทั้งหมดสำหรับกิจกรรมที่ mirror มาจาก ActiveCAMT ตอนนี้ปลดล็อกให้สแกน/ยืนยันเช็คอิน (`scan`/`confirm`/`lookup`) ผ่านได้ปกติ เหลือแค่ `score` (ให้คะแนนรายบุคคล) ที่ยังบล็อกไว้เหมือนเดิม (`src/app/api/admin/scan/route.ts`) — เช็คอินที่เกิดขึ้นฝั่ง Songsue จะยิง sync กลับไปหา ActiveCAMT ด้วย (ไฟล์ใหม่ `src/lib/activecamt-outbound-sync.ts`, เรียกจาก `scanner.service.ts` หลัง transaction commit เท่านั้น, best-effort ไม่ throw ไม่บล็อกเช็คอินจริงถ้า ActiveCAMT ล่ม) รวมถึง walk-in ที่ไม่เคยมีบัญชีฝั่ง ActiveCAMT มาก่อนก็สร้างบัญชีให้อัตโนมัติ (`src/modules/integrations/activecamt-sync.service.ts`)
- **แก้ payload sync กิจกรรมให้พก `walkInsEnabled`/`quota`/`quotaWalkIn` ไปด้วยทั้งสองทิศทาง** — เจอระหว่างทดสอบ E2E จริงว่ากิจกรรมที่ mirror มาจะปิด walk-in เสมอไม่ว่าต้นทางจะตั้งค่าไว้ยังไง เพราะ payload เดิมไม่เคยส่ง 3 ฟิลด์นี้มาเลย (`src/app/api/integrations/activecamt/events/route.ts`)
- **DB migration (step 89):** แก้ `houses.color` ให้ตรงกับ `color_group` จริง (บาง migration step ก่อนหน้าเคยตั้งค่าสีเหลือง/น้ำเงินสลับกัน และให้สีเขียวเป็นสีเทาแทน) — เป็น `UPDATE` ที่ scope ด้วย `color_group` ชัดเจน ไม่ทำลายข้อมูล รันผ่าน `/safe-deploy` บน Supabase prod แล้วก่อน merge โค้ดนี้ พร้อมแก้ fallback hex ในไฟล์ที่เกี่ยวข้อง (`faculties.ts`, `globals.css`, แดชบอร์ดแอดมิน, หน้า Battle) ให้ตรงกัน
- **แก้บั๊กที่เจอระหว่างทดสอบ E2E จริงบน DB local:**
  - ตั้งชื่อ cookie ของ NextAuth ใหม่ทั้งหมดให้มี prefix `songsue.*` (`src/auth.ts`) — เดิม Auth.js ใช้ชื่อ cookie เดียวกันทุกแอป ทำให้ Songsue กับ ActiveCAMT ที่รันคู่กันในเครื่อง (คนละพอร์ต) แย่ง cookie เดียวกันโดยไม่รู้ตัว
  - เอาการ redirect บังคับไปหน้า `/onboarding` ออกจาก `src/proxy.ts` — บัญชีที่ sync มาจาก ActiveCAMT อาจมีโปรไฟล์ไม่ครบได้โดยชอบธรรม ตอนนี้ให้กรอกทีหลังตอนที่ต้องใช้จริง (เช่นตอนลงทะเบียนกิจกรรม) ผ่าน component ใหม่ `QuickProfileModal` แทนการเช็คทุกหน้า
  - `pdpaConsent` ที่ส่งมาจาก client เคยหายไปเงียบ ๆ เพราะ Zod schema ของ `/api/profile` ไม่ได้รับฟิลด์นี้ไว้ — แก้ให้บันทึกค่าจริงแล้ว พร้อมกัน GET/PATCH พังกรณี session ยังอยู่แต่แถว user ถูกลบไปแล้ว (ตอบ 404 แทนการ throw error ที่อ่านไม่รู้เรื่อง)
- ตรวจสอบด้วย `npm run build` (ผ่าน), `npm run lint` (ไม่มี error/warning ใหม่ในไฟล์ที่แก้ — error เดิมยังอยู่ใน DoorCastle3D.tsx/SoftShadows.tsx ที่ไม่เกี่ยวข้อง), `npm test` (111/111 ผ่าน) และทดสอบ E2E จริงด้วย local Postgres ทั้งสองฝั่ง (บันทึกละเอียดไว้ใน `handoff.md` ในเครื่อง ไม่ได้ commit เข้า repo)
- ตัดแพ็กเกจนี้เป็น **tag/release v1.15.0**

### เพิ่มระบบแบ่งกิจกรรมตามคณะ (PR #29)

- **`events.faculty`** — ทำตามแพทเทิร์นเดียวกับ `announcements.faculty` (`src/lib/faculty-scope.ts`) มาใช้กับตารางกิจกรรม: ผู้ใช้ที่ไม่ใช่ super_admin (นักศึกษา, staff, ประธานชมรม/สาขา ฯลฯ) จะเห็น/ลงทะเบียนได้เฉพาะกิจกรรมของคณะตัวเองเท่านั้น ทั้งใน `GET /api/events` (ฝั่งนักศึกษา), `GET /api/admin/events` (ฝั่งแอดมิน) และ `POST /api/events/[id]/register`
- **กิจกรรมใหม่ถูก tag คณะอัตโนมัติจากผู้สร้าง** — ฟังก์ชันใหม่ `resolveWriteFaculty` บังคับใช้คณะของผู้สร้างเองเสมอสำหรับ staff ที่ผูกกับคณะเดียว กัน request ที่ปลอมค่า `faculty` มาจาก client — มีแค่ super_admin เท่านั้นที่เลือกคณะอื่นให้กิจกรรมได้ตรง ๆ
- **กิจกรรมเก่า/ที่ mirror มาจาก ActiveCAMT ไม่ต้องทำอะไร** — คอลัมน์เป็น `null` ได้ และ `null` จะถูกตีความเป็น CAMT เสมอ (ตามคอนเวนชัน null→CAMT เดิมของ `users.faculty`) กิจกรรมทุกอันก่อนหน้านี้จึงยังคงแสดงผลให้นักศึกษา CAMT เห็นเหมือนเดิมโดยไม่ต้อง backfill ข้อมูล
- **DB migration (step 90):** เพิ่มคอลัมน์ `events.faculty` (nullable text) + index `idx_events_faculty` เป็น additive/idempotent ล้วน ไม่มี backfill รันผ่าน `/safe-deploy` (rehearse บน local DB ก่อน แล้วค่อย apply บน Supabase prod) เสร็จก่อน merge โค้ดที่อ่านคอลัมน์นี้
- ตรวจสอบด้วย `npm run build` (ผ่าน) และ `npm run lint` (ไม่มี error/warning ใหม่ในไฟล์ที่แก้ — error เดิม 8 จุดยังอยู่ใน `DoorCastle3D.tsx`/`SoftShadows.tsx` ที่ไม่เกี่ยวข้อง)
- ตัดแพ็กเกจนี้เป็น **tag/release v1.16.0**
